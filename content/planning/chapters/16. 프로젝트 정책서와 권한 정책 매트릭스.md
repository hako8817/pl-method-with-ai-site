# 16장. 프로젝트 정책서와 권한 정책 매트릭스

> 이 장은 두 산출물을 함께 다룬다. 프로젝트 정책서는 PRD의 방향을 실행 규칙으로 내려보내는 중간 기준 문서다. 권한 정책 매트릭스는 그 정책 안에서 누가 어떤 기능과 데이터에 접근할 수 있는지를 정의한다. 두 산출물은 한 쌍으로 설계되어야 한다.

---

## 제1절. 프로젝트 정책서


> **이 장에서 다루는 것**
> - 정책서의 역할과 PRD → BPMN·DMN 사이의 위치
> - AI와 함께 정책 규칙을 구조화하는 3단계
> - 정책서 → BPMN·DMN 전환 조건

![](./figures/fig-16-policy-chain.svg)

> 도식: 정책서 산출물 체인: PRD 입력 → 정책 규칙·예외 처리·책임 분리 → BPMN·DMN·사양서 출력

> **전제 지식**: 11장(PRD), 13장(Feature Spec), 14장(Scenario와 AC) — 정책서는 PRD의 방향과 Feature의 시나리오·AC를 받아 실행 규칙으로 변환한다. 앞 세 장을 먼저 읽어야 정책서의 위치를 이해할 수 있다.

## 이 장의 목적

이 장은 PRD에서 정한 방향을 실제 실행 규칙으로 내려보내는 `프로젝트 정책서(Policy Document)`의 역할과 작성 방법을 설명한다. 이 책에서 정책서는 상위 원칙을 복사하는 문서가 아니다. 이번 과제에서 실제로 적용할 규칙·예외·허용/차단 기준·운영 승인 기준을 확정하는 `중간 기준 문서`다.

이 장을 읽고 나면 다음을 할 수 있어야 한다.

- 공통 정책 원칙과 프로젝트 정책서의 차이를 설명할 수 있다.
- 정책서에 담을 내용과 BPMN·DMN·사양서로 넘길 내용을 구분할 수 있다.
- AI를 활용해 정책서를 만들 때 무엇을 맡기고 무엇을 직접 확정해야 하는지 구분할 수 있다.
- 완성된 정책서가 어떤 조건에서 다음 산출물로 넘어가는지 확인할 수 있다.

---

## 1. 이 산출물이 왜 존재하는가

PRD는 방향과 범위를 정한다. 그러나 PRD만으로는 실제 협업이 흔들리는 지점을 막지 못한다. 팀은 대체로 다음 질문 앞에서 서로 다른 답을 내놓는다.

- 어떤 상태를 허용하고 어떤 상태를 막을 것인가
- 예외는 어디까지 인정할 것인가
- 어떤 조건에서 운영 승인이 필요한가
- 자동 처리와 수동 처리의 경계는 어디인가

이 질문들은 정책의 영역이다. 그런데 정책이 문서 밖에 남아 있으면 결과는 예측 가능하다.

- 기획은 A라고 이해했는데 개발은 B로 구현한다.
- QA는 기본 규칙만 보고 예외를 놓친다.
- 운영은 고객 응대 기준을 따로 해석한다.
- AI는 흩어진 정책 문장을 읽고 그럴듯하지만 잘못된 규칙을 만든다.

정책서는 이 구멍을 막는 문서다. PRD를 BPMN·DMN·사양서·테스트케이스로 안정적으로 내려보내기 위한 핵심 기준 문서다.

## 2. 입력과 출력

### 2-1. 정책서가 받는 주요 입력

| 입력 소스 | 내용 |
|---|---|
| PRD | 과제 범위·핵심 시나리오·성공 기준·리스크 |
| RIB | 정책 충돌 의심 지점·운영 예외 반복 패턴 |
| 공통 정책 원칙 | 상위 기준 문서 (보안·개인정보·운영 가이드라인) |
| 권한 정책 매트릭스 | 역할별 허용/금지 기준 |
| 운영팀 인터뷰 | 실제 수동 예외 처리 패턴·고객센터 응대 이슈 |
| As-is 분석 | 현재 정책 문서와 구현 간 불일치 지점 |

### 2-2. 정책서가 넘겨주는 출력

| 다음 산출물 | 정책서에서 받는 것 |
|---|---|
| **BPMN** | 흐름 후보 (누가·언제·어떤 순서로) |
| **DMN** | 판단 후보 (어떤 조건 조합에서 어떤 결과) |
| **사양서** | 상태값 정의·메시지 기준·예외 처리 조건 |
| **테스트케이스** | 규칙별 기대 결과·예외 케이스 |
| **운영 정책서** | 운영 승인 기준·수동 처리 범위 |
| **프로토타입** | 예외 상태에서 사용자가 받을 피드백 기준 |

> **핵심 원칙**: 정책서가 확정되면 BPMN·DMN·사양서가 같은 기준 위에서 시작된다. 정책서가 비면 뒤 산출물이 각자 다른 규칙을 만든다.

## 3. 정책서 구조와 템플릿

정책서는 7개 섹션으로 구성된다. 처음부터 모두 채울 필요는 없다. 단, BPMN·DMN 작성을 시작하기 전에 기본 규칙·예외 규칙·운영 승인 기준은 반드시 확정한다.

### 3-1. 구성 항목

| 섹션 | 내용 | 비고 |
|---|---|---|
| 정책 범위 | 이 정책서가 다루는 기능 영역 | PRD 범위와 일치해야 함 |
| 상위 기준 문서 | 참조하는 공통 정책·법규·보안 가이드 | 원칙과 적용 기준 구분 |
| 기본 규칙 | 기본 상태에서 적용하는 규칙 | 구체적 수치·조건 포함 |
| 예외 규칙 | 기본 규칙의 예외와 우선순위 | 예외 적용 조건 명시 |
| 운영 승인 기준 | 수동 개입·운영 승인이 필요한 조건 | 승인 주체·범위 명시 |
| 영향 산출물 | 이 정책이 바뀌면 영향 받는 산출물 목록 | 변경 시 파급 범위 관리 |
| BPMN·DMN 분류 | 흐름 후보·판단 후보 구분 표 | 다음 산출물 작성 기준 |

### 3-2. 템플릿 (마크다운 형식)


## 4. AI와 함께 만드는 법

### 4-1. 흐름 개요

```
PM → PRD + RIB 전달 → AI 규칙 초안 → PM 충돌 해소·확정 → AI 분류 → PM 승인
```

### 4-2. AI에게 맡기는 단계

**단계 1: PRD에서 정책 후보 추출**

```
다음 PRD를 읽고 정책서에 명문화해야 할 규칙 항목을 추출해줘.

[PRD 내용 붙여넣기]

추출 항목:
1. 기본 규칙 후보 (허용/차단/처리 기준이 필요한 것)
2. 예외 규칙 후보 (기본 규칙의 예외가 예상되는 것)
3. 운영 승인이 필요할 가능성이 있는 조건
4. 정책 충돌이 의심되는 항목
5. 확인이 필요한 규칙 (현재 불명확한 것)
```

**단계 2: 운영 인터뷰 반영 후 초안 보완**

```
위 정책 초안에 다음 운영팀 인터뷰 내용을 반영해서 예외 규칙을 보완해줘.

[인터뷰 요약]
- 고객센터에서 잠금 해제를 수동으로 처리하는 빈도: 월 약 200건
- 주요 사유: 본인확인 수단 없는 고령 사용자
- 현재 판단 기준: 담당자 개인 재량

추가 요청:
- 예외 처리 조건 명시
- 승인 주체와 범위 구분
```

**단계 3: BPMN·DMN 분류 초안**

```
위 정책서를 기준으로:
1. BPMN으로 그려야 할 흐름 후보 (역할·순서·상태 전환이 중요한 것)
2. DMN으로 정의해야 할 판단 후보 (조건 조합이 복잡한 것)
를 분류해줘.
각 항목에 이유를 한 줄로 포함해줘.
```

### 4-3. PM이 직접 판단·확정해야 하는 것

| 항목 | 이유 |
|---|---|
| 실제 운영 예외의 허용 범위 | 법무·보안·조직 정책 확인 필요 |
| 정책 충돌 해소 | 이해관계자 간 조율 필요 |
| 수동 승인 범위 확정 | 운영 부담·책임 소재 판단 필요 |
| 규칙 우선순위 확정 | 예외가 기본을 덮는 조건 PM이 책임 |
| 최종 승인 주체 명시 | 조직 책임 구조 반영 |

> **원칙**: AI는 정책 초안을 빠르게 구조화할 수 있다. 그러나 **정책의 책임 있는 확정**은 사람이 맡는다.

> **→ 프롬프트 예시**: [[wiser-prompt-examples-v1]] — 11. 정책서 초안 생성 예시

## 5. 품질 기준 / 체크리스트

### 5-1. 필수 확인 (이것이 없으면 BPMN·DMN을 시작하면 안 된다)

- [ ] PRD 범위와 직접 연결되는가
- [ ] 상위 공통 정책과 모순되지 않는가
- [ ] 기본 규칙과 예외 규칙이 분리되어 있는가
- [ ] 운영 승인 기준과 승인 주체가 명시되었는가
- [ ] BPMN 후보와 DMN 후보가 구분되었는가

### 5-2. 품질 향상 항목

- [ ] 예외 규칙에 우선순위가 있는가
- [ ] 정책 변경 시 영향 산출물 목록이 있는가
- [ ] 각 규칙에 ID가 부여되어 추적 가능한가
- [ ] 확인 필요 항목이 숨겨지지 않고 드러났는가

### 5-3. 자주 발생하는 오류

| 오류 유형 | 현상 | 수정 방향 |
|---|---|---|
| 상위 원칙 복사 | "개인정보보호법을 준수한다" 수준의 문장만 있음 | 이번 과제에 적용되는 구체적 수치·조건으로 재작성 |
| 흐름 혼입 | 규칙 내에 "사용자가 → 시스템이 → 운영자가"처럼 순서 서술 | 흐름은 제거하고 BPMN으로 이동 |
| 판단 혼입 | 정책서 내에 조건 조합 전체 기술 | 조건 조합은 DMN으로 분리 |
| 예외 미정의 | 기본 규칙만 있고 예외 없음 | 운영팀 인터뷰로 예외 발굴 |

## 6. 다음 단계 연결 규칙

### 6-1. 정책서 → BPMN·DMN 전환 조건

다음 조건이 충족되어야 BPMN·DMN 작성을 시작한다.

- 기본 규칙이 확정되었다
- 예외 규칙과 우선순위가 명시되었다
- 운영 승인 기준이 확정되었다
- BPMN 후보와 DMN 후보가 분류되었다
- 정책서 상태가 Approved로 변경되었다

### 6-2. 정책서가 열어주는 병렬 작업

정책서가 Approved 상태가 되면 다음 산출물을 병렬로 시작할 수 있다.

| 병렬 가능 산출물 | 정책서에서 받는 것 |
|---|---|
| BPMN | 흐름 후보·역할 개입 기준 |
| DMN | 판단 후보·조건 조합 |
| 권한 정책 매트릭스 | 역할별 허용/금지 기준 |

### 6-3. 정책서에서 해결하지 않는 것

정책서는 다음을 결정하지 않는다.

- 흐름의 순서와 역할 분기 (→ BPMN에서)
- 조건 조합별 판단 결과 (→ DMN에서)
- 구현 상세·상태값·API 명세 (→ 사양서에서)
- 검증 케이스 (→ 테스트케이스에서)

이 내용들이 정책서에 섞이면 정책서는 비대해지고, 뒤 산출물과 역할이 겹친다.

## 적용 예시: 회원가입/로그인 정책 복원 과제

이 과제의 정책서는 다음을 확정한다.

**기본 규칙 (발췌)**
- P-01: 로그인 실패 5회 누적 시 계정 잠금
- P-02: 잠금 해제는 본인확인 성공 또는 운영 승인으로만 가능
- P-03: 탈퇴 후 30일 이내 재가입 불가
- P-04: 휴면 전환 기준: 12개월 미로그인
- P-05: 휴면 해제는 본인확인 성공 후 즉시 활성 전환

**BPMN 후보**: 휴면 로그인 흐름, 계정 잠금 후 해제 흐름, 탈퇴 후 재가입 흐름
**DMN 후보**: 로그인 허용 여부 판단, 재가입 허용 여부 판단, 잠금 해제 가능 여부 판단

## 정리

프로젝트 정책서는 원칙을 나열하는 문서가 아니다. 좋은 정책서는 이번 과제에서 실제로 적용할 규칙과 예외를 확정하고, 그 규칙을 BPMN·DMN·사양서·테스트케이스로 안정적으로 내려보내는 기준 문서다.

정책서에서 기억할 세 가지:

1. **규칙과 예외를 확정한다** — 방향이 아니라 실행 기준을 적는다.
2. **흐름은 BPMN으로, 판단은 DMN으로 넘긴다** — 정책서에 모두 담으려 하면 비대해진다.
3. **운영 승인 기준을 명시한다** — 모호한 예외는 현장에서 임의로 처리된다.

---

## 다음 장으로의 연결

정책서가 완성되면 "규칙이 어떤 순서와 흐름으로 실행되는가"를 시각화할 차례다. 정책서의 순서 규칙과 조건 규칙은 BPMN의 Sequence Flow와 Gateway로 직접 변환된다.

### 정책서에서 BPMN으로 이동하는 항목

| 정책서 항목 | 이동 형태 | BPMN에서의 역할 |
|---|---|---|
| 순서 규칙 ("A 후에 B 가능") | Sequence Flow 조건 | 프로세스 실행 순서 |
| 조건부 규칙 ("A이면 B, 아니면 C") | Gateway 분기 조건 | Exclusive / Inclusive Gateway |
| 역할별 처리 규칙 | 역할-작업 매핑 | Pool / Swimlane 분리 기준 |
| 타임아웃·대기 규칙 | 시간 조건 | Timer Event |
| 예외 처리 규칙 | 예외 발생 조건과 처리 경로 | Error / Exception 분기 |
| 운영 개입 규칙 | 수동 처리 조건 | 운영자 Task 또는 Manual Task |

### 체인 전환 기준 (정책서 → BPMN)

| 확인 항목 | 기준 |
|---|---|
| 규칙 확정 | "확인 필요" 상태인 규칙이 없는가 |
| 흐름 후보 목록 | BPMN으로 변환할 주요 흐름이 정책서에서 식별되어 있는가 |
| 역할 정의 완료 | 각 규칙의 처리 주체(역할)가 명시되어 있는가 |
| 정책서 상태 | `Approved` 상태인가 |

- **이 장이 결정한 것**: 실행 규칙, 예외 처리, 역할별 처리 기준
- **다음 장이 시작하는 것**: 정책서의 흐름 규칙을 BPMN 프로세스 다이어그램으로 구조화

## 제2절. 권한 정책 매트릭스

## 이 장의 목적

이 장은 시스템 안에서 누가 어떤 기능에 접근할 수 있고 어떤 데이터를 볼 수 있는지를 정의하는 **권한 정책 매트릭스(Permission Policy Matrix)**를 다룬다.

권한 설계는 보안 담당자의 영역으로 넘기거나, 개발 단계에서 결정하는 경우가 많다. 그러나 권한은 정책의 일부다. 누가 무엇을 할 수 있는가는 비즈니스 규칙이고, 이 규칙이 모호하면 DMN에서 정의한 판단 기준과 충돌하고, 사양서와 테스트케이스에서 빈자리가 생긴다.

이 장을 읽고 나면 독자는 다음을 할 수 있어야 한다.

- 권한 정책이 왜 정책서·DMN과 함께 설계되어야 하는지 설명할 수 있다.
- RBAC(역할 기반 접근 제어) 기반 권한 매트릭스를 구조화할 수 있다.
- AI와 함께 권한 매트릭스 초안을 만들고 PM이 무엇을 확정해야 하는지 판단할 수 있다.
- 권한 정책이 사양서·테스트케이스·운영 정책서와 어떻게 연결되는지 이해한다.

---

## 1. 이 산출물이 왜 존재하는가

기능이 설계됐는데 "이 기능은 누가 쓸 수 있는가"에 대한 합의가 없으면 다음 문제가 반복된다.

- 관리자만 볼 수 있는 데이터를 일반 사용자가 API를 통해 접근하는 상황이 생긴다.
- 운영팀이 특정 기능의 접근 권한 확인을 문의할 때 기준 문서가 없다.
- 역할별 테스트케이스 작성 시 권한 조건이 명확하지 않아 누락이 생긴다.
- 신규 팀원이 어떤 역할이 무엇을 할 수 있는지 문서 없이 코드에서 확인해야 한다.

권한 정책 매트릭스는 이 문제를 해결하기 위해 **역할(Role)과 기능(Feature), 데이터(Data)의 접근 가능 여부를 명시적으로 정의한 문서**다.

정책서가 "탈퇴 후 30일간 개인정보를 보존한다"를 정의한다면, 권한 정책 매트릭스는 "그 30일간 보존된 데이터에 누가 접근할 수 있는가"를 정의한다. 둘은 한 쌍이다.

---

## 2. 무엇을 입력으로 받고 무엇을 넘기는가

**입력:**

| 입력 소스 | 내용 |
|----------|------|
| PRD | 사용자 유형, 주요 기능 목록 |
| 정책서 | 역할별 행위 제한, 데이터 접근 정책 |
| DMN | 조건별 허용/차단 판단 기준 |
| 이해관계자 정보 | 실제 운영 조직 구조, 역할 분류 |

**출력:**

| 다음 산출물 | 권한 매트릭스에서 넘겨주는 것 |
|------------|-------------------------------|
| **사양서** | 기능별 접근 제어 조건 (인증·권한 체크 로직) |
| **API 명세서** | API별 허용 역할, 인증 방식, 403 응답 조건 |
| **테스트케이스** | 역할별 접근 가능·불가 케이스, 권한 우회 시도 케이스 |
| **운영 정책서(OPD)** | 권한 부여·회수 운영 절차 |

---

## 3. 권한 정책 매트릭스 구조

### 3-1. 역할(Role) 정의

역할은 시스템이 구분하는 사용자 유형이다. 실제 직책과 다를 수 있다.

| 역할 코드 | 역할명 | 설명 |
|-----------|--------|------|
| GUEST | 비인증 사용자 | 로그인 없이 접근 가능한 영역만 허용 |
| USER | 일반 회원 | 본인 데이터 조회·수정, 서비스 이용 |
| OPERATOR | 운영자 | 회원 데이터 조회, 수동 처리 권한 |
| ADMIN | 시스템 관리자 | 전체 데이터 접근, 설정 변경, 역할 부여 |
| SUPER | 슈퍼 관리자 | 감사 로그 접근, ADMIN 권한 부여·회수 |

역할은 단순할수록 좋다. 역할이 너무 세분화되면 관리 비용이 급증한다.

### 3-2. 기능 권한 매트릭스

| 기능 | GUEST | USER | OPERATOR | ADMIN |
|------|-------|------|----------|-------|
| 회원가입 | ✓ | — | — | — |
| 로그인 | ✓ | — | — | — |
| 내 정보 조회 | — | ✓ 본인만 | — | ✓ |
| 내 정보 수정 | — | ✓ 본인만 | — | ✓ |
| 회원 탈퇴 | — | ✓ 본인만 | — | — |
| 타 회원 정보 조회 | — | — | ✓ 제한적 | ✓ |
| 계정 잠금 해제 | — | — | ✓ | ✓ |
| 역할 변경 | — | — | — | ✓ |
| 감사 로그 조회 | — | — | — | ✓ |

범례: `✓` 허용 / `—` 불허 / `✓ 조건` 조건부 허용

### 3-3. 데이터 접근 범위

기능 권한 외에 **어떤 데이터까지 볼 수 있는가**도 정의한다.

| 데이터 | USER | OPERATOR | ADMIN |
|--------|------|----------|-------|
| 본인 개인정보 | 전체 조회·수정 | 일부 조회 (이름·상태) | 전체 조회·수정 |
| 타 회원 개인정보 | 불허 | 제한 조회 (민감 항목 마스킹) | 전체 조회 |
| 결제 정보 | 본인 이력만 | 불허 | 조회 (수정 불가) |
| 감사 로그 | 불허 | 불허 | 전체 조회 |
| 탈퇴 후 보존 데이터 | 불허 | 요청 처리 목적 한정 | 전체 접근 |

## 4. AI와 함께 만드는 법

### 단계 1: 입력 준비

AI에게 다음을 제공한다.
- PRD의 주요 기능 목록
- 정책서의 역할 관련 규칙
- 실제 조직 역할 구조(운영팀, CS팀, 관리자 등)

### 단계 2: AI 초안 생성

프롬프트 예시:
```
아래 PRD 기능 목록과 정책 기준으로 RBAC 기반 권한 매트릭스를 작성해줘.

역할: 비인증 사용자 / 일반 회원 / 운영자 / 관리자
기능: 회원가입, 로그인, 내 정보 조회, 내 정보 수정, 계정 잠금 해제,
      타 회원 조회, 역할 변경, 감사 로그 조회

정책 원칙:
- 본인 데이터는 본인만 수정 가능
- 운영자는 회원 데이터 조회만 허용, 수정 불가
- 탈퇴 후 데이터는 운영자 접근 제한
```

### 단계 3: PM 확정 기준

| 항목 | 이유 |
|------|------|
| 역할 분류 확정 | 실제 조직 구조와 운영 방식에 맞춰야 함 |
| 민감 데이터 마스킹 범위 | 법무·컴플라이언스 검토 필요 |
| 조건부 권한 기준 | "제한적 조회"의 구체적 범위를 PM이 결정 |
| 역할 부여·회수 프로세스 | 운영 절차와 연동 필요 |

## 5. 품질 기준 / 체크리스트

| 항목 | 확인 기준 |
|------|-----------|
| 역할 완결성 | 시스템에 존재하는 모든 사용자 유형이 역할로 정의됐는가 |
| 기능 커버리지 | PRD의 모든 주요 기능이 매트릭스에 포함됐는가 |

| 조건부 권한 명시 | "가능" / "불가" 외 조건부 접근이 있는 항목이 명확히 정의됐는가 |
| 데이터 마스킹 | 민감 데이터(개인정보·결제)의 역할별 접근 범위가 정의됐는가 |
| 정책서 연결 | 정책서의 역할 관련 규칙과 매트릭스가 일치하는가 |
| 승인 상태 | Draft / Review / Approved 상태가 표시됐는가 |

## 6. 다음 단계 연결 규칙

- **사양서**: 기능별 API 엔드포인트에 권한 체크 조건 반영
- **API 명세서**: 각 API의 허용 역할, 401/403 응답 케이스 정의
- **테스트케이스**: 역할별 접근 가능/불가 케이스, 권한 경계 케이스 생성
- **운영 정책서(OPD)**: 역할 부여·변경·회수 운영 절차와 연동

---

## 7. 이 장의 핵심 메시지

> 권한은 보안 담당자가 나중에 결정하는 것이 아니라, 정책서와 함께 설계되어야 하는 비즈니스 규칙이다.

"누가 무엇을 할 수 있는가"에 대한 합의가 없으면 사양서와 테스트케이스에 공백이 생기고, 개발 완료 후 보안 리뷰에서 재설계가 발생한다. 권한 정책 매트릭스는 이 공백을 기획 단계에서 채운다.

---

## 8. 다음 장으로의 연결

권한 매트릭스가 확정되면 "어떤 데이터를 어떻게 구조화해야 이 권한 체계를 구현할 수 있는가"를 정의할 차례다. 역할별 접근 범위는 ERD의 엔티티 설계와 필드 속성에 직접 반영된다.

### 권한 정책에서 데이터 모델로 이동하는 항목

| 권한 정책 항목 | 이동 형태 | 데이터 모델에서의 역할 |
|---|---|---|
| 역할별 데이터 접근 범위 | 역할-데이터 범위 매핑 | ERD 엔티티의 접근 제어 속성 |
| 데이터 소유자 정의 | 엔티티-역할 매핑 | ERD owner 관계 및 FK 설계 |
| 특정 데이터 숨김·마스킹 규칙 | 마스킹 적용 필드 목록 | ERD 필드 속성 주석 |
| 역할 테이블 구조 | Role, Permission 관계 | RBAC 구현을 위한 Role/Permission 엔티티 |
| 운영자 전용 데이터 항목 | 관리자 전용 필드 목록 | ERD 접근 제한 필드 분리 |

### 체인 전환 기준 (권한 정책 → 데이터 모델)

| 확인 항목 | 기준 |
|---|---|
| 역할 목록 확정 | 시스템에서 사용할 모든 역할이 권한 매트릭스에 정의되어 있는가 |
| 데이터 접근 범위 명시 | 각 역할이 접근할 수 있는 데이터 범위(본인/팀/전체)가 기재되어 있는가 |
| 마스킹 항목 식별 | 역할에 따라 숨기거나 마스킹할 데이터 항목이 식별되어 있는가 |
| 권한 정책 상태 | `Approved` 상태인가 |

- **이 장이 결정한 것**: 역할 목록, 기능별 접근 권한, 데이터 접근 범위, 역할 상속 구조
- **다음 장이 시작하는 것**: 권한 범위를 반영한 정보 구조(IA), ERD, 상태 모델 설계
