# 5장. 도구 환경 셋팅: Obsidian, VS Code, AI, Git

## 이 장의 목적

이 장은 이 책에서 다루는 방법론을 실제로 실행하기 위해 어떤 도구가 왜 필요한지, 그리고 어떻게 연결해야 하는지를 설명한다.

도구를 먼저 설치하는 것이 목적이 아니다. 산출물 체인이 작동하려면 각 단계의 문서가 어디에서 관리되고, 누가 검토하며, 어떤 AI가 어느 단계에서 투입되는지가 명확해야 한다. 이 장은 그 흐름을 가능하게 하는 **최소 도구 구성과 연결 방식**을 다룬다.

이 장을 읽고 나면 독자는 다음을 할 수 있어야 한다.

- 왜 단일 도구가 아니라 도구 조합이 필요한지 설명할 수 있다.
- Obsidian, VS Code, AI, Git 각각의 역할과 경계를 이해한다.
- 이 네 가지를 어떻게 연결해서 산출물 체인을 운영하는지 설명할 수 있다.
- 이 책에서 이후 다룰 RIB, PRD, 정책서, BPMN, DMN, 사양서, 테스트케이스가 어떤 환경에서 만들어지는지 맥락을 잡는다.

---

## 1. 왜 도구 구성부터 이야기하는가

산출물 체인에서 다루는 문서는 기획자 혼자 완성하는 것이 아니다.

- 기획자가 초안을 잡고
- AI가 초안을 빠르게 전개하고
- PM이 검토하고 수정하며
- 팀이 승인하고 버전을 관리한다

이 흐름이 무너지면 문서는 만들어지지만 산출물 체인은 작동하지 않는다. 가장 자주 무너지는 이유는 도구가 없어서가 아니라 **도구가 연결되지 않아서**다.

노션에서 작성한 PRD를 슬랙에서 공유하고, 개발자가 지라에서 보고, QA가 구글 시트로 테스트케이스를 관리하면 — 내용은 있지만 체인이 없다. 어느 문서가 최신인지, 어떤 버전이 승인됐는지, 무엇이 변경됐는지를 아무도 한눈에 보지 못한다.

이 책이 제안하는 환경은 단순하다.

> **Obsidian으로 지식을 쌓고 → VS Code에서 산출물을 만들고 → AI가 초안과 변환을 담당하고 → Git으로 이력과 승인을 관리한다**

이 네 도구는 각자 최선이 다르고, 연결하면 각자의 약점을 보완한다.

---

## 2. 네 도구의 역할과 경계

### 2-1. Obsidian: 지식 축적과 컨텍스트 관리

Obsidian은 로컬 마크다운 파일 기반의 노트 앱이다. 이 책에서 Obsidian이 담당하는 것은 다음이다.

- **프로젝트 컨텍스트 관리**: 배경, 이해관계자, 제약사항, 의사결정 로그, 회의록
- **지식 베이스**: 용어집, 정책 원칙, 참조 문서, 도메인 지식
- **연결망 시각화**: 문서 간 참조 관계를 Canvas와 Graph View로 파악

Obsidian은 문서를 많이 쌓는 것이 목적이 아니다. **AI가 산출물을 만들기 전에 읽어야 할 컨텍스트를 정리해두는 곳**이다. RIB를 작성하기 전에, PRD를 전개하기 전에, AI에게 무엇을 알려줘야 하는지를 정리하는 공간이다.

Obsidian이 담당하지 않는 것도 있다. 최종 승인 산출물은 Obsidian에서 관리하지 않는다. 초안, 메모, 참조 자료는 Obsidian에서 관리하지만, 팀 전체가 기준으로 삼는 확정 문서는 Git에서 관리한다.

### 2-2. VS Code: 산출물 작업 공간

VS Code는 이 책에서 **산출물을 직접 작성하고 AI와 함께 편집하는 작업 공간**이다.

- **마크다운 산출물 작성**: PRD, 정책서, 사양서, 테스트케이스 등 실제 문서 작성
- **AI 확장 연결**: GitHub Copilot, Claude for VS Code 등 AI 도구와 직접 연동
- **파일 구조 관리**: 산출물 폴더 구조, 네이밍 규칙 적용
- **Git 연동**: VS Code 내에서 커밋, 브랜치, PR 생성까지 처리

Obsidian과 VS Code의 경계는 이렇게 나눈다.

| 구분 | Obsidian | VS Code |
|------|----------|---------|
| 목적 | 컨텍스트 축적 | 산출물 생산 |
| 파일 위치 | 볼트 내 노트 | 프로젝트 리포지토리 |
| 관리 방식 | 개인 지식 관리 | 팀 버전관리 |
| AI 역할 | RAG 컨텍스트 소스 | 직접 편집·생성 파트너 |

### 2-3. AI: 초안 생성과 변환 파트너

이 책에서 AI는 산출물을 대신 만드는 것이 아니라 **기획자가 정한 범위와 기준 위에서 초안을 전개하고 변환하는 파트너**다.

AI가 담당하는 작업은 다음이다.

- **초안 생성**: 입력 조건(RIB, 컨텍스트)을 기반으로 PRD, 정책서, BPMN 초안 작성
- **구조화**: 자연어 설명을 표, 목록, 분기 구조로 변환
- **점검**: 누락, 중복, 모순, 비약 감지
- **변환**: PRD → 테스트케이스 초안, 정책서 → DMN 표, BPMN → 사양서 섹션

AI가 담당하지 않는 것은 명확하다.

- 범위 결정과 우선순위 판단
- 정책 충돌 해소와 예외 처리 확정
- 이해관계자 간 합의와 승인
- Source of Truth 지정과 버전 승인

AI는 기획자의 판단을 빠르게 문서로 만드는 도구다. AI가 결정을 내리는 것이 아니라, 기획자의 결정을 구조화하고 문서화한다.

### 2-4. Git: 이력, 승인, 협업의 뼈대

Git은 코드 관리 도구로 알려져 있지만, 이 책에서는 **산출물의 버전 관리, 승인 이력, 팀 협업을 위한 운영 뼈대**로 사용한다.

Git이 제공하는 것은 다음이다.

- **버전 이력**: 누가 언제 무엇을 바꿨는지 모두 기록됨
- **브랜치**: 초안 작업과 승인된 기준 버전을 분리
- **Pull Request**: 변경 검토와 승인 프로세스를 명시화
- **CODEOWNERS**: 문서별 검토 책임자를 코드로 정의
- **Actions**: 문서 구조 검증, 링크 점검 등 자동 검사

파일명에 날짜를 붙이거나 폴더를 `최종`, `최최종`으로 나누는 방식은 버전관리가 아니다. Git은 그 혼선을 구조적으로 없앤다.

## 3. 네 도구의 연결 방식

### 도구 연결 구조

```
[Obsidian]                    [VS Code]
컨텍스트 축적                 산출물 작성
- 배경·제약                   - PRD, 정책서, BPMN
- 용어집                      - 사양서, TC
- 의사결정 로그               - 하네스 카드
       ↓                           ↓
  AI 컨텍스트 주입 →→→→→ [AI] ←←← 산출물 초안 요청
                                   ↓
                              초안 생성 + 점검
                                   ↓
                          [Git 리포지토리]
                          버전관리 + PR + 승인
```

구체적인 흐름은 다음과 같다.

1. **프로젝트 시작**: Obsidian에 배경, 이해관계자, 제약, 의사결정 컨텍스트 축적
2. **RIB 작성**: VS Code에서 AI에게 Obsidian 컨텍스트를 제공하고 RIB 초안 생성
3. **검토와 확정**: PM이 RIB 검토 → Git으로 커밋 → PR로 승인
4. **PRD 전개**: 승인된 RIB를 기반으로 AI가 PRD 초안 전개
5. **체인 반복**: 정책서 → BPMN → DMN → 사양서 순으로 동일 패턴 반복
6. **운영 반영**: 테스트, 운영, 피드백 → Obsidian에 학습 내용 축적 → 다음 RIB로 입력

## 4. 이 환경이 전제하는 것과 갖춰주는 것

산출물 장과 달리 환경 설정은 입력/출력 구조가 다르다. 이 장이 전제하는 최소 조건과, 이 장을 마치면 갖춰지는 것을 명확히 정리한다.

**이 장이 전제하는 것:**

- 산출물 체인을 실제로 운영하려는 의지가 있다 (단순 탐색 목적이 아님)
- Obsidian, VS Code, Git 중 최소 하나는 이미 써본 경험이 있다
- 프로젝트 단위로 문서를 관리할 공간(리포지토리 또는 폴더)이 있다

**이 장이 갖춰주는 것:**

| 갖춰지는 것 | 효과 |
|------------|------|
| 도구별 역할 경계 | Obsidian/VS Code/Git 중복 사용 혼선 제거 |
| 도구 연결 흐름 | 컨텍스트 → 산출물 → 승인의 단방향 흐름 확보 |
| AI 투입 위치 | 어느 단계에서 AI를 쓰는지 명확해짐 |
| 최소 셋업 기준 | 완벽하지 않아도 체인 시작 가능한 조건 |

## 5. 최소 셋업 기준

처음부터 완벽한 환경을 구성할 필요는 없다. 다음 최소 기준을 먼저 갖추는 것으로 시작한다.

### 4-1. Obsidian 셋업

- 볼트 생성 후 폴더 구조 최소화: `00.context/`, `01.project/`, `02.reference/`
- 템플릿 폴더 설정: 자주 쓰는 회의록, 의사결정 로그, 용어 카드 템플릿 준비
- Canvas 활성화: 문서 연결 관계 시각화에 사용

### 4-2. VS Code 셋업

- 마크다운 편집 확장 설치: Markdown All in One, Markdown Preview Enhanced
- AI 확장 설치: GitHub Copilot 또는 Claude for VS Code
- Git 연동 확인: Source Control 패널에서 리포지토리 연결 확인

### 4-3. Git 리포지토리 셋업

- 산출물 리포지토리 생성 (GitHub 권장)
- 브랜치 보호 규칙 설정: `main` 브랜치에 직접 push 금지, PR 필수
- CODEOWNERS 파일 작성: 산출물 유형별 검토 책임자 지정
- 폴더 구조 초기화: `docs/`, `policy/`, `spec/`, `testcase/` 등 산출물 유형별 분리

### 4-4. AI 연결 방식

이 책에서는 특정 AI 서비스를 고정하지 않는다. 다음 방식 중 프로젝트 상황에 맞게 선택한다.

| 방식 | 특징 | 적합한 상황 |
|------|------|------------|
| VS Code 내 AI 확장 | 파일 직접 편집, 컨텍스트 자동 인식 | 문서 초안 생성, 구조화 |
| 채팅 인터페이스 (Claude, GPT) | 긴 컨텍스트 제공, 대화형 검토 | 정책 검토, 모순 점검, 변환 |
| API 기반 자동화 | 반복 변환 자동화, CI 연동 | 테스트케이스 생성, 구조 검증 |

## 6. 이 환경이 만드는 것

이 네 도구가 연결되면 다음이 가능해진다.

- **기획자는 판단에 집중한다**: 초안 작성 시간을 AI에게 넘기고, 검토와 확정에 집중
- **산출물이 연결된다**: RIB → PRD → 정책서 → 사양서가 같은 리포지토리 안에서 참조 관계를 유지
- **변경이 추적된다**: 누가 언제 무엇을 바꿨는지, 어떤 이유로 승인됐는지 모두 기록
- **AI 입력 품질이 유지된다**: Obsidian의 정리된 컨텍스트가 AI의 입력 품질을 높인다

## 7. 환경 완성 체크리스트

다음 항목이 모두 갖춰지면 산출물 체인을 시작할 준비가 된 것이다.

| 항목 | 확인 기준 |
|------|-----------|
| Obsidian 볼트 | 프로젝트별 컨텍스트 폴더가 분리되어 있다 |
| VS Code 연결 | 산출물 리포지토리가 VS Code에서 열리고 마크다운 편집이 가능하다 |
| AI 확장 | VS Code에 AI 확장이 설치·연결되어 있거나, 채팅 인터페이스를 병행할 준비가 됐다 |
| Git 리포지토리 | 산출물 저장소가 생성됐고, main 브랜치에 PR 없이 직접 push가 금지됐다 |
| CODEOWNERS | 핵심 산출물 폴더에 검토 책임자가 지정됐다 |
| 폴더 구조 | `docs/`, `policy/`, `spec/`, `testcase/` 등 산출물 유형별 폴더가 초기화됐다 |
| 도구 경계 합의 | 팀 내에서 어느 도구에 어떤 문서를 두는지 간단히 합의됐다 |

이 중 Git과 CODEOWNERS는 혼자 시작할 경우 나중에 추가해도 된다. 그러나 팀 단위라면 리포지토리 구조와 CODEOWNERS는 초기에 잡지 않으면 나중에 고치기 어렵다.

## 8. 이 장의 핵심 메시지

> 도구는 산출물 체인이 작동하도록 돕는 기반이다.
> 중요한 것은 도구 하나의 성능이 아니라, 네 도구가 어떻게 연결되어 기획자의 판단을 빠르게 문서로 만들고 관리할 수 있는가다.

Obsidian이 컨텍스트를 쌓고, VS Code에서 산출물을 만들고, AI가 초안과 변환을 담당하고, Git이 버전과 승인을 관리한다. 이 연결이 이 책이 제안하는 작업 방식의 기반이다.

## 9. 다음 장으로의 연결

이 장에서 네 도구의 역할과 연결 방식을 정리했다. 환경이 갖춰지면 다음 질문이 자연스럽게 나온다.

> 그렇다면 어떤 문서를 어떤 층에 배치해야 하는가?

도구를 연결해도 문서의 위계와 책임이 정리되지 않으면 혼선은 도구 안에서 반복된다. 다음 장(6장)에서는 문서를 7개 레이어로 나누고 각 레이어의 역할과 책임을 어떻게 분리해야 하는지를 다룬다.

### 이 장에서 다음 장으로 이어지는 것

| 이 장에서 갖춰지는 것 | 다음 장이 이것을 전제하는 이유 |
|---|---|
| Obsidian 볼트 구조 (폴더 생성) | 6장의 레이어 구조를 실제 폴더에 배치하려면 볼트가 먼저 있어야 한다 |
| VS Code + Git 연결 | 6장에서 정의하는 레이어별 책임이 버전 관리와 연결되어야 운영된다 |
| AI 컨텍스트 파일 경로 설정 | 7장에서 다룰 Source of Truth 범위를 AI에게 넘기는 경로가 여기서 만들어진다 |
| 도구 간 데이터 흐름 확인 | 레이어 구조를 설계하기 전에 파일이 어느 도구에서 어떻게 이동하는지 알아야 한다 |

- **이 장(5장)이 갖춰주는 것**: 작업 환경 (Obsidian, VS Code, AI, Git 연결)
- **다음 장(6장)이 정의하는 것**: 문서 레이어 구조와 책임 경계
- **그 다음(7장)이 결정하는 것**: Source of Truth와 버전 관리 원칙

이 세 장이 함께 이 책의 작업 방식의 기반을 구성한다.
