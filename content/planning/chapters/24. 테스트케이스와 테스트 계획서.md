# 24장. 테스트케이스와 테스트 계획서

> 이 장은 두 산출물을 함께 다룬다. 테스트케이스는 설계가 검증 가능한 상태인지 확인하는 연결 문서다. 테스트 계획서는 무엇을, 어떤 방식으로, 언제, 누가 검증할 것인지를 정의하는 설계도다. 테스트케이스가 무엇인지 정의하고, 테스트 계획서가 그것을 어떤 체계로 운영할지를 정한다.

---

## 제1절. 테스트케이스


> **이 장에서 다루는 것**
> - 테스트케이스의 역할과 AC·DMN과의 1:1 관계
> - AI와 함께 TC를 생성·검증하는 3단계
> - TC → DoD·운영 정책서 환류 연결 기준

> **전제 지식**: 21장(사양서), 17장(BPMN과 DMN) — 테스트케이스는 사양서의 구현 조건, BPMN의 흐름 분기, DMN의 판단 규칙 각각에 대응하는 검증 항목을 만드는 문서다. 세 산출물을 먼저 이해해야 TC 커버리지를 설계할 수 있다.

## 이 장의 목적

이 장은 PRD, 정책서, BPMN, DMN, 사양서, 프로토타입에서 정의된 내용을 실제로 검증 가능한 기준으로 바꾸는 테스트케이스의 역할을 설명한다.

이 책에서 테스트케이스는 QA만 보는 마지막 문서가 아니다. 설계가 충분히 검증 가능한지 확인하는 연결 문서다.

이 장을 읽고 나면 아래를 이해해야 한다.

- Acceptance Criteria와 테스트케이스의 차이
- BPMN, DMN, 사양서와 테스트케이스를 연결하는 방법
- 정상, 예외, 판단 규칙, 상태 전이를 어떻게 검증 항목으로 바꾸는가
- QA뿐 아니라 PM, 개발, 운영도 왜 테스트케이스를 봐야 하는가
- AI를 활용해 테스트케이스 초안을 만들 때 무엇을 맡기고 무엇을 직접 판단해야 하는가

## 1. 이 산출물이 왜 존재하는가

많은 팀이 테스트케이스를 구현 막바지에 만든다. 그러면 설계 문제와 구현 문제가 뒤섞인다.

예를 들어 로그인 정책에서 아래가 모호하다고 하자.

- 실패 누적 기준이 세션 기준인지 계정 기준인지
- 잠금 해제 조건이 시간 경과인지 본인 확인인지
- 화면 상태에서 사용자에게 어떤 안내를 먼저 보여줘야 하는지

이런 문제는 QA 단계에서 버그처럼 보이지만, 실제로는 설계 빈틈인 경우가 많다. 따라서 테스트케이스는 구현 완료 후의 검수 문서가 아니라, 설계가 검증 가능한 상태인지 확인하는 조기 경보 장치다.

이 책에서 테스트케이스는 아래처럼 정의한다.

> 테스트케이스는 특정 기능 또는 시나리오가 어떤 전제 조건에서 어떤 입력을 받아 어떤 결과를 내야 하는지 명시해, 설계와 구현이 요구사항을 충족하는지 검증할 수 있게 만드는 문서다.

핵심은 두 가지다.

- 테스트케이스는 버그 찾기 문서 이전에 설계 검증 문서다.
- 테스트 결과만 기록하는 문서가 아니라 기대 결과를 구조화하는 문서다.

## 2. 무엇을 입력으로 받고 무엇을 다음 단계로 넘기는가

### 2-1. TC 작성에 필요한 입력

| 입력 산출물 | 역할 |
|---|---|
| AC (Acceptance Criteria) | 인수 기준 — 무엇이 충족되어야 하는지 |
| 정책서 | 허용·차단·예외 규칙 — 판단 기준 |
| BPMN | 정상·예외·운영 개입 흐름 — 경로 기준 |
| DMN | 조건 조합별 판단 결과 — 규칙 행 → TC 행 1:1 대응 |
| 사양서(Spec) | 상태값·메시지·API 응답·UI 조건 — 검증 기준 |
| 프로토타입 | 사용자 경험 흐름 — 화면 검증 기준 |
| 테스트 계획서(TP) | 범위·우선순위·환경·일정 제약 |

### 2-2. TC가 넘기는 출력

| 출력 산출물 | 용도 |
|---|---|
| 테스트케이스(TC) | QA 실행 기준 |
| RTM | 요구사항-TC 추적 매트릭스 |
| DoD (Definition of Done) | 기능 완료 기준 충족 여부 |
| 결함 보고서 | 검증 실패 기록 |
| 운영 정책서 | TC 기반 운영 이슈 패턴 반영 |

### 2-3. DMN 행 → TC 자동 도출

17장에서 확립한 원칙이 여기서 실현된다. DMN 의사결정 테이블의 규칙 행 하나는 TC 하나에 직접 대응한다.

예: 로그인 DMN-01 기준

| DMN 규칙 행 | TC ID | 조건 | 기대 결과 |
|---|---|---|---|
| 규칙 1 | TC-001 | 정상 계정 + 올바른 비밀번호 | 로그인 성공 |
| 규칙 2 | TC-002 | 정상 계정 + 잘못된 비밀번호 + 실패 4회 이하 | 실패 카운트 증가, 로그인 차단 |
| 규칙 3 | TC-003 | 정상 계정 + 잘못된 비밀번호 + 실패 5회 | 계정 잠금 전이, 잠금 안내 문구 노출 |
| 규칙 4 | TC-004 | 잠금 계정 + 어떤 비밀번호든 | 잠금 메시지 노출, 로그인 차단 |
| 규칙 5 | TC-005 | 잠금 계정 + 본인 확인 완료 | 잠금 해제, 로그인 정상 허용 |

이 방식으로 DMN-01(8행), DMN-02(6행), DMN-03(5행) = TC 19개를 자동으로 도출한다.

## 3. TC 구조와 마크다운 템플릿

### 3-1. TC에 반드시 들어가야 하는 항목

| 항목 | 내용 |
|---|---|
| TC ID | TC-[기능코드]-[순번] (예: TC-LOGIN-001) |
| 기능명 / 시나리오명 | 어떤 기능의 어떤 흐름인지 |
| 참조 산출물 | PRD 항목 ID / 정책 ID / BPMN 단계 ID / DMN 규칙 ID / Spec 섹션 |
| 전제 조건 | 사용자 상태, 계정 상태, 데이터 상태, 사전 처리 여부 |
| 입력 / 실행 절차 | 입력값, 사용자 액션, 시스템 이벤트, 외부 인터페이스 조건 |
| 기대 결과 | 화면 결과, 메시지, 상태 전이, 데이터 변경, 로그, API 결과 |
| 유형 분류 | 정상 / 예외 / 경계값 / 보안 / 운영 / 회귀 |
| 우선순위 | 높음 / 중간 / 낮음 |

### 3-2. 마크다운 TC 템플릿

```markdown
## [TC-LOGIN-001] 정상 계정 로그인 성공

| 항목 | 내용 |
|---|---|
| TC ID | TC-LOGIN-001 |
| 기능명 | 로그인 |
| 시나리오 | 정상 계정 + 올바른 비밀번호 → 로그인 성공 |
| 유형 | 정상 |
| 우선순위 | 높음 |

**참조 산출물**

- PRD: F-AUTH-01 (로그인 기능)
- 정책: P-01 (로그인 허용 기준)
- BPMN: BPMN-01 / Task-03 (인증 처리)
- DMN: DMN-01 / 규칙 1
- Spec: SEC-01 (상태 코드 200, 세션 발급)

**전제 조건**

- 계정 상태: 정상 (active)
- 실패 횟수: 0회
- 비밀번호 유효: 만료되지 않음

**입력 / 실행**

1. 이메일 입력: user@example.com
2. 비밀번호 입력: 올바른 비밀번호
3. 로그인 버튼 클릭

**기대 결과**

- 화면: 홈 화면으로 전환
- 상태: 세션 생성, 로그인 상태 전이
- 데이터: 마지막 로그인 시각 갱신
- 로그: 로그인 성공 이벤트 기록
- API: 200 OK + 세션 토큰 발급

---

## [TC-LOGIN-003] 5회 실패 → 계정 잠금 전이

| 항목 | 내용 |
|---|---|
| TC ID | TC-LOGIN-003 |
| 기능명 | 로그인 |
| 시나리오 | 실패 횟수 4회 상태 → 1회 추가 실패 → 잠금 전이 |
| 유형 | 예외 / 경계값 |
| 우선순위 | 높음 |

**참조 산출물**

- 정책: P-03 (반복 실패 잠금 기준)
- DMN: DMN-01 / 규칙 3
- Spec: ERR-03 (잠금 안내 문구)

**전제 조건**

- 계정 상태: 정상 (active)
- 실패 횟수: 4회

**입력 / 실행**

1. 이메일 입력: user@example.com
2. 비밀번호 입력: 잘못된 비밀번호
3. 로그인 버튼 클릭

**기대 결과**

- 실패 횟수: 5회로 증가
- 계정 상태: locked 전이
- 화면 메시지: "로그인 5회 실패로 계정이 잠금되었습니다. 본인 확인 후 해제하세요."
- 로그: 잠금 이벤트 기록
- API: 403 + 잠금 오류 코드
```

### 3-3. TC 유형별 커버리지 기준

| 유형 | 정의 | 예시 |
|---|---|---|
| 정상 | 의도된 성공 경로 | 정상 로그인 성공 |
| 예외 | 정책·시스템 오류 처리 경로 | 잠금 계정 접근 차단 |
| 경계값 | 임계값 직전/당/직후 | 실패 4회, 5회, 6회 |
| 보안 | 비정상 접근·권한 위반 | 타인 계정 강제 접근 |
| 운영 | 운영자 개입 필요 경로 | 잠금 해제 수동 처리 |
| 회귀 | 정책 변경 후 재검증 필수 | 잠금 해제 조건 변경 후 |

## 4. AI와 함께 만드는 법

### 4-1. 1단계: DMN 행 → TC 초안 변환

DMN 의사결정 테이블을 그대로 입력한다. AI는 각 규칙 행을 TC 형식으로 변환한다.

**입력 프롬프트:**
```
아래 DMN 의사결정 테이블을 TC 마크다운 템플릿으로 변환해줘.
각 규칙 행이 TC 1개가 되어야 하고,
전제 조건 / 입력 / 기대 결과 / 유형 분류가 포함되어야 해.

[DMN 테이블 붙여넣기]
```

**AI 담당:** DMN 행 기반 TC 행 변환, 유형 분류(정상/예외) 초안

**PM 확인:**
- 기대 결과의 메시지 문구가 사양서와 일치하는지
- API 응답 코드가 Spec과 맞는지
- 운영 알림이 필요한 케이스인지

### 4-2. 2단계: BPMN 경로 기반 누락 케이스 보완

BPMN 흐름도의 예외 경로, 운영 개입 경로를 입력한다. AI가 DMN 변환에서 누락된 흐름 케이스를 추가 제안한다.

**입력 프롬프트:**
```
아래는 로그인 BPMN 흐름의 경로 목록이야.
기존 TC 목록과 비교해서 누락된 흐름 케이스를 찾아줘.
특히 예외 경로, 운영 개입 경로, 역할 전환 지점을 중심으로 봐줘.

[BPMN 경로 목록]
[기존 TC 목록]
```

**AI 담당:** 누락 경로 후보 제안, 경계값 케이스 확장

**PM 확인:**
- 제안된 케이스의 실제 구현 가능 여부
- 테스트 비용 대비 중요도 판단
- 보안상 치명적인 분기 우선순위 지정

### 4-3. 3단계: 회귀 TC 묶음 추출

정책 변경이나 기능 변경 이후 반드시 재검증해야 하는 핵심 케이스 세트를 분리한다.

**입력 프롬프트:**
```
아래 TC 목록 중 정책 변경(잠금 기준, 비밀번호 만료 기준)이나
UI 변경 이후에 반드시 다시 실행해야 하는 TC를 회귀 세트로 분류해줘.
분류 기준은 정책 의존도와 상태 전이 여부야.

[TC 목록]
```

**AI 담당:** 회귀 TC 후보 분류, 정책 의존도 분석

**PM 확인:**
- 릴리즈 주기 대비 회귀 TC 실행 가능 규모
- 자동화 우선 케이스 지정

> **→ 프롬프트 예시**: [[wiser-prompt-examples-v1]] — 13. 테스트케이스 생성 예시

## 5. 품질 기준과 체크리스트

### 5-1. 필수 점검 — TC가 존재하는가

- [ ] DMN 규칙 행 전체가 TC로 변환되었는가
- [ ] BPMN의 정상 경로, 예외 경로, 운영 개입 경로 각각 TC가 있는가
- [ ] Acceptance Criteria 항목마다 대응하는 TC가 있는가
- [ ] 사양서의 상태값·메시지·API 응답이 기대 결과로 명시되어 있는가
- [ ] 경계값(임계값 직전·당·직후)이 포함되어 있는가

### 5-2. 품질 향상 — TC가 충분히 구체적인가

- [ ] 기대 결과에 화면 문구, 상태값, 로그 기록 여부가 명시되어 있는가
- [ ] 전제 조건이 재현 가능한 수준으로 구체적인가
- [ ] 유형 분류(정상/예외/경계값/보안/운영/회귀)가 정확한가
- [ ] 참조 산출물 ID(PRD, 정책, BPMN, DMN, Spec)가 연결되어 있는가
- [ ] 회귀 TC 묶음이 별도로 구분되어 있는가

### 5-3. 오류 유형 점검 — TC가 잘못 작성되지 않았는가

- [ ] "정상적으로 동작해야 한다"처럼 기대 결과가 모호한 TC가 없는가
- [ ] 전제 조건 없이 결과만 기술된 TC가 없는가
- [ ] 동일한 케이스가 중복으로 작성된 TC가 없는가
- [ ] 실제 구현 불가능하거나 환경 의존도가 너무 높은 TC가 없는가

## 6. 다음 단계 연결 규칙

### 6-1. RTM으로 연결

TC 작성이 완료되면 RTM(Requirements Traceability Matrix)을 갱신한다. RTM은 요구사항 ID → TC ID → 결과 상태를 추적한다. TC가 없는 요구사항은 RTM에서 누락으로 표시된다.

**연결 조건:** 모든 PRD Feature 항목에 TC가 1개 이상 대응되어야 RTM에 등록 가능하다.

### 6-2. DoD로 연결

기능 완료 기준(DoD)의 TC 항목은 아래 조건에서 충족된다.

- 해당 기능의 TC 전체가 작성 완료 상태
- 핵심 TC(P1) 통과율 100%
- 예외·경계값 TC 통과율 기준(팀 정의)을 충족

**연결 조건:** DoD TC 항목 미충족 시 기능 완료 처리 불가.

### 6-3. 운영 정책서로 환류

TC 실행 과정에서 발견된 예외 패턴은 운영 정책서로 환류한다. 반복 발생하는 예외는 정책 규칙으로 공식화한다.

**연결 조건:** 동일 유형의 TC 실패가 2회 이상 반복되면 운영 정책 검토 대상으로 등록한다.

### 6-4. 병렬 가능 산출물

TC 작성과 병렬로 진행 가능한 산출물:
- 테스트 계획서(TP): 범위, 우선순위, 환경, 일정
- 사양서(Spec): TC 결과로 Spec 내용 보완 가능

### 6-5. TC 비담당 영역

아래는 TC에서 결정하지 않는다.
- 기능 범위 변경 → PRD
- 정책 예외 추가 → 정책서
- 구현 방식 변경 → 개발팀
- 배포 승인 → DoD + 릴리즈 정책

---

## 7. Acceptance Criteria와 테스트케이스의 차이

둘은 가깝지만 역할이 다르다.

**Acceptance Criteria**는 기능이 수용되기 위한 조건이다. 요구사항 수준의 인수 기준을 비교적 간결한 문장으로 적는다.

예:
- 5회 로그인 실패 시 계정은 잠금 상태가 된다.
- 잠금된 계정은 본인 확인 이후에만 해제할 수 있다.

**테스트케이스**는 그 인수 기준을 어떤 조건과 입력으로 검증할지 적는 문서다. 전제 조건, 입력, 실행, 기대 결과를 포함하고 상태 변화와 예외를 더 구체적으로 다룬다.

예:
- 전제: 실패 횟수 4회인 계정 상태
- 입력: 잘못된 비밀번호로 로그인 시도
- 기대 결과: 실패 횟수 5회 기록, 계정 상태 잠금 전이, 잠금 안내 문구 노출

즉 AC는 무엇이 충족되어야 하는가이고, TC는 그것을 어떻게 확인할 것인가다.

---

## 8. TC를 실제로 만드는 순서

### 8-1. 시나리오 기준으로 큰 묶음을 만든다

먼저 기능이 아니라 시나리오 기준으로 테스트 묶음을 만든다.

- 정상 로그인
- 반복 실패
- 잠금
- 잠금 해제
- 비밀번호 만료
- 오류 복구

### 8-2. DMN 행 → TC 자동 도출

DMN 의사결정 테이블의 각 규칙 행을 TC 1개로 변환한다. 이 작업은 AI에 맡긴다.

### 8-3. BPMN 경로 기준으로 흐름 케이스 보완

정상 경로, 예외 경로, 운영 개입 경로를 BPMN 기준으로 분리하고, DMN 변환에서 누락된 흐름 케이스를 추가한다.

### 8-4. 사양서 기준으로 메시지와 상태 케이스 확인

화면 문구, 상태값, API 응답, 로그 기록 여부까지 기대 결과에 반영한다.

### 8-5. 회귀 TC 묶음을 별도 분리

정책 변경이나 UI 변경 이후에 반드시 다시 봐야 하는 핵심 테스트 세트를 구분한다.

## 10. QA만의 문서로 보면 안 되는 이유

좋은 테스트케이스는 QA만 보는 문서가 아니다.

- PM/PO는 요구사항이 검증 가능한지 확인한다.
- 개발자는 구현 누락과 예외 조건을 미리 본다.
- 디자이너는 프로토타입과 실제 결과가 일치하는지 본다.
- 운영은 어떤 이슈가 사용자에게 어떻게 보일지 예측한다.

즉 테스트케이스는 팀 전체의 공통 검증 언어다.

---

## 11. 21장, 23장, 24장을 연결해서 보면

이 세 장은 각각 독립 문서가 아니라 하나의 체인이다.

- 21장 사양서는 구현 조건을 고정한다.
- 23장 프로토타입은 그 조건이 사용자에게 어떻게 보이는지 검토한다.
- 24장 테스트케이스는 그 조건과 경험이 실제 구현 결과에서 맞는지 검증한다.

즉 21장은 무엇을 구현할 것인가, 23장은 어떻게 보일 것인가, 24장은 실제로 맞는가를 다룬다.

## 12. OPD(운영 정책서): TC 결과가 갱신하는 산출물

캔버스 산출물 체인에서 OPD(Operational Policy Document, 운영 정책서)는 TC 다음에 위치한다. TC 실행 이후 발견된 예외 패턴, 반복 실패 케이스, 운영 개입 필요 상황이 그대로 흘려보내지지 않고 운영 정책으로 공식화되는 것이 이 연결의 핵심이다.

OPD는 별도의 대규모 문서가 아니다. 정책서의 운영 실행 버전으로, "이 예외가 발생했을 때 누가 어떻게 처리하는가"를 규칙으로 남기는 문서다. TC가 없으면 OPD는 이론적 규칙만 나열하게 되고, OPD가 없으면 TC에서 발견된 예외가 구두 합의로 사라진다.

### 12-1. TC → OPD 환류가 필요한 상황

아래 상황이 발생하면 OPD 갱신 트리거로 본다.

- 동일 유형의 TC 실패가 2회 이상 반복된다
- TC 실행 중 정책서에 없는 예외 케이스가 발견된다
- "운영 개입 필요" 케이스가 TC 결과에 처음 등장한다
- 기존 운영 규칙과 실제 TC 기대 결과가 충돌한다

이 상황이 확인되면 TC 완료 후 OPD 갱신 검토를 별도 작업으로 올린다.

---

### 12-2. AI와 함께 만드는 법

OPD 갱신은 TC 결과에서 시작해 세 단계로 진행한다.

**Step 1: TC 실패 목록 → 운영 정책 보완 항목 추출**

입력: TC 실행 결과 (실패 케이스 목록, 실패 패턴 메모)

AI 역할: 실패 케이스를 유형별로 분류하고, 각 유형이 정책서에서 다루는지 여부를 점검한다. 정책 규칙이 없는 실패 유형은 OPD 보완 후보로 분류한다.

```
아래 TC 실패 목록을 읽고 실패 원인 유형별로 분류해줘.
각 유형에 대해:
- 실패 유형명
- 해당 케이스 목록
- 현재 정책서에 관련 규칙이 있는가 (예/아니오)
- 없다면 OPD 보완이 필요한 이유

[TC 실패 케이스 목록]
(붙여넣기)

[정책서 규칙 요약]
(붙여넣기)
```

PM 검토 기준: 실패가 설계 문제인지 구현 문제인지 운영 정책 문제인지 먼저 판단한다. 운영 정책 문제만 OPD 보완 대상이다.

---

**Step 2: 운영 예외 패턴 → OPD 규칙 초안 생성**

입력: Step 1에서 확인된 OPD 보완 대상 예외 목록

AI 역할: 각 예외 유형에 대해 운영 정책서 항목 초안을 생성한다. 발생 조건, 처리 방식, 책임자, 기록 기준, 에스컬레이션 경로를 포함한다.

```
아래 운영 예외 유형을 기반으로 운영 정책서(OPD) 항목 초안을 만들어줘.
각 항목은:
- 예외명
- 발생 조건
- 1차 처리 방식 (자동 / 수동 / 에스컬레이션)
- 처리 책임자
- 기록 방식 (로그 / 이슈 등록 / 알림 등)
- 에스컬레이션 조건 (있으면)

기존 정책서와 중복되는 부분은 "(기존 규칙 참조: [항목명])"으로 표시해줘.

[운영 예외 목록]
(붙여넣기)

[기존 정책서 항목 요약]
(붙여넣기)
```

PM 검토 기준: 운영팀과 합의가 필요한 항목인지 확인한다. 특히 처리 책임자와 에스컬레이션 경로는 실제 조직 구조와 맞아야 한다.

---

**Step 3: 기존 정책서 + OPD 갱신안 → diff 점검**

입력: 기존 정책서 + OPD 갱신 초안 (Step 2 결과)

AI 역할: 갱신 항목과 기존 정책서 규칙을 비교해서 변경·충돌·중복 항목을 표로 출력한다.

```
아래 기존 정책서와 OPD 갱신 초안을 비교해줘.
다음 항목을 분류해서 표로 정리해:
- 신규 추가 항목 (기존에 없던 규칙)
- 기존 규칙 수정 항목 (범위나 조건이 바뀐 것)
- 충돌 항목 (기존 규칙과 갱신안이 서로 다른 처리를 지시하는 것)
- 중복 항목 (이미 있는 규칙을 다시 쓴 것)

[기존 정책서]
(붙여넣기)

[OPD 갱신 초안]
(붙여넣기)
```

PM 검토 기준: 충돌 항목은 이해관계자 확인 후 해소한다. 갱신이 완료되면 정책서 버전을 올리고 변경 이력(Changelog)에 기록한다.

---

### 12-3. AI가 할 수 없는 것

- **설계 문제 vs 운영 문제 판단**: TC 실패가 어느 층의 문제인지는 PM이 직접 판단해야 한다.
- **운영팀 합의**: 처리 책임자와 에스컬레이션 경로는 실제 조직 협의가 필요하다.
- **정책 우선순위 결정**: 여러 예외 규칙이 충돌할 때 어느 것이 우선인지는 비즈니스 판단이다.
- **변경 승인**: OPD 갱신은 정책서 변경이므로 관련 이해관계자 승인이 필요하다.

---

### 12-4. OPD 체크리스트

- [ ] TC 실패 케이스가 유형별로 분류됐는가
- [ ] 정책서에 없는 예외 유형이 OPD 항목으로 올라왔는가
- [ ] 각 OPD 항목에 발생 조건 / 처리 방식 / 책임자 / 기록 기준이 있는가
- [ ] 기존 정책서와 충돌하는 항목이 해소됐는가
- [ ] 갱신 이후 정책서 버전과 Changelog가 업데이트됐는가
- [ ] 운영팀과 합의가 필요한 항목이 협의됐는가

---

## 13. 정리

테스트케이스는 마지막 문서가 아니다. 이 문서가 강해야 설계가 검증 가능해지고, 구현이 흔들리지 않으며, 운영 이슈도 다시 설계로 환류된다.

좋은 테스트케이스는 아래를 만족해야 한다.

- DMN 규칙 행과 1:1로 대응한다
- BPMN 흐름의 정상·예외·운영 경로를 모두 포함한다
- 사양서의 상태값·메시지·API 응답이 기대 결과로 명시된다
- 프로토타입 경험과 실제 구현 결과를 연결한다
- RTM, DoD, 운영 정책서로 이어진다

그리고 TC 실행 이후 발견된 예외 패턴은 OPD로 환류된다. 이 흐름까지 완성되어야 산출물 체인이 운영 단계까지 닫힌다.

## 다음 장으로의 연결

테스트케이스가 완성되면 "이것을 언제, 누가, 어떤 환경에서, 어떤 기준으로 실행하고 완료를 판단할 것인가"를 정의할 차례다. TC 목록이 테스트 계획서의 범위와 일정의 기반이 된다.

### 테스트케이스에서 테스트 계획서로 이동하는 항목

| 테스트케이스 항목 | 이동 형태 | 테스트 계획서에서의 역할 |
|---|---|---|
| TC 목록 전체 | TC ID, 유형, 우선순위 목록 | TP 테스트 범위 및 커버리지 계획 |
| TC 유형 분포 | 기능/비기능/통합/UI 비율 | TP 테스트 유형별 일정 계획 |
| TC 선행 조건 | 환경·데이터 조건 목록 | TP Entry Criteria 항목 |
| TC 완료 판단 기준 | Pass/Fail 기준 | TP Exit Criteria (DoD) |
| 회귀 TC 목록 | 리그레션 대상 TC ID | TP 회귀 테스트 계획 |
| 우선순위 상 TC | Critical·High TC 목록 | TP 차단 이슈(Blocker) 판단 기준 |

### 체인 전환 기준 (테스트케이스 → 테스트 계획서)

| 확인 항목 | 기준 |
|---|---|
| TC 커버리지 완성 | DMN 규칙 행과 BPMN 흐름 경로가 TC에 매핑되어 있는가 |
| 유형 분류 완료 | 기능/비기능/통합/UI 유형이 TC에 태그되어 있는가 |
| 우선순위 분류 완료 | Critical·High·Medium·Low가 TC에 표시되어 있는가 |
| TC 완료 기준 명시 | 각 TC에 Pass 판단 조건이 있는가 |

- **이 장이 결정한 것**: 테스트 시나리오, 기대 결과, 검증 항목 목록
- **다음 장이 시작하는 것**: TC 목록을 기반으로 테스트 일정·환경·완료 기준·책임을 정의하는 테스트 계획서

---

## 제2절. 테스트 계획서

> **전제 지식**: 24장 제1절(테스트케이스), 21장(사양서) — 테스트 계획서는 테스트케이스를 어떤 체계로 운영할지를 설계한다. TC가 무엇인지 먼저 알아야 계획서에서 진입·완료 기준을 정하는 의미를 이해할 수 있다.

## 이 장의 목적

이 장은 테스트케이스를 만들기 전에 **무엇을, 어떤 방식으로, 언제, 누가 검증할 것인지**를 정의하는 **테스트 계획서(Test Plan)**를 다룬다.

테스트케이스가 검증 항목의 목록이라면, 테스트 계획서는 테스트 전체를 어떤 체계로 운영할 것인가의 설계도다. 테스트 계획서 없이 테스트케이스만 작성하면, 어떤 테스트가 얼마나 완료됐는지, 어떤 기준에서 출시를 결정할 수 있는지를 아무도 판단하지 못한다.

이 장을 읽고 나면 독자는 다음을 할 수 있어야 한다.

- 테스트 계획서가 테스트케이스와 어떻게 다른지, 왜 둘 다 필요한지 설명할 수 있다.
- 테스트 계획서의 주요 구성 항목을 이해하고 작성할 수 있다.
- 진입 기준(Entry Criteria)과 완료 기준(Exit Criteria)을 정의할 수 있다.
- AI와 함께 테스트 계획서 초안을 만들고 PM이 무엇을 확정해야 하는지 판단할 수 있다.

---

## 1. 이 산출물이 왜 존재하는가

많은 팀이 테스트케이스는 만들지만 테스트 계획서는 생략한다. 그 결과는 예측 가능하다.

- 기능 테스트만 하고 성능 테스트는 "나중에"로 미뤄지다가 출시 직전에 처음 실행된다.
- QA 리소스 계획 없이 테스트가 시작되어 일정이 늘어난다.
- "테스트 완료"의 기준이 없어서 버그가 남아 있어도 완료로 선언된다.
- 릴리즈 결정을 누가 내려야 하는지 모호하다.
- AI가 생성한 테스트케이스를 어디까지 커버리지로 인정할지 기준이 없다.

테스트 계획서는 이 문제를 해결한다. **테스트의 범위·방식·일정·진입·완료 기준을 미리 정의해서 QA를 예측 가능하게 운영**하는 문서다.

## 2. 무엇을 입력으로 받고 무엇을 넘기는가

**입력:**

| 입력 소스 | 내용 |
|----------|------|
| PRD | 테스트 대상 기능 범위, 성공 기준 |
| NFR | 성능·가용성 테스트 기준값 |
| 테스트케이스 | 기능별 검증 항목 목록 |
| 일정 계획 | 개발 완료 시점, 출시 목표일 |
| DoR / DoD | 테스트 시작 가능 조건과 완료 기준 |

**출력:**

| 다음 산출물 | 테스트 계획서에서 넘겨주는 것 |

| **운영 정책서(OPD)** | 출시 전 최종 체크리스트, 롤백 조건 |
| **RTM** | 요구사항 → TC → 결과 추적 기준 |

## 3. 테스트 계획서 구조

### 3-1. 테스트 범위

| 항목 | 내용 |
|------|------|
| 테스트 대상 | 이번 릴리즈에 포함된 기능 목록 |
| 테스트 제외 | Out of Scope로 정의된 기능 또는 영역 |
| 기존 기능 회귀 | 영향 받는 기존 기능의 회귀 테스트 범위 |

### 3-2. 테스트 유형

| 테스트 유형 | 목적 | 주체 |
|------------|------|------|
| 기능 테스트 | 요구사항·AC 기반 기능 검증 | QA |
| 회귀 테스트 | 기존 기능 영향 없음 확인 | QA |
| 성능 테스트 | NFR 기준 응답시간·처리량 검증 | QA + 인프라 |
| 보안 테스트 | 취약점·권한 우회·입력 검증 | 보안 담당 또는 QA |
| 통합 테스트 | 외부 시스템 연동 정상 동작 | QA + 개발 |
| UAT | 실제 운영 환경 기반 사용자 수용 검증 | PO + 운영 |
| 탐색적 테스트 | 시나리오 기반 비정형 검증 | QA |

### 3-3. 진입 기준 (Entry Criteria)

테스트를 시작하기 위한 조건. 이것이 충족되지 않으면 QA를 시작하지 않는다.

- [ ] 개발 환경 배포 완료
- [ ] 기능 단위 테스트 통과율 70% 이상
- [ ] 테스트 데이터 준비 완료
- [ ] 테스트케이스 작성 완료 및 리뷰 통과
- [ ] 외부 연동 시스템 테스트 환경 접속 가능

### 3-4. 완료 기준 (Exit Criteria / DoD)

테스트를 완료하고 출시를 결정할 수 있는 기준.

- [ ] 기능 테스트 커버리지 100% (전체 TC 실행)
- [ ] Critical / High 버그 0건
- [ ] Medium 버그 이슈 등록 완료, PO 수용 결정
- [ ] 성능 테스트 NFR 기준 통과
- [ ] 보안 취약점 Critical 0건
- [ ] 회귀 테스트 Pass
- [ ] PO 최종 승인

### 3-5. 테스트 환경

| 환경 | 목적 | 데이터 |
|------|------|--------|
| 개발 (DEV) | 개발 중 단위 테스트 | 더미 데이터 |
| 스테이징 (STG) | QA 기능·통합 테스트 | 익명화 실데이터 |
| 운영 유사 (PRE-PROD) | 성능·UAT | 운영 수준 데이터 볼륨 |

### 3-6. 버그 분류 기준

| 등급 | 기준 |
|------|------|
| Critical | 핵심 기능 동작 불가, 데이터 손실, 보안 취약점 |
| High | 주요 시나리오 실패, 심각한 성능 저하 |
| Medium | 일부 기능 오작동, 사용성 저하 |
| Low | UI 오류, 문구 오탈자, 경미한 레이아웃 |

## 4. AI와 함께 만드는 법

### 단계 1: 입력 준비

AI에게 다음을 제공한다.
- PRD의 기능 범위와 성공 기준
- NFR의 성능·보안 기준값
- 프로젝트 일정(개발 완료 예상일, 출시 목표일)

### 단계 2: AI 초안 생성

프롬프트 예시:
```
아래 기준으로 테스트 계획서 초안을 작성해줘.

테스트 대상: 회원가입, 로그인, 계정 잠금/해제, 휴면 전환 기능
NFR 기준:
- 응답시간 95%ile 3초 이내
- 동시접속 5,000명 처리
일정:
- 개발 완료 예정: 2026-04-15
- 출시 목표: 2026-04-30

테스트 유형별 일정, 진입/완료 기준, 버그 분류 기준을 포함해줘.
```

### 단계 3: PM 확정 기준

| 항목 | 이유 |
|------|------|
| 완료 기준 수용 수준 | "Medium 버그 수용" 여부는 PM이 결정 |
| 테스트 환경 승인 | 스테이징 데이터 정책 확인 필요 |
| 일정 현실성 | QA 리소스와 일정을 PM이 조율 |
| UAT 참여자 | PO·운영팀 UAT 일정 협의 필요 |

## 5. 품질 기준 / 체크리스트

| 항목 | 확인 기준 |
|------|-----------|
| 범위 정의 | 테스트 대상과 제외 범위가 명시됐는가 |
| 유형 완결성 | NFR 기반 성능·보안 테스트가 계획에 포함됐는가 |
| 진입 기준 | QA 시작 조건이 명확한가 |
| 완료 기준 | 출시 결정 기준이 정량적으로 정의됐는가 |
| 환경 정의 | 테스트 환경별 목적과 데이터 기준이 있는가 |
| 버그 분류 | 등급 기준이 명확하고 팀 내 합의됐는가 |
| 승인 상태 | Draft / Review / Approved 상태가 표시됐는가 |

## 6. 다음 단계 연결 규칙

- **QA 실행**: 테스트 계획서를 기준으로 테스트케이스 실행, 버그 등록, 결과 리포트 작성
- **운영 정책서(OPD)**: 출시 전 체크리스트와 롤백 조건이 OPD와 연동
- **RTM**: 테스트 실행 결과가 요구사항 추적 행렬에 반영됨
- **Decision Log**: 완료 기준 조정, 버그 수용 결정 등 변경 내용 기록

---

## 7. 이 장의 핵심 메시지

> 테스트케이스는 무엇을 확인하는지를 정의하고, 테스트 계획서는 어떻게, 언제, 어떤 기준으로 완료를 판단하는지를 정의한다.

테스트 계획서가 없으면 QA는 시작할 수는 있지만 언제 끝낼 수 있는지 알 수 없다. 출시 결정은 "테스트 다 됐어요?" 가 아니라 테스트 계획서에 정의된 완료 기준의 충족 여부로 내려져야 한다.

---

## 8. 다음 장으로의 연결

테스트 계획과 실행이 완료되면 "출시 이후 서비스를 어떻게 유지하고 대응할 것인가"를 정의할 차례다. TP의 Exit Criteria가 OPD의 출시 판단 기준이 되고, 테스트 환경 정의가 운영 환경 기준으로 이어진다.

### 테스트 계획서에서 OPD로 이동하는 항목

| 테스트 계획서 항목 | 이동 형태 | OPD에서의 역할 |
|---|---|---|
| Exit Criteria (DoD) | 출시 가능 완료 조건 목록 | OPD 출시 판단 기준 |
| 테스트 환경 정의 | 환경·인프라 구성 명세 | OPD 운영 환경 기준 |
| 버그 심각도 기준 | 1~4등급 정의 | OPD 장애 등급 분류 기준 |
| 성능 테스트 결과 | 응답시간·TPS 실측값 | OPD SLA 기준값 확정 근거 |
| Blocker 버그 처리 기준 | 차단 이슈 판단 기준 | OPD 출시 보류 기준 |
| 회귀 테스트 범위 | 배포 후 검증 TC 목록 | OPD 배포 후 스모크 테스트 기준 |

### 체인 전환 기준 (테스트 계획서 → OPD)

| 확인 항목 | 기준 |
|---|---|
| Exit Criteria 달성 | DoD 항목이 모두 Pass 상태인가 |
| Critical 버그 없음 | 미해결 Critical 버그가 없는가 |
| 성능 기준 충족 | NFR에서 정의한 응답시간·TPS 기준을 실측에서 달성했는가 |
| TP 상태 | `Approved` 또는 테스트 완료 서명 상태인가 |

- **이 장이 결정한 것**: 테스트 범위, 일정, 책임, Entry/Exit Criteria
- **다음 장이 시작하는 것**: 출시 이후 서비스 품질을 유지하는 운영 정책서(OPD)
