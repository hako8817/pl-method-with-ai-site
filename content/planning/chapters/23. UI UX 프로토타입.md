# 23장. UI/UX (User Interface / User Experience) 프로토타입


> **이 장에서 다루는 것**
> - 프로토타입의 역할과 설계 입력 산출물과의 관계
> - AI와 함께 화면 목록·분기를 구조화하는 3단계
> - 프로토타입 → 테스트케이스 전환 포인트

![](./figures/fig-20-prototype-chain.svg)

> 도식: 프로토타입 산출물 체인: 입력 문서(PRD·BPMN·정책서) → Lo/Mid/Hi-Fi 단계 → 사용자 검증·TC·개발 핸드오프


## 이 장의 목적

이 장은 PRD, 정책서, BPMN, DMN, 사양서로 정리된 내용을 실제 사용자가 경험할 수 있는 화면 구조와 상호작용으로 번역하는 방법을 다룬다. 여기서 말하는 프로토타입은 보기 좋은 시안이 아니다. 프로토타입은 정책과 흐름과 판단과 구현 조건이 사용자 경험으로 어떻게 드러나는지 검토 가능한 상태로 만든 산출물이다.

이 장의 목적은 두 층으로 나뉜다.

- 방법론 층: 프로토타입이 산출물 체인 안에서 어떤 역할을 맡는지 설명한다.
- 실용 층: 실제로 어떤 도구와 환경을 써서 프로토타입을 만들고 개발과 QA로 넘겨야 하는지 설명한다.

이 장을 읽고 나면 다음을 이해해야 한다.

- 디자인 시스템과 프로토타입의 관계
- 와이어프레임, UI 시안, 인터랙션 프로토타입의 역할 차이
- Figma에서 실제로 어떤 환경을 구성해야 하는지
- VS Code와 연결해 어떻게 개발 친화적 산출물로 넘겨야 하는지
- AI를 활용할 때 무엇을 자동화하고 무엇을 사람이 직접 확정해야 하는지

## 1. 이 산출물이 왜 존재하는가

앞 장들에서 PRD, 정책서, BPMN, DMN, 사양서를 잘 써도 실제 팀 협업에서는 다음 질문에서 자주 막힌다.

- 사용자는 어느 화면에서 무엇을 먼저 보게 되는가
- 정책 규칙은 어떤 문구와 어떤 차단 경험으로 드러나는가
- 예외 상황은 어떤 흐름으로 복구되는가
- 개발자는 어느 프레임이 실제 구현 기준인지 어떻게 식별하는가
- QA는 어떤 상호작용을 어디까지 테스트해야 하는가

이 질문에 답하는 산출물이 바로 UI/UX 프로토타입이다. 따라서 프로토타입은 디자인 단계의 부록이 아니라, 기획 산출물이 실제 경험으로 번역되었는지 확인하는 핵심 검증 단계다.

---

## 2. 이 책에서 프로토타입이 맡는 역할

이 책에서 프로토타입은 아래 네 가지 역할을 동시에 가진다.

### 2-1. 경험 번역 산출물

문서에 적힌 규칙과 흐름을 사용자가 실제로 겪는 화면, 상태, 문구, 클릭 흐름으로 바꾼다.

### 2-2. 검토 산출물

PO, 디자이너, 개발자, QA, 운영이 같은 화면을 보며 “정책이 이렇게 드러나도 되는가”를 확인하게 만든다.

### 2-3. 핸드오프 산출물

개발자가 구현할 프레임, 상태, 컴포넌트, 문구, 자산을 명확히 식별할 수 있게 한다.

### 2-4. 테스트 입력 산출물

QA가 정상 흐름, 예외 흐름, 차단 메시지, 상태 전이를 테스트케이스로 바꿀 수 있게 한다.

즉 프로토타입은 미적 시안이 아니라 `정책 + 흐름 + 판단 + 구현 조건`이 사용자 경험으로 드러난 결과물이다.

## 3. 디자인 시스템과 프로토타입의 경계

실무에서는 디자인 시스템과 프로토타입을 자주 섞어 말하지만 역할은 다르다.

### 3-1. 디자인 시스템이 답하는 질문

- 어떤 토큰을 쓸 것인가
- 어떤 컴포넌트가 표준인가
- 버튼, 입력창, 모달, 배지, 토스트의 상태는 무엇인가
- 문구 톤과 UI 규칙은 무엇인가
- 접근성, 일관성, 반응형 원칙은 무엇인가

### 3-2. 프로토타입이 답하는 질문

- 사용자는 어떤 순서로 화면을 지나가는가
- 어떤 상태에서 어떤 문구와 액션이 노출되는가
- 어떤 예외가 어떤 인터랙션으로 드러나는가
- 어떤 정책 차단이 사용자에게 어떤 경험으로 느껴지는가

### 3-3. 관계 정리

- 디자인 시스템은 재사용 가능한 부품과 규칙의 기준이다.
- 프로토타입은 그 부품과 규칙이 특정 시나리오 안에서 어떻게 작동하는지 보여주는 산출물이다.
- 디자인 시스템이 없으면 프로토타입은 화면마다 흔들린다.
- 프로토타입이 없으면 디자인 시스템은 실제 업무 흐름 검증과 연결되지 않는다.

## 4. 무엇을 입력으로 받고 무엇을 넘기는가

### 4-1. 입력

| 입력 소스 | 내용 |
|----------|------|
| PRD | 범위, 사용자 목표, 핵심 시나리오, 성공 기준 |
| 정책서 | 허용/차단/예외 규칙 |
| BPMN | 흐름과 역할, 예외 분기 |
| DMN | 판단 규칙 조합 |
| 사양서 초안 | 상태, 메시지, 인터페이스 조건 |

### 4-2. 출력

| 출력 산출물 | 받는 대상 | 역할 |
|-----------|---------|------|
| Lo-Fi 와이어프레임 | PO·기획자 | 화면 구조·시나리오 검토 |
| Mid-Fi UI 시안 | 개발·QA | 컴포넌트·상태·문구 기준 |
| Hi-Fi 인터랙션 프로토타입 | 사용자 검증·QA | 실제 흐름 검증, TC 입력 |
| 개발 핸드오프 파일(Figma Dev Mode) | 개발팀 | 프레임·토큰·에셋 기준 |
| 테스트케이스(TC) 입력 자료 | QA | 정상/예외 흐름 기준 |

**핵심 원칙:** 프로토타입은 디자인의 시작점이 아니라, 입력 산출물이 실제 경험으로 번역될 준비가 됐는지 확인하는 단계다.

프로토타입은 입력 산출물이 비어 있으면 가장 빨리 흔들린다. 위 입력 중 하나라도 누락된 상태에서 시작하면 다음 문제가 생긴다.

- 화면은 그럴듯한데 정책이 빠진다.
- 예외 흐름이 누락된다.
- 운영 개입이 화면에서 보이지 않는다.
- 개발과 QA가 같은 화면을 다르게 해석한다.

## 5. 실제 상세 업무 프로세스

실무에서는 아래 순서로 가는 것이 가장 안정적이다.

### 5-1. Step 0. 입력 산출물 정리

- PRD에서 범위와 핵심 시나리오 확인
- 정책서에서 규칙과 예외 확인
- BPMN에서 흐름과 운영 개입 지점 확인
- DMN에서 판단 결과와 예외 조건 확인
- 사양서에서 상태/메시지/인터페이스 조건 확인

### 5-2. Step 1. 화면 목록과 시나리오 목록 확정

먼저 예쁜 화면을 그리지 말고 화면 목록과 시나리오 목록을 확정한다.

예:

- 화면 목록: 로그인, 본인확인, 비밀번호 재설정, 잠금 안내, 고객센터 연결
- 시나리오 목록: 정상 로그인, 반복 실패, 잠금, 휴면 해제, 비밀번호 만료
- 상태 목록: 활성, 잠금, 휴면, 인증 대기, 실패

이 단계의 결과는 Screen Inventory와 Scenario Map이다.

### 5-3. Step 2. 와이어프레임 작성

와이어프레임은 정보 구조와 인터랙션 구조를 검토하는 단계다. 이 단계에서는 시각 polish보다 아래를 먼저 맞춘다.

- 어떤 정보가 먼저 보여야 하는가
- 주요 액션은 무엇인가
- 차단 메시지는 어디에 노출되는가
- 성공/실패 후 다음 화면은 무엇인가
- 모바일과 데스크톱에서 정보 우선순위가 달라지는가

정상 흐름만 그리지 말고 예외 흐름까지 반드시 포함해야 한다.

### 5-4. Step 3. 디자인 시스템 매핑

와이어프레임 구조가 잡히면 화면을 디자인 시스템에 연결한다.

- 어떤 컴포넌트를 재사용할지 결정
- 필요한 상태 variant 정의
- 색상, 타이포, 스페이싱, radius, icon 사용 기준 적용
- 문구 컴포넌트와 에러 패턴 통일
- 반응형 레이아웃 규칙 확정

새 컴포넌트가 필요하면 프로젝트 전용 예외로 만들지 말고 시스템 편입 대상인지 먼저 판단해야 한다.

### 5-5. Step 4. 인터랙션 프로토타입 작성

이제 클릭 가능한 흐름으로 연결한다.

- 클릭 시 이동 프레임 연결
- 정상/실패/차단/만료 분기 연결
- 오버레이, 모달, 드로어, 토스트 상태 연결
- 입력 오류, 로딩, 비활성 상태 표현
- 복구 경로와 재시도 경로 표현

이 단계에서 BPMN과 DMN이 큰 역할을 한다.

- BPMN은 어떤 순서와 분기가 프로토타입에 있어야 하는지 알려준다.
- DMN은 어떤 조건 조합에서 어떤 문구와 결과가 나와야 하는지 알려준다.

### 5-6. Step 5. 리뷰와 수정

프로토타입 리뷰는 디자인 피드백 회의가 아니라 산출물 정합성 검토 회의여야 한다.

확인 질문:

- PRD의 핵심 시나리오가 모두 반영됐는가
- 정책서의 차단 규칙과 예외가 화면 경험으로 드러나는가
- BPMN의 예외 경로가 빠지지 않았는가
- DMN의 판단 결과가 문구와 액션으로 연결되는가
- QA가 테스트케이스로 바꾸기 충분한가

### 5-7. Step 6. 핸드오프와 개발 연결

프로토타입은 개발 전달 시점에 아래 상태를 만족해야 한다.

- `ready for dev` 프레임 구분
- 컴포넌트, 변수, 스타일 정리
- 문구 최종안 또는 문구 소스 링크 연결
- API/사양/정책 링크 연결
- 구현 우선순위와 제외 범위 명시

### 5-8. Step 7. QA와 운영 반영

- QA는 프로토타입을 기준으로 테스트케이스를 세분화한다.
- 운영은 VOC, 실패 로그, 이탈 지점을 바탕으로 다음 개선 대상을 잡는다.
- 수정된 정책과 흐름은 다시 PRD, BPMN, DMN, 프로토타입으로 반영한다.

## 6. Figma에서 실제로 어떻게 작업해야 하는가

실무적으로는 “Figma를 쓴다” 수준이 아니라 `어떤 구조로 운영할 것인가`를 먼저 정해야 한다.

### 6-1. 추천 파일 구조

하나의 기능 단위 파일이라면 최소 아래 수준으로 나누는 것이 좋다.

- `00_Cover / Guide`: 파일 목적, 범위, 관련 링크
- `01_Flow`: 화면맵, 사용자 흐름도, BPMN 요약 링크
- `02_Wireframe`: 저충실도 구조 검토
- `03_Screens`: 실제 화면 시안
- `04_Prototype`: 인터랙션 검토용 연결 프레임
- `05_Ready for Dev`: 개발 전달 기준 프레임
- `06_Archive`: 폐기안, 비교안

디자인 시스템은 별도 라이브러리 파일로 분리하는 것이 좋다.

### 6-2. 섹션 운영 원칙

- 화면 페이지와 시스템 페이지를 섞지 않는다.
- 시나리오 단위로 섹션을 끊는다.
- 정상/예외/운영개입 흐름을 같은 화면 묶음에서 볼 수 있게 배치한다.
- `ready for dev` 대상 프레임은 별도로 표시한다.

### 6-3. 컴포넌트와 변수

Figma에서 디자인 시스템을 실제로 쓰려면 아래가 정리돼 있어야 한다.

- Variables: color, spacing, radius, typography, mode
- Components: input, button, modal, toast, tabs, status badge
- Variants: default, hover, disabled, error, success, loading
- Interactive components: 토글, 탭, 드롭다운, 아코디언

이렇게 해야 프로토타입이 단순 그림이 아니라 “실제 구현될 인터페이스의 상태 집합”으로 기능한다.

### 6-4. Auto Layout과 반응형

주요 프레임은 Auto Layout 기준으로 구성하는 편이 좋다.

- spacing과 padding은 변수 기준으로 통일
- 컴포넌트 resizing rule 명확화
- 모바일/데스크톱 breakpoint 대응 프레임 분리
- 리스트/폼/모달은 상태별로 재사용 가능하게 구성

### 6-5. 문구와 상태

프로토타입이 약해 보이는 가장 큰 이유 중 하나는 문구가 placeholder 상태로 남기 때문이다. 특히 아래는 실제 검토용 문구 수준으로 넣어야 한다.

- 정상 문구
- 오류 문구
- 차단 문구
- 복구 안내 문구
- 고객센터/운영 연계 문구

이 다섯 개는 UI 시안이 아니라 정책의 표현이다.

## 7. Figma의 실제 기능을 어떻게 써야 하는가

현재 기준으로 Figma 쪽에서 실무적으로 중요한 기능은 아래다. 이 부분을 알아야 프로토타입이 “그림”이 아니라 “개발 연결 산출물”이 된다.

### 7-1. Dev Mode

Dev Mode는 디자이너용 편집 화면이 아니라 개발 전달과 검사에 맞춘 모드다. 개발자는 여기서 다음을 본다.

- `ready for dev` 상태의 프레임
- 코드 스니펫
- 컴포넌트 속성
- 변경 비교
- 개발 리소스 링크

따라서 프로토타입 산출물은 “보기 좋은 화면”보다 “개발자가 Dev Mode에서 읽을 수 있는 화면”이 되어야 한다.

### 7-2. Figma for VS Code

VS Code 안에서 디자인 파일을 탐색하고 검사할 수 있으면 개발자는 브라우저, 문서, 디자인, 코드를 덜 왕복한다.

실무 흐름 예시:

1. VS Code에서 Jira 티켓을 연다.
2. 티켓에서 PRD/정책서 링크를 확인한다.
3. 같은 작업 문맥에서 Figma 프레임을 확인한다.
4. 기존 코드 컴포넌트와 비교한다.
5. 구현 또는 AI 보조 생성을 진행한다.

### 7-3. Dev Resources

Figma 프레임에는 아래 링크가 같이 붙는 것이 좋다.

- Jira Story
- PRD 섹션
- 정책서 섹션
- API 문서
- Storybook
- Git 저장소 또는 구현 파일

이렇게 해야 프레임이 단순 시안이 아니라 개발 연결 노드가 된다.

### 7-4. Code Connect 관점

디자인 컴포넌트와 코드 컴포넌트가 연결되면 아래가 쉬워진다.

- 실제 구현 컴포넌트 확인
- 컴포넌트 재사용 여부 판단
- 디자인-코드 불일치 점검
- 리뷰 속도 향상

### 7-5. MCP 서버 관점

AI 에이전트가 Figma 컨텍스트를 읽을 수 있는 환경을 만들면 문서-디자인-코드 연결 품질이 좋아진다. 다만 이때도 기준 문서가 먼저 정리돼 있어야 한다.

AI에게 맡기기 좋은 일:

- 프레임 구조 요약
- 상태별 화면 차이 정리
- 누락된 예외 화면 질문 생성
- 디자인-문서 불일치 후보 탐지

---

## 8. VS Code에서 어떤 환경을 만들어야 하는가

프로토타입이 실제 개발 연결 산출물이 되려면 VS Code 쪽 작업 환경도 같이 설계해야 한다.

### 8-1. 최소 개발 문맥

개발자가 한 기능을 구현할 때 최소 아래가 같은 작업 문맥 안에 있어야 한다.

- Jira 이슈
- PRD 링크
- 정책서 링크
- Figma 프레임
- API/사양 링크
- 구현 코드

### 8-2. 권장 VS Code 셋업

- GitHub Copilot 또는 팀이 쓰는 AI 코딩 도구
- Figma for VS Code
- GitHub Pull Requests / GitLens 류 확장
- Markdown 링크 확인 또는 위키 탐색용 확장

핵심은 확장 수가 아니라, `문서-디자인-코드 컨텍스트가 같은 곳에서 보이게 하는 것`이다.

### 8-3. 브랜치/이슈 연결 규칙

실무적으로는 아래 정도가 있으면 충분하다.

- 브랜치명에 이슈 키 포함
- PR 설명에 PRD/정책서/Figma 링크 포함
- 구현 완료 시 테스트 포인트와 영향 문서 확인

## 9. 팀이 처음 시작할 때의 최소 셋업

실용서 성격으로 보면 이 장은 여기까지 가야 한다. 팀이 처음 시작할 때 최소 아래를 정하면 된다.

### 9-1. 문서 기준점

- PRD/정책서/사양서/추적 문서는 어디에 둘 것인가
- 승인된 공식 기준은 어디인가

### 9-2. 디자인 기준점

- 디자인 시스템 파일은 어디인가
- 프로젝트 프로토타입 파일 구조는 어떻게 할 것인가
- `ready for dev`는 어떤 상태를 의미하는가

### 9-3. 개발 기준점

- 어떤 이슈 체계로 작업할 것인가
- Figma 링크는 어디에 붙일 것인가
- PR 설명에는 어떤 문서 링크를 반드시 넣을 것인가

### 9-4. QA 기준점

- 프로토타입과 테스트케이스를 어떻게 연결할 것인가
- 예외 흐름 검증 기준은 어디에 둘 것인가

## 10. 로그인/회원가입 케이스에 적용하면

이 대표 케이스에서는 아래 식으로 연결하는 것이 좋다.

- PRD: 로그인/휴면/재가입 핵심 시나리오 정의
- 정책서: 잠금/휴면/재가입 규칙과 예외 정의
- BPMN: 사용자/시스템/운영 개입 흐름 정의
- DMN: 로그인 허용, 재가입 허용, 잠금 해제 판단 규칙 정의
- Figma: 정상/실패/차단/복구 화면 구조와 인터랙션 정의
- VS Code: 관련 이슈, 디자인 프레임, 구현 코드, 테스트를 같은 문맥에서 작업

예를 들어 휴면회원 로그인 시나리오라면:

- BPMN은 `로그인 시도 → 상태 조회 → 본인확인 → 활성 전환 → 로그인 성공` 흐름을 보여준다.
- DMN은 `계정 상태 + 본인확인 결과 + 예외 승인 여부` 조합으로 허용 여부를 판단한다.
- 프로토타입은 이 판단 결과가 어떤 화면과 문구로 보이는지 보여준다.
- 테스트케이스는 각 규칙 조합별 기대 결과를 검증한다.

## 11. AI와 함께 만드는 법

프로토타입 단계에서 AI 결과 품질을 좌우하는 것은 프롬프트 문장보다 **어떤 컨텍스트를 함께 주느냐**다. PRD·Feature·정책서·BPMN·DMN이 입력으로 있으면 AI는 화면 목록과 분기 구조를 빠르게 초안으로 만든다. 그 초안 위에 사람이 경험 우선순위와 문구를 확정하는 것이 이 단계의 구조다.

### 11-1. Step 1: Feature + Scenario → 화면 목록 초안 생성

**입력**

- Feature 목록 (14장 결과)
- Scenario 정상·예외 흐름
- BPMN 흐름 (분기 경로 중심)

**AI 역할**: 각 Scenario에 대응하는 화면 목록과 화면 간 이동 흐름을 텍스트로 초안 생성한다. 예외 경로가 빠지지 않도록 BPMN 분기를 함께 반영한다.

**프롬프트 예시**

```
아래 Feature·Scenario와 BPMN 흐름을 기반으로 프로토타입 화면 목록을 만들어줘.
각 화면은 아래 형식으로:
- 화면명
- 진입 조건 (어떤 상태에서 이 화면이 나타나는가)
- 주요 요소 (버튼, 메시지, 상태 표시)
- 이동 대상 화면 (버튼별)

정상 흐름뿐 아니라 예외·실패·차단 흐름도 빠짐없이 포함해줘.

[Feature/Scenario]
(붙여넣기)

[BPMN 분기 흐름]
(붙여넣기)
```

**PM 검토 기준**

- BPMN의 모든 분기 경로가 화면 목록에 반영됐는가
- 실패·차단·복구 화면이 빠지지 않았는가
- 진입 조건이 DMN 판단 결과와 연결되는가

---

### 11-2. Step 2: DMN 판단 결과 → 상태별 화면 분기 + 문구 초안

**입력**

- DMN 의사결정 테이블 (판단 조건별 결과값)
- 화면 목록 초안 (Step 1 결과)
- 정책서 메시지 규칙 (안내 문구 기준이 있으면)

**AI 역할**: DMN 판단 결과별로 어떤 화면 상태와 사용자 안내 문구가 나와야 하는지 초안을 생성한다.

**프롬프트 예시**

```
아래 DMN 판단 결과와 화면 목록을 연결해서 각 판단 결과별 화면 상태와 문구 초안을 만들어줘.
각 항목은 다음 형식으로:
- 판단 조건 (DMN 규칙 행)
- 연결 화면
- 화면 상태 (버튼 활성화/비활성화, 모달 노출 여부 등)
- 사용자 안내 문구 초안

문구는 정중하고 명확하게, 사용자가 다음에 할 행동을 안내하는 방향으로 써줘.

[DMN 의사결정 테이블]
(붙여넣기)

[화면 목록]
(붙여넣기)
```

**PM 검토 기준**

- DMN 규칙 행 하나가 화면 상태 하나에 대응하는가
- 사용자 안내 문구가 정책서 조건에서 벗어나지 않는가
- 차단·실패 경험이 복구 경로를 명시하고 있는가
- 기존 디자인 시스템의 문구 톤과 일관되는가

---

### 11-3. Step 3: 프로토타입 초안 → TC 전환 포인트 추출

**입력**

- 화면 목록 + 화면 분기 구조 (Step 1·2 결과)
- AC 목록 (14장 결과)

**AI 역할**: 화면과 분기 구조를 기반으로 TC로 전환 가능한 테스트 포인트 목록을 생성한다. AC와 매핑해서 커버되지 않는 화면 케이스도 함께 출력한다.

**프롬프트 예시**

```
아래 화면 목록과 AC를 비교해서 테스트 포인트 목록을 만들어줘.
각 항목은:
- 테스트 포인트명
- 검증 화면
- 검증 조건
- 연결 AC (있으면)
- AC가 없는 화면 케이스는 별도로 표시해줘

[화면 목록]
(붙여넣기)

[AC 목록]
(붙여넣기)
```

**PM 검토 기준**

- 모든 예외·차단 화면에 대응하는 TC 포인트가 있는가
- AC가 없는 화면이 있다면 AC 보완인지 TC 제외 결정인지 판단이 필요한가
- 이 목록이 24장(테스트케이스) 작성의 직접 입력으로 쓸 수 있는가

---

### 11-4. AI가 할 수 없는 것

- **사용자 경험 우선순위 판단**: 어떤 실패 경험을 더 친절하게 다루고 어떤 것을 간결하게 처리할지는 UX 판단이다.
- **정책 표현 문구 확정**: AI가 초안을 만들어도 법적·운영적 문구 기준은 사람이 승인해야 한다.
- **디자인 시스템 편입 결정**: 이 화면이 시스템 컴포넌트를 쓸지 예외 처리할지는 디자인 팀과의 협의가 필요하다.
- **`ready for dev` 승인**: 개발 전달 기준은 PM·디자이너·개발자가 함께 확인해야 한다.

---

### 11-5. 컨텍스트가 먼저다 — 실무 도구 연결 방식

AI를 `채팅창 하나`로 쓰는 것보다 도구 사이를 잇는 보조 레이어로 쓰는 편이 더 안정적이다.

- **Figma**: 화면 목록, 상태, 문구 초안을 만드는 실행 공간
- **VS Code**: 산출물 링크, 사양 메모, 구현 연결 포인트를 정리하는 기준 문서 공간
- **AI**: 승인된 문맥 위에서 초안과 검토 포인트를 빠르게 생성하는 보조 도구

프로토타입 단계의 AI 활용은 `대신 설계해 주는 도구`가 아니라, `승인된 컨텍스트 위에서 초안을 만드는 도구`로 이해해야 한다.

- [ ] DMN 판단 결과가 화면 상태와 문구로 이어지는가
- [ ] 정상 / 예외 / 차단 / 복구 화면이 모두 있는가

**연결 기준**
- [ ] AC 하나에 대응하는 화면 케이스가 있는가
- [ ] 화면 분기 조건이 DMN 규칙 행과 일치하는가
- [ ] TC 전환 포인트 목록이 21장 TC 작성의 입력으로 쓸 수 있는가

**품질 기준**
- [ ] 디자인 시스템과 일관되는가

- [ ] `ready for dev` 승인이 완료됐는가
- [ ] AI 초안 이후 PM·디자이너가 문구와 경험 우선순위를 직접 확정했는가

## 13. 다음 단계 연결 규칙

| 이 산출물 | 넘어가는 다음 산출물 | 연결 내용 |
|---|---|---|
| **화면 목록·흐름** | 테스트케이스(TC) | 화면 케이스별 → TC 입력 조건 |
| **문구·상태** | 사양서(Spec) | 메시지·화면 항목 ↔ Spec 메시지와 피드백 항목 상호 검증 |
| **예외 화면** | 운영 정책서(OPD) | 차단·복구 안내 화면 → OPD 운영 대응 규칙 |
| **Handoff 자료** | 개발 구현 | 화면 상태·컴포넌트·문구 → 개발 구현 기준 |

**연결 확인 규칙**

- 프로토타입에 "계정 잠금 안내" 화면이 있다면, Spec에 동일한 상태와 메시지 항목이 있어야 한다. 없으면 Spec과 프로토타입이 다른 기준을 가진 것이다.
- AC에 없는 화면 케이스가 발견됐다면, AC를 보완하거나 해당 케이스를 의도적으로 TC에서 제외하는 결정이 있어야 한다.
- 프로토타입의 예외 경로가 BPMN보다 더 많다면, BPMN이 미완성이거나 프로토타입이 범위를 초과한 것이다. 어느 쪽인지 먼저 확인한다.

## 이 장의 핵심 메시지

프로토타입은 의사결정의 도구다. "이렇게 만들면 어때?"라는 질문을 화면으로 보여주는 방식이다. 좋은 프로토타입은 기능을 화려하게 보여주는 것이 아니라, 핵심 흐름과 상태를 명확하게 보여주는 것이다.

AI 시대에 프로토타입의 역할이 바뀌고 있다. 이제 프로토타입은 사람만 보는 문서가 아니다. Figma 컴포넌트와 코드가 연결되고, AI가 디자인 토큰을 읽어 코드를 생성한다. 기획자가 프로토타입을 만들 때 이 연결을 염두에 두는 것이 실행 효율을 높인다.

## 정리

프로토타입은 예쁜 시안이 아니다. 좋은 프로토타입은 정책, 흐름, 판단, 구현 조건이 사용자 경험으로 어떻게 드러나는지 검토 가능하게 만들고, 그 결과를 개발과 QA가 바로 사용할 수 있게 한다. 그래서 23장은 디자인 부록이 아니라, 산출물 체인이 실제 경험으로 번역되는 핵심 연결 장이다.

이 장의 핵심은 분명하다.

- 프로토타입은 경험 번역 산출물이다.
- 디자인 시스템과 프로토타입은 역할이 다르다.
- Figma, Dev Mode, VS Code 연결까지 설계해야 실무형 산출물이 된다.
- 방법론 설명에서 끝나지 않고 실제 셋업과 핸드오프 기준까지 내려가야 한다.

AI를 실제로 어느 단계에서 어떻게 써야 하는지 더 실무적으로 보려면 [[ai-planning-practical-guide-v1]] 를 같이 보는 것이 좋다.

## 다음 장으로의 연결

프로토타입이 완성되면 설계가 실제로 검증 가능한 형태로 갖춰진 것이다. 이제 이 경험 흐름을 기반으로 "무엇을 어떻게 테스트할 것인가"를 정의할 차례다.

### 프로토타입에서 테스트케이스로 이동하는 항목

| 프로토타입 항목 | 이동 형태 | 테스트케이스에서의 역할 |
|---|---|---|
| 화면 흐름 (Happy Path) | Step 시퀀스 | 시나리오 TC의 기본 흐름 단계 |
| 화면 흐름 (예외·오류 경로) | 분기 경로 목록 | 예외 TC의 시나리오 |
| 인터랙션 정의 | 사용자 액션-반응 매핑 | UI TC의 액션-결과 검증 항목 |
| 검토 피드백으로 변경된 화면 | 변경 전·후 화면 목록 | 회귀 TC 대상 확인 |
| 에러 메시지 화면 | 에러 상황-메시지 매핑 | 에러 TC의 기대 결과 |
| 권한별 화면 차이 | 역할-표시 항목 매핑 | 권한 TC의 시나리오 |

### 체인 전환 기준 (프로토타입 → 테스트케이스)

| 확인 항목 | 기준 |
|---|---|
| 핵심 흐름 검토 완료 | 이해관계자 검토 피드백이 반영되었는가 |
| 예외 경로 식별 | Happy Path 외 주요 예외 경로가 프로토타입에 포함되어 있는가 |
| 화면-API 연결 확인 | 각 화면이 어떤 API를 호출하는지 명시되어 있는가 |
| 프로토타입 버전 확정 | 리뷰 통과 버전이 명확히 구분되어 있는가 |

- **이 장이 결정한 것**: 화면 흐름, 인터랙션 기준, 핸드오프 스펙
- **다음 장이 시작하는 것**: 프로토타입 흐름을 기반으로 테스트 시나리오와 AC 검증 기준을 테스트케이스로 문서화







## 가상 적용 예시 4. 주문 취소/환불 기능을 정책·흐름·데이터로 확장하기

쇼핑몰 플랫폼에서 주문 취소나 환불 기능을 만든다고 하면, 처음에는 단순한 버튼 기능처럼 보일 수 있다.
주문 상세 화면에 “취소 요청” 버튼을 넣고, 환불 처리를 하면 끝나는 일처럼 느껴진다.
하지만 실제로는 이 기능이야말로 왜 정책과 흐름과 데이터 구조가 따로 필요해지는지를 가장 잘 보여준다.

예를 들어 사용자가 주문을 취소하고 싶다고 해보자.
겉으로는 하나의 행동이지만, 시스템 안에서는 여러 질문이 한꺼번에 열린다.
언제까지 취소가 가능한가, 결제 완료 전과 후의 처리는 같은가, 배송 준비가 시작된 뒤에는 어떻게 달라지는가, 부분 취소는 가능한가, 쿠폰을 사용한 주문은 환불 금액을 어떻게 계산하는가, 운영자 승인 없이 자동 환불이 가능한 조건은 무엇인가.
이 질문들에 답하지 않은 채 화면만 만들면, 기능은 생기지만 기준은 없다.

그래서 이 기능은 먼저 정책서가 필요하다.
정책서는 취소와 환불의 원칙을 정하는 문서다.
예를 들어 결제 완료 후 배송 준비 전까지는 사용자가 직접 취소할 수 있고, 배송 시작 이후에는 운영자 검토를 거쳐 환불 요청으로 전환된다고 정할 수 있다. 쿠폰이 적용된 주문은 실제 결제 금액 기준으로 환불하며, 부분 취소 시에는 사용 쿠폰을 다시 계산한다는 원칙도 여기서 정한다.
이 규칙이 문장 속에 숨어 있으면 구현 단계에서 빠지기 쉽고, AI는 일반적인 쇼핑몰 상식을 끌어와 임의의 방식으로 채울 가능성이 높다. 그래서 정책은 앞단에서 먼저 분리되어야 한다.

권한 정책 매트릭스는 그다음으로 필요하다.
취소와 환불은 사용자만의 기능이 아니기 때문이다.
고객은 취소 요청만 할 수 있고, 운영자는 취소 승인이나 환불 보류를 결정할 수 있으며, 정산 담당자는 환불 완료를 확인할 수 있다면, 이 차이는 권한 규칙으로 분명히 나뉘어야 한다.
취소 버튼이 보이는지, 승인 화면에 접근할 수 있는지, 환불 금액을 수정할 수 있는지는 모두 권한 문제다. 이 구분 없이 API나 화면을 만들면, 나중에 가장 뒤에서 권한 예외가 터진다.

흐름은 BPMN으로 드러난다.
취소/환불 기능은 정책만 있다고 구현되지 않는다.
사용자가 취소를 요청하고, 시스템이 주문 상태를 확인하고, 조건에 따라 자동 취소 또는 운영자 검토로 분기하고, PG 환불 API를 호출하고, 결과에 따라 상태를 바꾸고, 고객에게 알림을 보내는 전체 흐름이 있어야 한다.
이 흐름은 단순 문장으로 설명하면 길어지고, 예외 분기에서 금방 흐려진다. 반대로 BPMN으로 정리하면 사용자, 시스템, 운영자, 외부 결제 시스템이 어디서 개입하는지가 한눈에 보인다.

DMN은 판단을 문서화하는 데 필요하다.
예를 들어 이런 조건을 생각해볼 수 있다.
주문 상태가 “결제 완료”이고 배송 상태가 “배송 준비 전”이면 자동 취소 가능, 배송 상태가 “출고 완료”이면 자동 취소 불가, 결제 수단이 특정 PG이고 부분 환불 금액이 기준 이하이면 자동 환불 가능, 그 외에는 운영자 검토 필요.
이런 규칙은 문장으로 적으면 금방 중복되고, 개발자는 if 문으로 흩어 넣기 쉽다. DMN으로 정리하면 어떤 입력 조건에서 어떤 결과가 나와야 하는지 판단표로 고정할 수 있다. 취소/환불처럼 규칙이 복잡한 기능일수록 이 문서의 가치가 커진다.

정보 구조와 데이터 모델도 뒤따라야 한다.
취소/환불은 단순 상태 변경이 아니라 데이터 구조를 바꾸는 기능이기 때문이다.
주문 상태와 환불 상태를 같은 값으로 둘 것인지, 따로 분리할 것인지가 먼저 정해져야 한다. 환불 사유 코드는 별도 코드 테이블로 둘지, 쿠폰 재계산 이력은 어디에 남길지, 환불 요청 시점과 승인 시점, 완료 시점을 모두 저장할지 같은 질문도 있다.
이런 기준은 ERD와 데이터 사전이 있어야 명확해진다. 화면에서는 “환불 완료” 한 줄로 보일 수 있지만, 데이터 관점에서는 요청, 검토, 승인, PG 처리, 정산 반영이 다른 의미를 가진다. 그래서 상태의 이름과 데이터의 의미를 함께 고정해야 한다.

시스템 아키텍처 문서와 인터페이스 정의도 필요해진다.
취소/환불 기능은 대개 쇼핑몰 플랫폼 내부 기능으로 끝나지 않는다.
주문 서비스, 결제 서비스, 알림 서비스, 관리자 백오피스, 외부 PG API가 연결된다. 어떤 서비스가 취소 가능 여부를 판단하고, 어떤 서비스가 환불 호출을 담당하며, 실패 시 재시도 책임은 어디에 있는지가 시스템 구조 안에서 정리되어야 한다.
이 문서가 없으면 기능은 구현돼도 책임 경계가 흐려진다.

그다음이 사양서와 API 명세서다.
정책과 흐름과 데이터가 정리된 뒤에야, 구현 기준을 더 구체적으로 적을 수 있다.
사양서에는 취소 요청 시 입력값, 상태 전이 규칙, 예외 처리, 로그 기록, 이벤트 태깅, 오류 메시지 기준이 들어간다. API 명세서에는 취소 요청 API, 환불 상태 조회 API, 관리자 승인 API의 요청과 응답 구조, 에러 코드, 인증 조건이 들어간다.
즉, 이 문서들은 앞에서 정한 정책과 구조를 실제 구현 가능한 언어로 번역하는 역할을 한다.

UI/UX 프로토타입은 마지막에 사용자의 관점에서 이 구조를 확인하게 해준다.
사용자에게는 언제 취소 버튼이 보이는지, 취소 불가일 때는 어떤 안내를 주는지, 환불 진행 상태는 어디서 확인하는지, 부분 환불일 경우 금액 계산은 어떻게 표시되는지가 중요하다.
운영자에게는 승인 대기 목록, 환불 사유, 상태 변경 이력, 예외 케이스 확인 화면이 중요해진다.
즉, 앞에서 만든 정책과 흐름과 데이터 구조가 화면 위에서 어떻게 나타나는지를 프로토타입으로 검토해야 한다.

이렇게 보면 주문 취소/환불 기능은 버튼 하나로 시작하지만, 실제로는 정책, 권한, 흐름, 판단, 데이터, 구조, 구현 기준이 모두 함께 움직이는 기능이다.
AI에게 “주문 취소 기능 만들어줘”라고 말하면 화면과 API의 뼈대는 만들어질 수 있다.
하지만 취소 가능 조건, 환불 계산 방식, 관리자 승인 기준, 상태값 정의가 문서로 구조화되어 있지 않으면, 기능은 동작해도 시스템으로는 불안정하다.

바이브 코딩은 속도를 높여주지만, 구조를 대신 설계해주지는 않는다.
그래서 정책은 정책서에, 흐름은 BPMN에, 판단은 DMN에, 데이터 의미는 ERD와 데이터 사전에, 구현 기준은 사양서와 API 명세서에 남겨야 한다.
취소/환불 같은 기능일수록 그 차이가 선명하게 드러난다.

결국 좋은 산출물은 설명을 늘리는 문서가 아니다.
복잡한 기능 안에 섞여 있는 서로 다른 종류의 판단을 분리하고, 그 분리가 다시 구현 가능한 기준으로 이어지게 만드는 문서다.
주문 취소/환불 기능은 그 사실을 가장 잘 보여주는 대표적인 예다.

**소결**
정책과 상태가 복잡한 기능일수록, 바이브 코딩의 성패는 코드 생성 능력보다 구조화된 명세의 유무에서 갈린다.
