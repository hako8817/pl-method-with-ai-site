# 31장. Skill, Harness Engineering, Harness Operating Environment

> 이 장은 AI 작업 체계의 엔지니어링 계층 세 가지를 다룬다. Skill은 반복 가능한 행동 매뉴얼로, AI의 반복 작업 패턴을 사전에 정의한다. Harness Engineering은 출시 이후 운영까지 이어지는 지속 가능한 실행 체계를 구성한다. Harness Operating Environment는 하네스가 안전하고 반복 가능하게 작동하는 환경을 정의한다.

> **전제 지식**: 30장(Context Engineering과 Prompt Engineering) — 스킬은 컨텍스트와 프롬프트를 반복 가능한 패턴으로 정의한 것이다. 컨텍스트 엔지니어링과 프롬프트 엔지니어링을 먼저 이해해야 스킬·하네스·운영 환경이 왜 그 위에 필요한지 알 수 있다.

---

## 제1절. Skill: 스킬

## 이 장의 목적

이 장은 AI의 반복 행동 패턴을 미리 정의해두는 개념인 스킬을 다룬다. 스킬은 AI에게 "어떤 상황에서 어떻게 행동하라"는 규칙 묶음을 사전에 만들어두고 재사용하는 방식이다. 프롬프트 엔지니어링이 단발 지시라면, 스킬은 반복 가능한 행동 매뉴얼이다.

이 장을 읽고 나면 다음을 이해해야 한다.

- 스킬이 프롬프트·컨텍스트 엔지니어링과 어떻게 다른지
- 스킬의 구조와 설계 기준
- 기획 실무에서 어떤 행동 패턴을 스킬로 만들면 효과적인지
- 스킬이 하네스 엔지니어링과 어떻게 연결되는지

---

## 1. 스킬이란 무엇인가

스킬은 미리 만들어둔 행동 매뉴얼이다. AI에게 "이런 상황이 오면 이 순서로 이렇게 행동해라"라는 규칙을 사전에 정의해두고, 필요할 때 불러서 쓰는 방식이다.

비유를 들면 이렇다. 신호등 없는 교차로를 지나는 운전 매뉴얼이 있다.

직진 시도 → 막히면 우회 → 우회도 막히면 대기

상황에 따른 행동 규칙을 미리 정해둔다. 매번 운전할 때마다 생각할 필요 없이, 상황이 오면 자동으로 적용한다.

AI에게 스킬을 주는 것도 같다. "PRD 초안 검토 시에는 이 순서로 이 기준을 적용해라", "회의록을 정리할 때는 이 구조로 이 항목을 채워라"처럼 반복되는 작업 패턴을 미리 규칙화해두는 것이다.

---

## 2. 스킬의 구조

스킬은 일반적으로 다음 요소로 구성된다.

**트리거 조건**: 이 스킬이 언제 발동되는지 정의한다. "PRD 초안이 들어왔을 때", "요구사항 목록이 주어졌을 때"처럼 발동 조건을 명확히 한다.

**행동 규칙**: 어떤 순서로 무엇을 하는지 정의한다. 단계별 행동, 판단 기준, 분기 조건이 포함된다.

**출력 형식**: 결과를 어떤 형태로 내놓는지 정의한다. 표, 목록, 서술, 특정 템플릿 등.

**경계 조건**: 이 스킬이 적용되지 않아야 하는 경우, 스킬의 범위를 벗어난 상황을 정의한다.

이 네 가지가 갖춰진 스킬은 반복 가능하다. 누가 사용해도 같은 작업 방식이 재현된다.

### 스킬 작성 예시: 회의록 정리 스킬

위 네 가지 요소를 실제로 작성하면 다음과 같다.

---

**스킬명**: 회의록 정리 스킬 (meeting-summary-v1)

**트리거 조건**
- 회의 전사 텍스트 또는 메모가 입력될 때
- "회의록 정리해줘", "회의 내용 구조화해줘" 요청이 들어왔을 때

**행동 규칙**
1. 전체 텍스트를 읽고 회의 목적을 1~2줄로 파악한다
2. 다음 네 가지 항목으로 분류한다:
   - **결정 사항**: "~하기로 했다", "~로 확정됐다" 패턴
   - **액션 아이템**: "~해야 한다", "담당자가 ~를 처리한다" 패턴. 담당자·기한 포함
   - **미결 사항**: "확인 필요", "나중에 논의", "검토 예정" 패턴
   - **다음 회의 안건**: "다음에 논의", "다음 주에 확인" 패턴
3. 각 항목을 날짜·담당자·우선순위와 함께 정리한다
4. 전체 내용을 마크다운 형식으로 출력한다

**출력 형식**
```
## 회의록 요약
날짜: [날짜]  참석자: [참석자 목록]  회의 목적: [1~2줄]

## 결정 사항
- [결정 내용] — 근거: [간략한 이유]

## 액션 아이템
| 내용 | 담당자 | 기한 |
|---|---|---|
| ... | ... | ... |

## 미결 사항
- [내용] — 결정 필요 이유: [간략 설명]

## 다음 회의 안건
- [안건 내용]
```

**경계 조건 (이 스킬이 적용되지 않는 경우)**
- 전사 텍스트가 불완전하거나 50% 이상 누락된 경우: 정리 시도하지 않고 "전사 내용 불완전" 알림
- 기술적 의사결정이 포함된 경우: 기획·개발 결정을 구분하지 않고 그대로 결정 사항에 포함 (분류 판단은 사람이 함)
- 참석자가 5명 이상이고 논의가 2개 이상 주제: 주제별로 섹션 분리해서 출력

---

이 스킬 정의가 있으면 회의마다 새로 설명할 필요 없이 "meeting-summary-v1 적용해줘"라고 말하면 항상 같은 방식으로 결과가 나온다.

---

## 3. 기획 실무에서 스킬로 만들면 효과적인 것

모든 작업을 스킬로 만들 필요는 없다. 다음 조건에 맞는 작업이 스킬로 만들기 적합하다.

**반복 빈도가 높다**: 프로젝트마다, 스프린트마다, 또는 정기적으로 반복되는 작업이다.

**판단 기준이 일정하다**: 매번 다른 기준으로 하지 않고, 팀 안에서 합의된 방식이 있다.

**출력 형식이 일정하다**: 결과물의 구조가 항상 비슷하다.

기획 실무에서 대표적인 스킬 후보는 다음과 같다.

- **PRD 구조 검토 스킬**: PRD가 들어오면 범위·우선순위·성공 기준·Out of Scope가 갖춰져 있는지 체크한다.
- **요구사항 분류 스킬**: 요구사항 목록이 들어오면 기능/비기능/정책/제약으로 분류하고 누락 여부를 확인한다.
- **회의록 정리 스킬**: 회의 전사 텍스트가 들어오면 결정 사항, 액션 아이템, 미결 질문, 다음 회의 안건으로 구조화한다.
- **정책 충돌 탐지 스킬**: 정책서 초안이 들어오면 기존 정책과 충돌 가능한 항목을 탐지하고 목록으로 낸다.
- **Acceptance Criteria 검증 스킬**: AC 목록이 들어오면 테스트 가능한 형식인지, Given/When/Then 구조가 맞는지 점검한다.

---

## 4. 스킬 설계 시 주의점

스킬은 강력하지만, 잘못 설계하면 오히려 방해가 된다.

**너무 광범위하면 안 된다**: "모든 기획 문서를 검토해라"처럼 범위가 불분명하면 AI가 무엇을 어떻게 해야 하는지 모른다. 스킬은 작고 명확해야 한다.

**예외를 무시하면 안 된다**: "항상 이 구조로 출력해라"처럼 경계 없이 규칙만 있으면 상황에 맞지 않는 결과가 나온다. 스킬에는 이 규칙이 적용되지 않는 경우도 정의해야 한다.

**갱신 주기가 있어야 한다**: 스킬은 팀의 기준이 바뀌면 함께 바뀌어야 한다. 오래된 스킬을 아무도 업데이트하지 않으면 틀린 기준으로 계속 작동한다.

---

## 5. 스킬과 하네스 엔지니어링의 차이

스킬과 하네스 엔지니어링은 다른 역할을 한다.

스킬은 "어떻게 행동하라"는 행동 규칙이다. 목적지까지 어떤 경로로, 어떤 방식으로 가는지를 정의한다.

하네스 엔지니어링은 "목적지까지 안전하게 도착하는 구조"다. 스킬이 정의한 행동을 실제 작업 환경에서 안전하게 실행하고, 오류가 생겼을 때 복구하고, 중간 상태를 모니터링하는 시스템이다.

비유로 설명하면 이렇다. 스킬은 "직진 → 우회 → 대기"라는 주행 규칙이다. 하네스는 속도 센서, 장애물 감지 시스템, 비상 제동 장치처럼 그 규칙이 실제 도로에서 안전하게 작동하도록 감싸주는 구조다. 스킬이 "어떻게 뛰어라"라면, 하네스는 "도착까지 책임지는 구조"다.

---

## 6. 스킬 설계 체크리스트

- 이 스킬이 발동되는 트리거 조건이 명확한가
- 행동 순서와 판단 기준이 단계별로 정의되어 있는가
- 출력 형식이 지정되어 있는가
- 이 스킬이 적용되지 않는 경우가 정의되어 있는가
- 팀 기준이 바뀌면 이 스킬도 갱신할 체계가 있는가

---

## 7. 정리

스킬은 반복되는 AI 작업 패턴을 규칙화해서 재사용 가능하게 만드는 방식이다. 프롬프트가 단발 지시이고 컨텍스트가 작업 환경 설계라면, 스킬은 반복 가능한 행동 매뉴얼이다.

잘 만들어진 스킬은 작업 방식을 팀 안에서 표준화하고, 매번 처음부터 설명하지 않아도 되게 만든다. 하지만 스킬 자체는 그 행동이 안전하게 실행되는지, 오류가 생겼을 때 어떻게 되는지까지 책임지지 않는다. 그것은 하네스 엔지니어링의 역할이다.

---

## 다음 장으로의 연결

스킬은 어떻게 행동하라는 규칙이다. 하지만 규칙대로 행동하는 것과, 실제 목적지까지 안전하게 도착하는 것은 다르다. 중간에 예상치 못한 상황이 생겼을 때 어떻게 복구하고, 어디까지 진행됐는지 어떻게 파악하고, 오류가 생겼을 때 어떻게 멈추는가 — 그 구조를 다루는 것이 31장 제2절(하네스 엔지니어링)이다.

---

## 제2절. Harness Engineering: 하네스 엔지니어링

## 이 장의 목적

이 장은 이 책이 제안한 산출물 체계가 배포 직전까지만 유효한 문서 묶음이 아니라, 출시 이후 운영과 반복 개선까지 이어지는 지속 가능한 실행 체계가 되려면 무엇이 필요한지 설명한다. 동시에 실제 팀이 이 체계를 어떤 순서로 도입해야 하는지도 다룬다.

이 장은 후반부 4개 장 중 **실행 결론** 장이다. 무엇을 해야 하는가(운영 반영과 도입 순서)를 다루며, 리스크 경고는 32장, PM의 책임 기준은 33장, 사람으로서의 마무리는 에필로그에서 각각 다룬다.

이 장의 목적은 두 층으로 나뉜다.

- 방법론 층: 왜 운영 반영과 도입 로드맵이 필요한지 설명한다.
- 실용 층: 실제 팀이 어떤 순서로 문서/도구/운영 구조를 셋업해야 하는지 설명한다.

이 장에서 말하는 `운영 반영`은 단순히 운영 문서를 남기는 것이 아니다.  
운영 과정에서 나온 이슈, FAQ, 사용자 피드백, 로그 분석 결과를 다시 다음 RIB와 PRD 입력에 반영하는 구조를 뜻한다.

---

## 1. 왜 출시 이후가 더 중요해졌는가

기획 산출물이 배포 전까지만 중요하다고 생각하면, 조직은 배운 것을 시스템에 남기지 못한다. AI 시대에는 문서와 모델과 코드가 다음 개선의 입력이 되기 때문에 운영 이후 문서가 더 중요하다.

출시 이후에도 계속 생기는 일:

- 고객 문의
- 운영 예외
- 정책 해석 차이
- 보안/법무 이슈
- 실제 데이터 기반 개선 필요

즉 출시 이후 산출물은 부록이 아니라 `다음 RIB와 PRD를 만드는 입력 자산`이다.

---

## 2. 반드시 유지해야 하는 운영 산출물

- 릴리즈 노트
- FAQ / 고객센터 대응 가이드
- 운영 가이드
- RTM / Decision Log / Changelog 최신본
- 운영 이슈 로그

이 다섯 개가 있어야 운영 이슈가 다시 산출물 체인에 반영될 수 있다.

---

![](./figures/fig-30-adoption-roadmap.svg)

> 도식: 산출물 체계 도입 로드맵: Phase 1(기준 정렬) → Phase 2(파일럿) → Phase 3(추적 체계) → Phase 4(확장/자동화)

## 3. 실제 도입은 어떤 순서로 해야 하는가

좋은 체계를 본 팀은 종종 한 번에 다 바꾸려 한다. 하지만 현실에서는 `기준 → 파일럿 → 연결 → 확장` 순서가 가장 안전하다.

### Phase 1. 최소 기준 정렬

- 용어집
- 문서 경계
- 최소 템플릿
- Source of Truth 규칙
- 승인 상태

### Phase 2. 대표 케이스 1건 파일럿

- RIB
- PRD
- 프로젝트 정책서
- BPMN
- DMN
- 사양서
- 프로토타입
- 테스트케이스

### Phase 3. 추적 체계 도입

- Input Gate
- DoR / DoD
- RTM
- Decision Log
- Changelog

### Phase 4. 팀 확장과 자동화

- 주간 리뷰 리듬
- AI 초안 생성 워크플로
- FAQ / 릴리즈 노트 초안 자동화
- 운영 이슈 → RIB 반영 루프

### 도입 저항 패턴과 대응

산출물 체계 도입 시 가장 자주 만나는 저항은 다섯 가지다. 저항의 근원을 파악하면 설득 방식이 달라진다.

**저항 1 — "문서 쓸 시간이 없다"**

이 말의 실제 의미는 "지금 방식으로도 잘 되고 있다"거나 "추가 부담이 싫다"는 것이다. 기존에 문서 없이 개발해왔는데 별 문제가 없었다면, 저항은 타당한 경험에서 나온다.

대응: 문서를 추가하는 것이 아니라 기존 구두 결정을 구조화하는 것임을 보여준다. RIB 한 장으로 시작하되, 작성 시간이 30분 이내여야 한다. 시간이 더 걸리면 템플릿이 잘못된 것이다.

**저항 2 — "AI 쓰면 되는 거 아닌가요"**

AI에 대한 기대치가 높고, 산출물 체계를 AI가 대체할 것으로 보는 시각이다. 이 저항은 틀리지 않다 — AI는 초안을 빠르게 만든다.

대응: AI의 출력 품질은 입력의 품질에 달려 있다는 것을 직접 시연한다. 동일한 기능을 "로그인 만들어줘"로 요청한 결과와 RIB·정책서를 입력으로 준 결과를 비교한다. 차이가 논거보다 설득력이 강하다.

**저항 3 — "우리 팀에 맞지 않는 방식이다"**

조직 맥락에 대한 자부심이거나, 과거에 방법론 도입이 실패한 경험에서 비롯된 거부감이다.

대응: 전체 체계를 강요하지 않는다. "다음 프로젝트에서 정책서 한 장만 써보자"처럼 가장 작은 단위로 시작한다. Phase 1의 최소 기준 정렬부터 시작하면 팀이 저항 없이 따라올 수 있다.

**저항 4 — "위에서 시키지 않으면 못 한다"**

의사결정 권한이 없어 실행을 주저하는 패턴이다. PM 한 명이 바꾸고 싶어도 조직 구조 때문에 움직이지 못한다.

대응: 윗선의 승인을 기다리지 않고 할 수 있는 것부터 시작한다. RIB와 PRD는 PM 혼자 작성해도 된다. 정책서와 사양서는 개발자와 1:1로 시작할 수 있다. 팀 전체 합의가 필요한 건 DoR/DoD처럼 프로세스에 관련된 것뿐이다. 그것은 Phase 3에서 하면 된다.

**저항 5 — "해봤는데 중간에 흐지부지됐다"**

가장 현실적인 저항이다. 도입 초반에는 의지가 있었지만 유지하지 못한 경험이다.

대응: 실패 원인을 먼저 파악한다. 대부분은 템플릿이 너무 복잡했거나, 도입 책임자가 없었거나, 리뷰 리듬이 없었기 때문이다. 이번에는 Phase 4의 주간 리뷰를 먼저 확정한다. 문서를 만드는 것보다 검토하는 리듬을 먼저 세우는 것이 더 중요하다.

## 5. 이 정도면 뒤에서 실제 개발 체계가 굴러간다는 기준

아래가 보이면 뒤에서 바이브 코딩이나 AI 개발 체계도 어느 정도 안정적으로 굴러간다.

- PRD가 범위와 시나리오를 분명히 잡고 있다
- Feature / Scenario / AC가 구현 단위로 내려가 있다
- 정책서가 규칙과 예외를 정리한다
- BPMN이 흐름과 역할을 보인다
- DMN이 판단 규칙을 보인다
- 사양서가 상태/메시지/API/예외를 보인다
- 프로토타입이 경험을 보여준다
- 시스템 구성도가 모듈 경계를 보여준다
- Jira/VS Code/Git 연결이 된다
- RTM/Decision Log/Changelog가 운영된다

이 정도면 UML 산출물을 보고 개발 구조를 예측하듯, 이 산출물 체계를 보고도 “어떤 모듈과 화면과 API와 테스트가 나올지” 예측할 수 있다.

## 7. 인하우스형과 수주형은 도입 순서가 어떻게 달라지는가

도입 로드맵도 같은 방식으로 시작하면 비효율이 크다.

- 인하우스형은 대체로 `문제 정의 → RIB → PRD → 파일럿 산출물 체인 → 운영 반영` 순서가 맞다.
- 수주형은 대체로 `프로젝트 착수 자료 → 수주형 RIB → PRD → 검수 기준/RTM → 파일럿 산출물 체인 → 인수/운영` 순서가 맞다.

### 도입 순서 차이 요약

| 단계 | 인하우스형 | 수주형 |
| --- | --- | --- |
| 시작 입력 | 로드맵, 문제 정의, 운영 이슈 | 프로젝트 착수 자료, 고객 요청사항 |
| 1차 정렬 | RIB 작성과 성공 기준 정렬 | 수주형 RIB 작성과 범위/검수 기준 정렬 |
| 파일럿 핵심 | PRD, Feature / Scenario / AC, 프로토타입 | PRD, RTM, 사양서, 테스트케이스 |
| 중간 강화 포인트 | 정책서, BPMN, DMN, 운영 반영 구조 | BPMN, DMN, RTM, 승인 상태, 인수 자료 |
| 확산 전 체크 | 운영 반영 루프가 도는가 | 변경 통제와 검수 기준이 안정적인가 |

### 도입 관점에서 보면

- 인하우스형은 `빠르게 돌려보고 배우는 구조`를 먼저 세워야 한다.
- 수주형은 `어긋나지 않게 합의하고 추적하는 구조`를 먼저 세워야 한다.

이 차이는 [[inhouse-vs-delivery-deliverable-flow-v1]] 에 단계별로 더 상세히 정리돼 있다. 도입 순서 차이를 한 장으로 보려면 [[inhouse-vs-delivery-flow-overview-v1.canvas]] 를 같이 열어 보는 편이 빠르다.

## 8. 산출물 단계별 AI 활용 매트릭스

도입 로드맵을 실무에 내릴 때는 “어느 단계에서 AI를 어떻게 쓸지”가 먼저 보이는 편이 좋다.

| 단계 | AI가 잘하는 일 | 사람이 끝까지 책임질 일 |
| --- | --- | --- |
| RIB | 회의 메모 정리, 누락 질문 추출, 입력 구조 초안 | 문제 정의, 범위, 성공 기준 확정 |
| PRD | 초안 작성, 옵션 비교, Out of Scope 후보 정리 | 우선순위, 제품 방향, 승인 |
| Feature / Scenario / AC | 시나리오 분해, Given / When / Then 초안 | 완료 기준, 예외 처리, 검수 기준 |
| 정책서 | 규칙 충돌 후보, 예외 규칙 정리 | 정책 확정, 책임 경계 승인 |
| BPMN | 서술형 흐름을 BPMN 2.0 초안으로 변환 | 최종 흐름, 운영 개입, 역할 경계 확정 |
| DMN | 정책 문장을 decision table 초안으로 변환 | 최종 판단 규칙, 예외 승인 |
| 사양서 | 상태/입력/메시지/API/예외 표 초안 | 구현 조건 확정, 외부 연동 책임 결정 |
| 프로토타입 | 화면 목록, 상태별 문구, 검토 포인트 초안 | 경험 우선순위, `ready for dev` 승인 |
| 시스템 구성도 | 책임 경계 초안, 모듈 후보, 인터페이스 후보 | 최종 경계, 모듈 책임, 배포 구조 판단 |
| 테스트케이스 | 정상/예외 케이스 확장, 누락 케이스 탐지 | 통과 기준, 회귀 범위, 릴리즈 승인 |
| 운영 문서 | 이슈 분류, FAQ 초안, 변경 이력 요약 | 공식 운영 기준, 외부 공지, 반영 우선순위 |

### 도입 관점에서 핵심

- AI는 각 단계의 초안과 비교와 누락 탐지를 빠르게 한다.
- 사람은 승인, 우선순위, 정책, 책임 경계를 끝까지 붙든다.
- 결국 좋은 도입은 “프롬프트 잘 쓰기”가 아니라 “승인된 문맥 위에서 AI를 쓰기”다.

더 자세한 실무판은 [[ai-planning-practical-guide-v1]] 를 같이 보는 것이 좋다.

### 8-1. 피드백 관리와 백로그 연결도 도입 대상이다

도입 로드맵에서 자주 빠지는 것이 피드백 관리다. 하지만 실제 팀 운영에서는 운영 문의, VOC, 회의 메모, 고객 요청사항이 백로그와 문서 체인으로 다시 연결되지 않으면 AI 도입 효과가 금방 줄어든다.

- 회의 요약은 피드백 항목으로 남겨야 한다.
- 피드백은 관련 PRD / Idea / Story와 연결돼야 한다.
- 반복 테마는 우선순위 논의 입력으로 올라와야 한다.
- 이 연결은 AI가 도와줄 수 있지만, 최종 우선순위는 사람이 판단해야 한다.

즉 팀 도입은 `문서 작성 자동화`만이 아니라 `피드백 정리와 백로그 연결 자동화`까지 포함해야 한다.

## 9. 체크리스트

### 9-1. Phase 1 — 기준 정렬

- [ ] 용어집과 공통 정책이 조직 수준에서 정해졌는가
- [ ] 산출물 종류별 Source of Truth 위치가 합의됐는가
- [ ] 승인 상태와 문서 경계가 정의됐는가
- [ ] 최소 템플릿이 팀 공유 공간에 배포됐는가

### 9-2. Phase 2 — 파일럿 1건

- [ ] 대표 케이스 1건이 선정됐는가
- [ ] RIB → PRD → 정책서 → BPMN → DMN → 사양서 → TC 체인이 끝까지 돌았는가
- [ ] 각 산출물에 승인자와 리뷰어가 지정됐는가
- [ ] 파일럿 회고(어떤 산출물이 실제로 유지됐는가)가 진행됐는가

### 9-3. Phase 3 — 추적 체계

- [ ] Input Gate / DoR / DoD 기준이 정의됐는가
- [ ] RTM / Decision Log / Changelog가 프로젝트에 연결됐는가
- [ ] 운영 이슈와 FAQ가 다음 RIB 입력으로 연결되는 구조가 있는가

### 9-4. Phase 4 — 확산과 반복 가능성

- [ ] 주간 리뷰 리듬이 팀에 정착됐는가
- [ ] AI 초안 생성 흐름이 반복 가능한 워크플로로 정의됐는가
- [ ] 문서/모델/디자인/코드/운영 산출물이 다음 과제에서도 같은 체계로 연결되는가
- [ ] 한 번에 다 바꾸지 않고 단계별 확산 계획이 있는가

## 정리

좋은 산출물 체계는 배포 전 문서 묶음이 아니라 운영 이후에도 학습을 남기는 실행 체계다. 그리고 좋은 도입은 교육 자료 배포가 아니라, 대표 케이스 1건을 같은 규칙과 같은 도구 연결 위에서 끝까지 돌려보는 것에서 시작한다.

## 이 산출물은 4분면마다 어떻게 달라지는가

### 신규 구축 + 자사 서비스

- 도입은 작게 시작하고 빠르게 검증하는 방식이 적합하다.
- 대표 케이스 1건과 최소 문서 세트부터 시작해도 된다.
- 운영 리듬은 짧은 스프린트와 빠른 회고 중심이 좋다.

### 고도화/운영 + 자사 서비스

- 도입은 As-is 복원과 운영 이슈 수집부터 시작하는 것이 맞다.
- 정책서, BPMN, DMN, 테스트, 운영 문서까지 함께 셋업해야 효과가 크다.
- 운영 반영 구조를 만들지 않으면 도입 효과가 오래 가지 않는다.

### 신규 구축 + 수주형

- 도입은 템플릿 배포보다 승인 구조와 검수 기준 정렬이 먼저다.
- 고객 검토 흐름과 내부 승인 흐름을 같이 세팅해야 한다.
- 파일럿도 합의/인수 기준이 분명한 과제로 잡는 편이 좋다.

### 고도화/운영 + 수주형

- 도입은 레거시 복원, 영향도 분석, 변경 통제 기준부터 세워야 한다.
- RTM / Decision Log / Changelog, 운영 가이드까지 같이 굴려야 한다.

## 제3절. Harness Operating Environment: 하네스 운영 환경

## 이 장의 목적

이 장은 하네스가 실제로 작동하는 환경, 즉 하네스 운영 환경(Harness Operating Environment)이 무엇이고 왜 별도로 설계해야 하는지 다룬다. 하네스를 잘 만들어도 그것이 실행되는 환경이 통제되지 않으면 안전하고 반복 가능한 작동을 보장할 수 없다.

이 장을 읽고 나면 다음을 이해해야 한다.

- 하네스 운영 환경이 하네스 자체와 어떻게 다른지
- 운영 환경의 구성 요소
- 환경을 통제하지 않을 때 생기는 문제
- 운영 환경을 설계하고 관리하는 방법

---

## 1. 하네스와 운영 환경의 차이

하네스는 AI가 어떻게 작동해야 하는지를 정의하는 구조다. 어떤 툴을 쓰고, 어떤 순서로 실행하고, 어떤 조건에서 멈추는지를 제어한다.

하네스 운영 환경은 하네스가 실제로 실행되는 외부 조건이다. 하네스가 악보라면, 운영 환경은 그 악보가 연주되는 공연장이다. 아무리 좋은 악보도 음향이 망가진 공연장에서는 제대로 들리지 않는다.

운영 환경을 따로 설계해야 하는 이유:
- 같은 하네스도 환경에 따라 동작이 달라진다
- 환경이 통제되지 않으면 재현이 불가능해진다
- 환경의 변화가 하네스 동작에 영향을 주지만 로그에는 남지 않는다

---

## 2. 운영 환경의 구성 요소

### 2-1. 런타임 환경

하네스가 실행되는 실제 컴퓨팅 환경이다.

- 어떤 서버 또는 컨테이너에서 실행되는가
- 메모리, CPU, 타임아웃 제한이 어떻게 설정되어 있는가
- 실행 격리(sandbox)가 적용되어 있는가

런타임 환경이 불안정하면 하네스가 중간에 끊기거나 예상치 못한 방식으로 동작한다.

### 2-2. 툴 접근 환경

하네스가 사용하는 외부 도구와 API에 대한 접근 구조다.

- 어떤 API 키와 인증 정보가 사용되는가
- API 호출 제한(rate limit)이 어떻게 적용되는가
- 툴이 응답하지 않을 때 어떻게 처리되는가
- 개발 환경과 운영 환경에서 연결되는 툴이 동일한가

툴 접근 환경이 다르면 개발 환경에서 작동하던 하네스가 운영 환경에서 실패한다.

### 2-3. 컨텍스트 저장소

에이전트가 여러 스텝에 걸쳐 작업할 때 상태와 기억을 저장하는 구조다.

- 작업 중 생성된 중간 결과는 어디에 저장되는가
- 이전 대화나 결정 이력은 어떻게 유지되는가
- 저장소가 초기화되는 조건은 무엇인가

컨텍스트 저장소가 명확하지 않으면 하네스가 이전 작업 내용을 잊거나 오래된 상태를 참조한다.

### 2-4. 로그와 모니터링

하네스가 무엇을 했는지, 어디서 문제가 생겼는지 파악하는 구조다.

- 각 스텝의 입력과 출력이 기록되는가
- 오류 발생 시 원인을 추적할 수 있는가
- 비정상 동작을 감지하는 기준이 있는가
- 로그가 사람이 읽을 수 있는 형태로 남는가

로그가 없으면 하네스 동작 문제를 재현하거나 원인을 파악하기 어렵다.

### 2-5. 시크릿과 접근 제어

민감한 정보와 권한을 관리하는 구조다.

- API 키, 비밀번호, 토큰이 코드 외부에 안전하게 관리되는가
- 하네스가 접근할 수 있는 데이터와 시스템의 범위가 제한되어 있는가
- 접근 권한이 환경(개발/스테이징/운영)마다 다르게 설정되어 있는가

시크릿이 코드에 포함되거나 접근 범위가 통제되지 않으면 보안 리스크가 발생한다.

---

## 3. 환경을 통제하지 않을 때 생기는 문제

**재현 불가능**: 같은 입력을 줘도 다른 결과가 나온다. 원인이 하네스인지 환경인지 알 수 없다.

**예측 불가능한 실패**: 하네스 로직은 맞지만 툴 연결이 끊기거나 타임아웃이 발생해 작업이 중단된다.

**환경 간 불일치**: 개발 환경에서 잘 되던 것이 운영 환경에서 다르게 동작한다. 원인 파악이 어렵다.

**감사 불가능**: 무슨 일이 일어났는지 추적할 수 없어 문제 발생 시 책임 소재가 불분명해진다.

---

## 4. 운영 환경 설계 원칙

**환경을 코드처럼 관리한다**: 환경 설정이 문서나 개인 기억에만 있으면 안 된다. 코드와 함께 버전 관리되어야 한다.

**환경을 분리한다**: 개발, 스테이징, 운영 환경을 분리하고 각 환경에서 동작이 다를 수 있음을 전제로 설계한다.

**시크릿은 코드 밖에 둔다**: API 키, 비밀번호, 토큰은 환경 변수나 시크릿 관리 도구로 분리한다. 코드에 직접 넣지 않는다.

**로그를 기본으로 설계한다**: 무슨 일이 일어났는지 나중에 추적할 수 있도록 모든 주요 스텝의 입출력을 기록한다.

**접근 범위를 최소화한다**: 하네스가 작업에 필요한 것보다 더 넓은 범위에 접근하지 않도록 제한한다.

---

## 5. 하네스 운영 환경 체크리스트

- 하네스가 실행되는 런타임 환경이 문서화되어 있는가
- 툴과 API 접근에 필요한 인증 정보가 코드 외부에 안전하게 관리되는가
- 개발 환경과 운영 환경의 차이가 명시되어 있는가
- 주요 스텝의 입출력이 로그로 남는가
- 오류 발생 시 원인을 추적할 수 있는가
- 하네스가 접근 가능한 범위가 업무 목적으로 제한되어 있는가
- 환경 설정이 버전 관리되고 있는가

---

## 6. 정리

하네스 운영 환경은 하네스 자체와 별개로 설계하고 관리해야 한다. 아무리 잘 만든 하네스도 환경이 통제되지 않으면 안전하고 반복 가능하게 작동하지 않는다. 런타임, 툴 접근, 컨텍스트 저장소, 로그, 시크릿 관리가 갖춰진 환경 위에서 하네스가 실행될 때 비로소 신뢰할 수 있는 AI 운영이 가능하다.
