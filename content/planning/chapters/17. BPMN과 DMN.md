# 17장. BPMN과 DMN: 흐름과 판단 고정하기

> 이 장은 정책서에서 확정한 규칙을 두 가지 방식으로 구조화하는 것을 다룬다. BPMN은 누가·언제·어떤 순서와 책임으로 흐름이 진행되는지를 정의한다. DMN은 흐름 안의 판단 지점에서 어떤 조건 조합이 어떤 결과를 내는지를 정의한다. 흐름(BPMN)과 판단(DMN)은 분리하여 관리해야 한다.

---

> **전제 지식**: 16장(프로젝트 정책서) — BPMN은 정책서에서 확정한 규칙을 흐름으로 구조화한다. 정책서 없이 BPMN을 그리면 레인과 분기의 기준이 모호해진다.

## 이 장의 목적

이 장은 정책서에서 확정한 규칙을 실제 업무 흐름으로 구조화하는 `BPMN(Business Process Model and Notation)`의 역할과 작성 방법을 설명한다. 이 책에서 BPMN은 예쁜 프로세스 그림을 그리는 기술이 아니다. 누가·언제·어떤 순서와 책임으로 일을 처리하는지를 팀이 공통으로 이해하게 만드는 `실행 흐름 문서`다.

이 장을 읽고 나면 다음을 할 수 있어야 한다.

- 정책서의 규칙을 BPMN 흐름으로 변환하는 기준을 설명할 수 있다.
- UX 플로우와 BPMN의 차이를 구분할 수 있다.
- AI를 활용해 BPMN을 만들 때 무엇을 맡기고 무엇을 직접 확정해야 하는지 구분할 수 있다.
- 완성된 BPMN이 사양서·프로토타입·테스트케이스로 어떤 입력을 넘기는지 확인할 수 있다.

> **BPMN을 처음 접하는 독자에게**
>
> BPMN은 국내 기획·PM 실무에서 사용 빈도가 낮은 편이다. Pool·Lane·Gateway·Task·Event 같은 개념이 처음에는 낯설 수 있다. 이 장은 BPMN 입문서가 아니므로 기호 전체를 설명하지는 않는다. 대신 기획자가 실무에서 반드시 사용하는 최소한의 요소만 다루고, 그 요소가 왜 필요한지를 중심으로 설명한다.
>
> 처음 접하는 독자라면 §1~§3을 읽으면서 "이 흐름을 그림으로 표현할 때 무엇이 필요한가"를 먼저 이해하고, 도구(Camunda Modeler, draw.io, Miro)는 나중에 선택해도 된다. BPMN 기호를 익히는 데 걸리는 시간은 보통 2~3시간이면 기본 수준에 도달한다. 전문적으로 익히는 데는 더 걸리지만, 기획자에게 필요한 수준은 그 정도면 충분하다.

## 1. 이 산출물이 왜 존재하는가

PRD와 정책서는 문제·범위·규칙을 설명한다. 그러나 이 두 문서만으로는 실제 협업에서 반드시 생기는 질문을 막지 못한다.

- 어디서 시스템이 검증하고 어디서 운영자가 개입하는가
- 실패 시 어느 단계로 되돌아가는가
- 상태는 어느 시점에 바뀌는가
- 외부 인증과 내부 회원 서비스는 어떤 순서로 상호작용하는가

이 질문들은 규칙의 영역이 아니라 흐름의 영역이다. 흐름이 문서화되지 않으면 팀은 각자 다르게 해석한다. 기획이 A라고 이해한 것을 개발은 B로 구현하고 QA는 C 기준으로 검수한다. BPMN은 이 해석 차이를 막는 문서다.

BPMN이 특히 필요한 상황:
- 사용자·시스템·운영자 등 역할이 둘 이상 얽혀 있을 때
- 분기와 예외 흐름이 정상 흐름보다 많을 때
- 상태 전환 시점이 구현 결과를 결정할 때
- 운영의 수동 개입이 설계에 포함될 때

반대로 화면 한두 개가 바뀌는 단순 개선처럼 역할·분기·상태 변화가 거의 없으면 BPMN은 과할 수 있다.

## 2. 입력과 출력

### 2-1. BPMN이 받는 주요 입력

| 입력 소스 | 내용 |
|---|---|
| 정책서 | 기본 규칙·예외 규칙·운영 승인 기준 — BPMN 흐름 후보 |
| PRD | 핵심 시나리오·역할 목록·과제 범위 |
| Feature 정의 | 이번 BPMN이 다루는 기능 목록 |
| 시나리오 / AC | 정상·예외·운영 시나리오 |
| 권한 정책 매트릭스 | 역할별 허용/금지 기준 |
| As-is BPMN (고도화 시) | 현재 프로세스 — 변경점 식별 기준 |

### 2-2. BPMN이 넘겨주는 출력

| 다음 산출물 | BPMN에서 받는 것 |
|---|---|
| **DMN** | 판단이 필요한 게이트웨이 지점 (조건 조합 구조화 의뢰) |
| **사양서** | 처리 순서·예외 분기·상태 전환 타이밍·외부 시스템 호출 지점·운영 개입 지점 |
| **프로토타입** | 예외 상태에서 사용자가 받을 피드백 기준·분기별 화면 요구 |
| **테스트케이스** | 정상 흐름·예외 흐름·분기별 기대 결과·상태 전환 결과 |
| **운영 정책서** | 운영 개입 지점·수동 처리 기준 |

> **핵심 원칙**: 흐름은 BPMN이 맡는다. 판단 기준은 DMN으로, 구현 조건은 사양서로, 경험 검토는 프로토타입으로 분리한다.

## 3. BPMN 구조와 구성 요소

### 3-1. UX 플로우와 BPMN의 차이

두 문서는 모두 흐름을 다루지만 답하는 질문이 다르다.

| 관점 | UX 플로우 | BPMN |
|---|---|---|
| 주요 질문 | 사용자가 어떤 화면을 보는가 | 누가 어떤 이벤트에서 일을 시작하는가 |
| 분기 기준 | 사용자 선택·경험 | 시스템 판단·역할 조건 |
| 운영 개입 | 표현 안 함 | 별도 Lane으로 명시 |
| 상태 변화 | 화면 전환으로 암시 | 명시적 상태 변화 이벤트로 표기 |
| 외부 연동 | 표현 안 함 | 외부 시스템 Pool/Lane으로 명시 |

UX 플로우는 사용자 경험을 검토하고, BPMN은 처리 책임과 흐름의 정확성을 검토한다. 둘은 대체재가 아니라 보완재다.

### 3-2. BPMN 핵심 구성 요소

| 요소 | 설명 | 사용 기준 |
|---|---|---|
| **Pool** | 프로세스 전체 참여자 묶음 (예: 회원 서비스) | 외부 시스템이 있으면 별도 Pool |
| **Lane** | Pool 내 역할 구분 (사용자·시스템·운영자) | 역할이 2개 이상이면 Lane 사용 |
| **Start Event** | 프로세스 시작 지점 | 반드시 명시 |
| **End Event** | 프로세스 종료 지점 | 정상·예외 종료 모두 명시 |
| **Task** | 실제 수행 작업 | 주어 + 목적어 형식으로 기술 |
| **Gateway (XOR)** | 조건 분기 — 하나만 선택 | 판단 지점마다 명시 |
| **Gateway (AND)** | 병렬 처리 — 모두 실행 | 병렬 작업이 있을 때 |
| **Intermediate Event** | 중간 이벤트 (타이머·오류·메시지) | 외부 연동·타임아웃 있을 때 |
| **Message Flow** | Pool 간 메시지 흐름 | 외부 시스템 연동 표현 시 |

### 3-3. BPMN 구조 텍스트 예시 (AI 입력 형식)

BPMN 도구에 넣기 전, 다음 형식으로 먼저 구조를 정리한다.

```
프로세스명: 로그인 처리

참여자:
  - Pool: 회원 서비스
    - Lane: 사용자
    - Lane: 인증 시스템
    - Lane: 운영자
  - Pool: 외부 인증 서비스 (본인확인)

시작 이벤트: 사용자 로그인 시도

주요 단계:
  1. 자격증명 입력 (Lane: 사용자)
  2. 자격증명 검증 (Lane: 인증 시스템)
  3. [XOR] 검증 결과 판단
     → 성공: 계정 상태 조회
     → 실패: 실패 횟수 증가
  4. [XOR] 계정 상태 판단 (← DMN 의뢰)
     → 활성: 로그인 성공 처리
     → 휴면: 본인확인 요청
     → 잠금: 잠금 안내 및 해제 경로 제공
  5. 본인확인 처리 (Pool: 외부 인증 서비스)
  6. [XOR] 본인확인 결과
     → 성공: 상태 전환 → 로그인 성공
     → 실패: 운영 개입 요청 또는 종료
  7. 운영자 수동 처리 (Lane: 운영자)

종료 이벤트:
  - 로그인 성공
  - 잠금 상태 유지
  - 본인확인 실패 종료
```

## 단계별 워크스루: 한글 정책 문서 → BPMN

BPMN을 처음 접하는 팀이 가장 막히는 지점은 "정책 문서가 있는데 어디서부터 시작하는가"다. 도구(Camunda, draw.io)를 열기 전에, 한글 정책 문장에서 BPMN 요소를 식별하고 텍스트 구조를 완성하는 5단계를 먼저 익혀야 한다.

아래 로그인 정책 예시를 입력으로 받아 각 단계를 순서대로 진행한다.

**[입력: 한글 정책 문서 발췌]**

```
P-01: 사용자가 아이디와 비밀번호를 입력하면 시스템이 자격증명을 검증한다.
P-02: 검증 실패 횟수가 5회 이상이면 계정을 잠금 상태로 전환한다.
P-03: 잠금 상태 계정은 본인확인 성공 시 잠금을 해제할 수 있다.
P-04: 본인확인은 외부 본인확인 서비스를 통해 처리한다.
P-05: 잠금 해제가 불가능한 경우 운영자가 수동으로 검토 후 처리한다.
P-06: 로그인 성공 시 세션을 생성하고 메인 페이지로 이동한다.
```

---

### Step 1. 주체 식별 → Pool/Lane 결정

정책 문장의 **주어**를 찾는다. 주어가 Lane이 된다.

| 정책에서 발견한 주어 | 역할 분류 | Lane/Pool |
|---|---|---|
| 사용자 | 서비스 내부 행위자 | Lane: 사용자 |
| 시스템 | 서비스 내부 처리 | Lane: 인증 시스템 |
| 운영자 | 서비스 내부 역할 | Lane: 운영자 |
| 외부 본인확인 서비스 | 외부 시스템 | **별도 Pool** |

> **판단 기준**: 같은 서비스 내 역할은 Lane, 외부 API·타사 시스템은 별도 Pool로 분리한다. Pool 간 통신은 Message Flow로 표현한다.

결정: Pool 2개 (회원 서비스 / 외부 본인확인 서비스), Lane 3개 (사용자 / 인증 시스템 / 운영자)

---

### Step 2. 이벤트 식별 → 시작·종료·중간 이벤트 결정

프로세스의 **경계**를 먼저 확정한다. 경계가 없으면 흐름이 어디서 끊기는지 알 수 없다.

**시작 이벤트 후보**: "~를 시도한다", "~를 요청한다", "~가 발생한다"
→ P-01: "아이디와 비밀번호를 입력한다" → **[시작 이벤트: 로그인 시도]**

**종료 이벤트 후보**: "~가 완료된다", "~로 이동한다", "~가 거부된다"
→ P-06: "메인 페이지로 이동한다" → **[종료 이벤트: 로그인 성공]**
→ P-02: 잠금 전환 후 더 이상 처리 없음 → **[종료 이벤트: 잠금 유지]**
→ P-05: 운영자 처리 후 종료 → **[종료 이벤트: 운영 처리 완료]**

> **판단 기준**: 종료 이벤트는 정상 1개 + 예외 N개로 설계한다. 예외 분기에 종료가 없으면 흐름이 끊긴다. 나중에 추가하는 것보다 지금 식별하는 것이 낫다.

---

### Step 3. 태스크 식별 → Task 목록 작성

**행동 문장**("~를 한다", "~를 처리한다")을 Task로 변환한다. 태스크명은 `[Lane: 동사+목적어]` 형식을 원칙으로 한다.

| 정책 문장 | Task명 | Lane |
|---|---|---|
| 아이디와 비밀번호를 입력한다 | 자격증명 입력 | 사용자 |
| 시스템이 자격증명을 검증한다 | 자격증명 검증 | 인증 시스템 |
| 계정을 잠금 상태로 전환한다 | 실패 횟수 증가 및 잠금 전환 | 인증 시스템 |
| 잠금 안내를 제공한다 (묵시적) | 잠금 안내 제공 | 인증 시스템 |
| 외부 본인확인을 요청한다 | 본인확인 요청 | 인증 시스템 |
| 외부 서비스가 본인확인을 처리한다 | 본인확인 처리 | 외부 Pool |
| 운영자가 수동으로 검토한다 | 수동 검토 및 처리 | 운영자 |
| 세션을 생성한다 | 세션 생성 | 인증 시스템 |

> **주의**: 하나의 Task는 하나의 Lane에만 속한다. 두 Lane이 동시에 수행하는 것처럼 보이면 Task를 분리해야 한다. "묵시적"으로 존재하는 Task(정책에서 명시하지 않았지만 흐름상 필요한 것)는 괄호로 표시하고 나중에 PM이 확인한다.

---

### Step 4. 분기 식별 → Gateway 결정

**조건 문장**("~이면", "~인 경우", "~가 성공하면/실패하면")을 찾는다. 이것이 XOR 게이트웨이가 된다.

| 정책에서 발견한 조건 | Gateway명 | 분기 조건 | DMN 분리 여부 |
|---|---|---|---|
| P-02: 실패 횟수 5회 이상 여부 | [XOR: 실패 횟수 판단] | < 5회 / ≥ 5회 | 단순 이진 → 게이트웨이 직접 명시 |
| P-03: 본인확인 성공 여부 | [XOR: 본인확인 결과] | 성공 / 실패 | 단순 이진 → 게이트웨이 직접 명시 |
| (묵시적) 계정 상태 판단 | [XOR: 계정 상태 판단] | 활성 / 휴면 / 잠금 | 조건 조합 복잡 → **DMN 분리** |
| (묵시적) 운영자 처리 결과 | [XOR: 운영 처리 결과] | 해제 / 거부 | 단순 이진 → 게이트웨이 직접 명시 |

> **DMN 분리 기준**: 입력 변수가 2개 이상이거나 조합이 복잡한 판단은 DMN으로 분리한다. 단순 이진 분기(Y/N, 성공/실패)는 게이트웨이에 직접 쓴다. 판단을 게이트웨이에 몰아넣으면 흐름도가 읽기 어려워진다.

---

### Step 5. 흐름 연결 → BPMN 텍스트 구조 완성

1~4단계에서 식별한 요소를 순서대로 연결한다. 도구를 열기 전에 텍스트 구조를 먼저 완성한다.

```
풀: 회원 서비스
  레인: 사용자 / 인증 시스템 / 운영자

[시작 이벤트: 로그인 시도]
  └─ [Task: 자격증명 입력] (사용자)
  └─ [Task: 자격증명 검증] (인증 시스템)
  └─ [XOR: 실패 횟수 판단]
       ├─ < 5회 → [XOR: 계정 상태 판단 ← DMN-01] (인증 시스템)
       │            ├─ 활성  → [Task: 세션 생성] → [종료: 로그인 성공]
       │            ├─ 휴면  → [Task: 본인확인 요청] → (Message Flow → 외부 Pool)
       │            └─ 잠금  → [Task: 잠금 안내 제공] → [종료: 잠금 유지]
       └─ ≥ 5회 → [Task: 실패 횟수 증가 및 잠금 전환] (인증 시스템)
                   └─ [Task: 본인확인 요청] → (Message Flow → 외부 Pool)
                   └─ [XOR: 본인확인 결과]
                        ├─ 성공 → [Task: 세션 생성] → [종료: 로그인 성공]
                        └─ 실패 → [Task: 수동 검토 요청] (인증 시스템)
                                   └─ [Task: 수동 검토 및 처리] (운영자)
                                   └─ [XOR: 운영 처리 결과]
                                        ├─ 해제 → [Task: 세션 생성] → [종료: 로그인 성공]
                                        └─ 거부 → [종료: 잠금 유지]

풀: 외부 본인확인 서비스
  [Task: 본인확인 처리]
  └─ [Message Flow → 인증 시스템: 본인확인 결과 반환]
```

> **검증 포인트**: 모든 게이트웨이 분기에 종료 이벤트가 연결되는지 확인한다. 연결이 끊긴 분기가 있으면 흐름이 미완성이다.

---

### 5단계 요약

| 단계 | 찾는 것 | 결과물 |
|---|---|---|
| 1. 주체 식별 | 문장의 주어 | Pool / Lane 목록 |
| 2. 이벤트 식별 | 시작·끝·외부 연동 시점 | 시작·종료·중간 이벤트 목록 |
| 3. 태스크 식별 | 행동 문장 | Task 목록 (Lane별) |
| 4. 분기 식별 | 조건 문장 | Gateway 목록 + DMN 분리 후보 |
| 5. 흐름 연결 | 순서와 경로 | BPMN 텍스트 구조 |

텍스트 구조가 완성되면 Camunda Modeler 또는 draw.io에서 그림으로 옮긴다. 반대로 도구를 먼저 열면 요소 분류를 건너뛰게 되어 빠진 분기가 나중에 발견된다.

## 4. AI와 함께 만드는 법

### 4-1. 흐름 개요

```
PM → 정책서 + 시나리오 전달 → AI 구조 초안 → PM 역할·분기 검토 → BPMN 도구 보정 → PM 승인
```

### 4-2. AI에게 맡기는 단계

**단계 1: 정책서 → BPMN 구조 초안 요청**

```
다음 정책서와 시나리오를 읽고 BPMN 흐름 구조 초안을 만들어줘.

[정책서 내용 붙여넣기]
[핵심 시나리오 붙여넣기]

출력 형식:
1. 참여자 목록 (Pool / Lane)
2. 시작 이벤트
3. 주요 Task 목록 (Lane별)
4. XOR 게이트웨이 후보와 분기 조건
5. 예외 흐름 목록
6. 종료 이벤트 (정상·예외 구분)
7. DMN으로 분리해야 할 판단 지점

주의: 판단 기준 전체를 게이트웨이에 넣지 말 것.
판단 기준은 DMN으로 분리 예정이므로 게이트웨이에는 판단명만 표기할 것.
```

**단계 2: 누락 예외 흐름 보완 요청**

```
위 BPMN 초안에서 다음 예외 시나리오가 커버되는지 확인하고,
빠진 것이 있으면 추가해줘.

- 외부 본인확인 서비스 타임아웃 발생 시
- 운영자 수동 처리 중 재요청 발생 시
- 연속 실패로 계정 잠금 전환 시
```

**단계 3: BPMN XML 초안 요청 (Camunda용)**

```
위 BPMN 구조를 Camunda Modeler에서 열 수 있는
BPMN 2.0 XML 초안으로 변환해줘.
Pool, Lane, Task, XOR Gateway, Start/End Event를 포함할 것.
DMN 판단 지점은 Business Rule Task로 표기할 것.
```

### 4-3. PM이 직접 판단·확정해야 하는 것

| 항목 | 이유 |
|---|---|
| 실제 운영 개입 지점 확정 | 조직 운영 구조·책임 소재 판단 필요 |
| 상태 전환 시점 확정 | 비즈니스 규칙과 기술 제약 동시 고려 필요 |
| 외부 시스템 연동 순서 | 실제 API 계약·응답 시간 고려 필요 |
| DMN 분리 지점 확정 | 흐름과 판단의 경계는 PM이 결정 |
| 법·보안 필수 예외 경로 | 누락되면 컴플라이언스 문제 발생 |

> **원칙**: AI는 BPMN 초안을 빠르게 구조화할 수 있다. 그러나 **실제 업무 흐름의 책임 구조 확정**은 사람이 맡는다.

### 4-4. Camunda Modeler 활용 흐름

1. VS Code 또는 AI 채팅에서 BPMN 구조 텍스트·XML 초안 생성
2. `.bpmn` 파일로 저장
3. Camunda Modeler에서 열어 Lane·Gateway·Event 시각 보정
4. Business Rule Task에 DMN 연결 지점 표시
5. 최종본을 정책서·사양서·테스트케이스와 연결

> **→ 프롬프트 예시**: [[wiser-prompt-examples-v1]] — 3. BPMN 초안 생성 예시

## 6. 다음 단계 연결 규칙

### 6-1. BPMN → 사양서·DMN 전환 조건

다음 조건이 충족되어야 사양서 작성을 시작한다.

- 정상 흐름과 주요 예외 흐름이 완성되었다
- 역할·운영 개입·상태 전환이 명시되었다
- DMN으로 분리할 판단 지점이 표시되었다
- Camunda Modeler에서 오류 없이 검증되었다
- BPMN이 정책서·PRD와 범위 일치가 확인되었다

### 6-2. BPMN이 열어주는 병렬 작업

BPMN이 확정되면 다음 산출물을 병렬로 시작할 수 있다.

| 병렬 가능 산출물 | BPMN에서 받는 것 |
|---|---|
| DMN | 게이트웨이 판단 지점 목록 |
| 프로토타입 | 예외 상태별 화면 요구·분기별 UX |
| 사양서 (초안) | 처리 단계·상태 전환·외부 호출 지점 |

### 6-3. BPMN에서 해결하지 않는 것

BPMN은 다음을 결정하지 않는다.

- 판단 조건의 세부 기준 (→ DMN에서)
- 화면 UI 설계 (→ 프로토타입에서)
- API 필드·응답 포맷·에러 코드 (→ 사양서에서)
- 검증 케이스 (→ 테스트케이스에서)

판단 기준이 BPMN 게이트웨이에 쌓이면 흐름도가 읽기 어려워지고, 사양서·테스트케이스가 BPMN과 중복된다.

## 적용 예시: 회원가입/로그인 정책 복원 과제

정책서의 BPMN 후보를 다음 3개의 흐름으로 분리한다.

**BPMN-01: 로그인 처리 흐름**
참여자: 사용자 / 인증 시스템 / 외부 본인확인 / 운영자
DMN 연결: 로그인 허용 여부 판단, 잠금 해제 가능 여부 판단

**BPMN-02: 휴면회원 재활성화 흐름**
참여자: 사용자 / 회원 서비스 / 외부 본인확인
DMN 연결: 휴면 해제 가능 여부 판단

**BPMN-03: 탈퇴 후 재가입 흐름**
참여자: 사용자 / 회원 서비스 / 운영자
DMN 연결: 재가입 허용 여부 판단

하나의 BPMN에 모든 흐름을 넣으면 읽기 어렵다. 판단 기준이 얽힌 지점마다 BPMN을 분리하고 DMN으로 연결하는 것이 실무적으로 더 안정적이다.

## 정리

BPMN은 프로세스를 예쁘게 그리는 기법이 아니다. 좋은 BPMN은 일이 실제로 어떤 흐름과 책임 구조 안에서 처리되는지를 팀이 같은 그림으로 보게 만든다.

BPMN에서 기억할 세 가지:

1. **흐름과 책임을 구조화한다** — 역할·분기·상태 전환·운영 개입을 명시한다.
2. **판단은 DMN으로 분리한다** — 게이트웨이에 판단 기준을 몰아 넣지 않는다.
3. **사양서·프로토타입·테스트케이스의 공통 입력이 된다** — BPMN이 확정되어야 뒤 산출물이 같은 기준으로 시작된다.

## 다음 장으로의 연결

BPMN이 완성되면 Gateway에서 판단이 필요한 분기를 별도의 구조로 상세화할 차례다. BPMN의 분기 조건 중 "복수의 입력 조건이 결합되는 판단"은 BPMN 안에 쓰면 가독성이 무너진다. 이것을 분리해 표 형태로 구조화하는 것이 DMN이다.

### BPMN에서 DMN으로 이동하는 항목

| BPMN 항목 | 이동 형태 | DMN에서의 역할 |
|---|---|---|
| 복수 조건이 결합된 Gateway | 조건 변수 목록 | Decision Table의 Input 컬럼 |
| Gateway의 결과 분기값 | 결과 레이블 목록 | Decision Table의 Output 컬럼 |
| 판단 반복 패턴 (동일 구조의 Gateway) | 동일 조건 구조 여부 | Hit Policy 결정 근거 |
| 외부 데이터를 참조하는 분기 | 참조 데이터 소스 이름 | DMN Input Expression 정의 |
| 타임아웃·기간 기반 분기 | 기간 변수 | DMN 시간 조건 Input |

### 체인 전환 기준 (BPMN → DMN)

| 확인 항목 | 기준 |
|---|---|
| Gateway 식별 완료 | DMN으로 분리할 판단 지점이 BPMN에서 명시되어 있는가 |
| 입력 변수 목록 | 각 판단의 입력 변수가 최소 2개 이상 식별되어 있는가 |
| 결과 레이블 확정 | 각 분기의 결과값이 이름으로 확정되어 있는가 |
| BPMN 상태 | `Approved` 상태인가 |

- **이 장이 결정한 것**: 흐름 구조, 책임 분리, 처리 순서, 이벤트·예외 경로
- **다음 장이 시작하는 것**: BPMN Gateway의 복합 판단 조건을 DMN Decision Table로 구조화

> **이 장에서 다루는 것**
> - DMN의 역할과 BPMN 게이트웨이와의 관계
> - AI와 함께 판단 테이블을 만드는 3단계
> - DMN → 사양서·테스트케이스 연결 기준

![](./figures/fig-18-bpmn-vs-dmn.svg)

> 도식: BPMN 게이트웨이 → DMN 판단 테이블 구조: 흐름 제어와 판단 분리

> **전제 지식**: 17장 제1절(BPMN), 16장(프로젝트 정책서) — DMN은 BPMN의 게이트웨이에서 드러난 판단 지점을 구조화한다. BPMN을 먼저 그려봐야 어디에 DMN이 필요한지 감이 온다.

## 이 장의 목적

이 장은 BPMN의 게이트웨이에서 드러난 판단 지점을 조건과 결과의 구조로 정리하는 `DMN(Decision Model and Notation)`의 역할과 작성 방법을 설명한다. 이 책에서 DMN은 단순한 표 작성법이 아니다. 사람의 기억과 회의 속에 흩어진 판단 규칙을 입력 조건·규칙·결과의 구조로 바꿔 개발·QA·운영이 같은 기준으로 반복 적용하게 만드는 `판단 문서`다.

이 장을 읽고 나면 다음을 할 수 있어야 한다.

- BPMN 게이트웨이를 DMN 의사결정 테이블로 변환하는 방법을 설명할 수 있다.
- 정책 문장을 입력 조건·규칙 행·결과로 분해할 수 있다.
- AI를 활용해 DMN을 만들 때 무엇을 맡기고 무엇을 직접 확정해야 하는지 구분할 수 있다.
- 완성된 DMN이 사양서·테스트케이스·운영 문서로 어떤 입력을 넘기는지 확인할 수 있다.

---

## 1. 이 산출물이 왜 존재하는가

실무에서 흐름보다 더 자주 흔들리는 것은 판단이다. 프로세스 순서는 팀마다 비슷하게 설명하지만, 다음 질문 앞에서는 답이 갈린다.

- 이 사용자는 지금 로그인 가능한가
- 이 계정은 재가입을 허용해야 하는가
- 잠금 해제는 어떤 조건을 충족해야 가능한가
- 운영 승인 예외는 어디까지 허용되는가

규칙은 이미 정책서에 문장으로 존재한다. 그러나 문장으로 존재한다고 판단 기준이 구조화된 것은 아니다. 예를 들어 "휴면회원은 본인확인 후 로그인 가능"이라는 문장은 명확해 보이지만, 실제로는 다음 조건들이 숨어 있다.

- 본인확인 방식은 무엇인가
- 본인확인 실패 후 재시도는 몇 번 허용되는가
- 운영 승인 예외가 가능한가
- 상태 전환은 언제 확정되는가

이처럼 하나의 정책 문장은 실제로 여러 입력 조건과 결과를 가진다. 그 판단을 문장 안에만 두면 사양서·QA·운영·개발이 서로 다른 기준을 적용한다. DMN은 이 구멍을 막는 문서다.

DMN이 특히 필요한 상황:
- 입력 조건이 2개 이상이고 조합이 복잡할 때
- 같은 규칙이 정책서·사양서·테스트케이스·운영 가이드에 반복될 때
- 예외 규칙이 기본 규칙보다 많을 때
- 변경 가능성이 높은 판단 로직일 때

단순 서비스의 경우 정책서 텍스트로 충분하다. 조건 분기가 3개 미만이고 예외가 없으면 DMN이 과할 수 있다.

## 2. 입력과 출력

### 2-1. DMN이 받는 주요 입력

| 입력 소스 | 내용 |
|---|---|
| 정책서 | 기본 규칙·예외 규칙·우선순위 |
| BPMN | DMN으로 분리할 판단 게이트웨이 지점 목록 |
| 권한 정책 매트릭스 | 역할별 허용/금지 기준 |
| 용어집 / 상태 정의 | 입력 조건에 사용할 용어·상태값 |

### 2-2. DMN이 넘겨주는 출력

| 다음 산출물 | DMN에서 받는 것 |
|---|---|
| **사양서** | 입력값 정의·결과값 정의·조건 조합별 처리 기준·우선순위와 예외 처리 |
| **테스트케이스** | 규칙 행별 기대 결과·경계값·허용/차단/예외 케이스 |
| **운영 정책서** | 운영 승인 예외 기준·상태별 처리 가이드·복구 기준 |
| **개발** | 구현 판단 로직 기준 (코드로 옮길 규칙 표) |

> **핵심 원칙**: DMN의 각 규칙 행은 테스트케이스 후보다. DMN이 확정되면 사양서·테스트케이스가 같은 기준 위에서 시작된다.

## 3. DMN 구조와 템플릿

### 3-1. 정책서 · BPMN · DMN의 역할 비교

| 문서 | 역할 | 답하는 질문 |
|---|---|---|
| 정책서 | 규칙의 배경·원칙 설명 | 왜 이런 규칙이 필요한가 |
| BPMN | 흐름과 책임 구조 | 언제 판단이 일어나는가, 누가 처리하는가 |
| DMN | 판단 기준 구조화 | 어떤 조건 조합에서 어떤 결과를 내는가 |

### 3-2. DMN 의사결정 테이블 구조

DMN 테이블은 4가지 영역으로 구성된다.

| 영역 | 내용 |
|---|---|
| **판단명 (Decision Name)** | 이 테이블이 내리는 판단의 이름 |
| **입력 컬럼 (Input)** | 판단에 필요한 조건 변수 목록 |
| **출력 컬럼 (Output)** | 판단 결과값 목록 |
| **규칙 행 (Rule Row)** | 입력 조건 조합별 결과 |

**Hit Policy(적중 정책)** — 가장 자주 쓰는 두 가지:
- `U (Unique)`: 조건이 겹치지 않음. 하나의 행만 매칭됨 — 기본값
- `F (First)`: 위에서부터 첫 번째 매칭 행을 사용 — 예외 우선순위 적용 시

### 3-3. 의사결정 테이블 예시: 로그인 허용 여부

| # | 계정 상태 | 실패 횟수 | 본인확인 성공 | 운영 예외 승인 | **결과** | **후속 처리** |
|---|---|---|---|---|---|---|
| 1 | 활성 | < 5 | — | — | **허용** | 로그인 성공 처리 |
| 2 | 활성 | ≥ 5 | — | — | **잠금 전환** | 잠금 안내 및 해제 경로 제공 |
| 3 | 휴면 | — | true | — | **허용** | 상태 활성 전환 후 로그인 |
| 4 | 휴면 | — | false | — | **거부** | 본인확인 재시도 또는 운영 요청 |
| 5 | 잠금 | — | true | — | **해제 후 허용** | 잠금 해제 후 로그인 |
| 6 | 잠금 | — | false | true | **해제 후 허용** | 운영 승인 기록 후 잠금 해제 |
| 7 | 잠금 | — | false | false | **거부** | 잠금 유지, 안내 제공 |
| 8 | 탈퇴 | — | — | — | **거부** | 재가입 경로 안내 |

> 이 표의 각 행이 테스트케이스 1개씩 대응된다.

### 3-4. 템플릿 (마크다운 형식)

```markdown
# DMN: [판단명]

판단 ID: DMN-[번호]
연결 BPMN: [[BPMN 파일명]] — [게이트웨이 이름]
연결 정책: P-[번호] ([정책 항목명])
Hit Policy: U (Unique) / F (First)
작성일: YYYY-MM-DD
작성자:

---

## 판단 목적
[이 DMN이 내리는 판단을 한 문장으로]

## 의사결정 테이블

| # | [입력1] | [입력2] | [입력3] | **결과** | **후속 처리** |
|---|---|---|---|---|---|
| 1 | | | | | |
| 2 | | | | | |

## 예외 및 우선순위
- [예외 조건과 처리 방법]
- [우선순위 규칙]

## 참조
- 정책서: P-[번호]
- BPMN: [[파일명]] — 게이트웨이 [이름]
- 테스트케이스 후보: TC-[번호 범위]
```

## 4. AI와 함께 만드는 법

### 4-1. 흐름 개요

```
PM → 정책서 + BPMN 판단 지점 전달 → AI 테이블 초안 → PM 조건·결과 검증 → Camunda 보정 → PM 승인
```

### 4-2. AI에게 맡기는 단계

**단계 1: 정책 문장 → DMN 테이블 초안 요청**

```
다음 정책서의 규칙을 DMN 의사결정 테이블로 변환해줘.

[정책 항목 붙여넣기]
예: 로그인 허용 여부 판단
- P-01: 실패 5회 누적 시 잠금
- P-02: 잠금 해제는 본인확인 또는 운영 승인
- P-03: 휴면은 본인확인 성공 시 허용
- E-01: 운영자 계정은 본인확인 예외

출력 형식:
- 판단명
- 입력 컬럼 목록 (조건 변수)
- 출력 컬럼 목록 (결과값)
- 규칙 행 (조건 조합별 결과)
- Hit Policy 추천과 이유
- 예외 및 우선순위 메모
```

**단계 2: 누락 조건·충돌 검토 요청**

```
위 DMN 테이블에서 다음을 검토해줘.

1. 입력 조건 조합 중 매칭 규칙이 없는 빈 케이스
2. 두 개 이상의 규칙 행이 동시에 매칭될 수 있는 충돌 케이스
3. 정책서 내용과 불일치하는 행
4. 테스트케이스로 추가해야 할 경계값 후보
```

**단계 3: 테스트케이스 후보 추출 요청**

```
위 DMN 테이블의 각 행을 테스트케이스로 변환해줘.

형식:
- TC 번호
- 입력 조건값
- 기대 결과
- 검증 포인트
```

### 4-3. PM이 직접 판단·확정해야 하는 것

| 항목 | 이유 |
|---|---|
| 입력 조건 최종 목록 | 어떤 변수를 판단에 사용할지 비즈니스 결정 |
| 경계값 확정 | "≥ 5회"처럼 수치 경계는 정책 책임자가 확정 |
| 예외 우선순위 결정 | 충돌하는 규칙 간 우선순위는 PM이 확정 |
| Hit Policy 선택 | 비즈니스 의도에 따라 U/F 선택 |
| BPMN 연결 지점 확인 | DMN이 BPMN의 어느 게이트웨이에 연결되는지 PM이 확인 |

> **원칙**: AI는 DMN 초안을 빠르게 구조화할 수 있다. 그러나 **규칙의 책임 있는 확정**은 사람이 맡는다.

### 4-4. Camunda Modeler 활용 흐름

1. AI가 만든 DMN 테이블 구조를 `.dmn` 파일로 변환 요청
2. Camunda Modeler에서 열어 입력·출력 컬럼·규칙 행 시각 보정
3. FEEL 식(조건 표현식)과 타입 점검
4. DRD(Decision Requirements Diagram)로 복수 판단 간 연결 표현
5. 최종본을 사양서·테스트케이스·운영 정책서와 연결

> **→ 프롬프트 예시**: [[wiser-prompt-examples-v1]] — 4. DMN Decision Table 초안 예시

## 5. 품질 기준 / 체크리스트

### 5-1. 필수 확인 (이것이 없으면 사양서 작성을 시작하면 안 된다)

- [ ] BPMN의 판단 지점과 연결되는가
- [ ] 입력 컬럼과 출력 컬럼이 명확한가
- [ ] 모든 입력 조건 조합에 대응하는 규칙 행이 있는가
- [ ] 규칙 행 간 충돌이 없는가 (또는 Hit Policy로 해소되었는가)
- [ ] 예외와 우선순위가 명시되었는가

### 5-2. 품질 향상 항목

- [ ] 각 규칙 행이 테스트케이스 후보로 연결 가능한가
- [ ] 경계값(5회, 30일 등)이 정책서 수치와 일치하는가
- [ ] Camunda Modeler에서 오류 없이 검증되었는가
- [ ] 복수 DMN이 있는 경우 DRD로 연결 구조가 보이는가

### 5-3. 자주 발생하는 오류

| 오류 유형 | 현상 | 수정 방향 |
|---|---|---|
| 조건 공백 | 일부 입력 조합에 매칭 규칙이 없음 | 빠진 조합에 기본 결과 행 추가 |
| 규칙 충돌 | 두 행이 동시에 매칭 가능 | Hit Policy를 F로 변경하거나 조건을 더 구체화 |
| 정책 불일치 | 정책서 수치와 DMN 경계값이 다름 | 정책서 기준으로 수치 통일 |
| 과도한 입력 | 판단과 무관한 변수가 입력 컬럼에 포함 | 이 판단에 실제로 영향을 주는 변수만 포함 |

## 6. 다음 단계 연결 규칙

### 6-1. DMN → 사양서·테스트케이스 전환 조건

다음 조건이 충족되어야 사양서·테스트케이스 작성을 시작한다.

- 모든 BPMN 판단 지점에 대응하는 DMN이 완성되었다
- 규칙 행 간 충돌이 없거나 Hit Policy로 명확히 해소되었다
- 정책서 수치와 DMN 경계값이 일치한다
- Camunda Modeler에서 오류 없이 검증되었다

### 6-2. DMN이 열어주는 병렬 작업

DMN이 확정되면 다음 산출물을 병렬로 시작할 수 있다.

| 병렬 가능 산출물 | DMN에서 받는 것 |
|---|---|
| 사양서 | 입력값·결과값 정의·조건 조합별 처리 기준 |
| 테스트케이스 | 규칙 행별 기대 결과·경계값·예외 케이스 |
| 운영 정책서 | 운영 승인 예외 기준·복구 기준 |

### 6-3. DMN에서 해결하지 않는 것

DMN은 다음을 결정하지 않는다.

- 프로세스 흐름과 책임 구조 (→ BPMN에서)
- API 필드·응답 포맷·에러 코드 (→ 사양서에서)
- 화면 UI 설계 (→ 프로토타입에서)
- 검증 실행 순서와 범위 (→ 테스트 계획서에서)

DMN에 흐름이 섞이면 BPMN과 중복되고, DMN에 구현 조건이 섞이면 사양서와 경계가 무너진다.

## 적용 예시: 회원가입/로그인 정책 복원 과제

BPMN의 판단 게이트웨이 3개를 각각 별도 DMN으로 분리한다.

**DMN-01: 로그인 허용 여부**
입력: 계정 상태·실패 횟수·본인확인 성공 여부·운영 예외 승인 여부
결과: 허용 / 잠금 전환 / 본인확인 필요 / 거부
테스트케이스 후보: 8개 (위 3-3 테이블 참조)

**DMN-02: 재가입 허용 여부**
입력: 탈퇴 상태 여부·탈퇴 후 경과 기간·계정 유형·제재 이력 여부
결과: 즉시 허용 / 제한 기간 후 허용 / 운영 승인 후 허용 / 불가
테스트케이스 후보: 6개

**DMN-03: 잠금 해제 가능 여부**
입력: 잠금 사유·경과 시간·본인확인 성공 여부·운영 승인 여부
결과: 자동 해제 / 본인확인 후 해제 / 운영 승인 후 해제 / 불가
테스트케이스 후보: 5개

DMN 3개의 규칙 행 합계: 19개 → 테스트케이스 19개의 직접 기준이 된다.

## 정리

DMN은 표를 예쁘게 만드는 기법이 아니다. 좋은 DMN은 사람이 기억으로 처리하던 규칙을 조건과 결과의 구조로 바꿔, 개발·QA·운영이 같은 판단 기준을 보게 만든다.

DMN에서 기억할 세 가지:

1. **BPMN의 판단 지점을 구조화한다** — 흐름은 BPMN이, 기준은 DMN이 맡는다.
2. **각 규칙 행은 테스트케이스 후보다** — DMN이 완성되면 TC 목록이 거의 자동으로 나온다.
3. **사양서·테스트케이스·운영의 공통 기준이 된다** — DMN이 확정되어야 세 산출물이 같은 기준 위에서 시작된다.

## 다음 장으로의 연결

DMN이 완성되면 "누가 무엇을 할 수 있는가"를 명시적으로 정의할 차례다. DMN의 Role 입력이 권한 매트릭스의 행이 되고, DMN의 접근 가능 여부 결과가 권한 정책의 기능 제한 항목이 된다.

### DMN에서 권한 정책 매트릭스로 이동하는 항목

| DMN 항목 | 이동 형태 | 권한 정책에서의 역할 |
|---|---|---|
| Role을 Input으로 쓰는 Decision | 역할 목록 (GUEST/USER/ADMIN 등) | 권한 매트릭스 행(역할) 정의 |
| 특정 역할만 접근 가능한 규칙 | 역할-기능 제한 조건 | 권한 매트릭스의 기능 허용/거부 |
| 데이터 접근 범위 판단 | 소유자·범위 조건 | 데이터 접근 권한 정의 |
| 역할 조합 판단 (AND 조건) | 복합 역할 조건 | 다중 역할 권한 처리 방식 |
| 결과값 "접근 불가" 경로 | 거부 조건 | 권한 매트릭스 거부 사유 |

### 체인 전환 기준 (DMN → 권한 정책 매트릭스)

| 확인 항목 | 기준 |
|---|---|
| Role 입력 식별 | DMN에서 역할(Role)을 Input으로 쓰는 Decision이 식별되어 있는가 |
| 접근 결과 확정 | 역할별 허용/거부 결과값이 DMN에서 확정되어 있는가 |
| 데이터 범위 조건 존재 | 본인 데이터 vs. 전체 데이터 접근 범위 판단이 있는가 |
| DMN 상태 | `Approved` 상태인가 |

- **이 장이 결정한 것**: 조건 기반 판단 구조, 규칙 행 목록, 결과값 체계
- **다음 장이 시작하는 것**: 정책·흐름·판단이 확정된 서비스 안의 정보를 사용자 관점의 구조(IA)와 콘텐츠 모델로 정리
