# 21장. 사양서


> **이 장에서 다루는 것**
> - 사양서의 역할과 구현 조건 문서 개념
> - AI와 함께 사양서를 구조화하는 3단계
> - 사양서 → API 명세·테스트케이스 연결 규칙

![](./figures/fig-19-spec-chain.svg)

> 도식: 사양서 산출물 체인: BPMN·DMN 입력 → API·비즈니스 로직·오류 처리·성능보안 4섹션 → 개발·TC·RTM 출력

> **전제 지식**: 16장(프로젝트 정책서), 17장(BPMN과 DMN) — 사양서는 BPMN의 흐름·레인·예외와 DMN의 판단 테이블을 구현 조건으로 변환한 문서다. 두 산출물을 먼저 작성해야 사양서 작성 범위가 확정된다.

## 이 장의 목적

이 장은 PRD, 정책서, BPMN, DMN으로 정리된 요구사항을 실제 구현 가능한 조건으로 번역하는 사양서의 역할을 다룬다. 이 책에서 말하는 사양서는 과거식 상세설계서의 부활이 아니라, AI와 개발자가 같은 기준으로 읽을 수 있는 `구현 조건 문서`다.

이 장의 목적은 두 층으로 나뉜다.

- 방법론 층: 사양서가 왜 필요한지, 다른 산출물과 무엇이 다른지 설명한다.
- 실용 층: 어느 수준까지 써야 뒤에서 AI 코드 생성, 바이브 코딩, QA 검증이 가능한지 설명한다.

---

## 1. 이 산출물이 왜 존재하는가

PRD만 잘 쓰면 개발이 된다는 생각은 실무에서 거의 항상 무너진다. 실제 구현 단계에서는 아래 질문이 나온다.

- 어떤 입력이 유효한가
- 어떤 조건에서 어떤 상태로 바뀌는가
- 어떤 메시지를 노출하는가
- 어떤 API를 호출하는가
- 어떤 경우는 화면에 보여주고 어떤 경우는 로그만 남기는가
- 어떤 값이 필수이고 어떤 값이 선택인가

이 질문에 답하지 못하면 사람 개발자도, AI도 각자 해석으로 빈칸을 채운다. 따라서 사양서는 방향 문서가 아니라 `해석 차이를 줄이는 문서`다.

---

## 2. 무엇을 입력으로 받고 무엇을 넘기는가

### 2-1. 사양서가 받는 입력

| 입력 소스 | 내용 |
|---|---|
| PRD | 기능 범위·핵심 시나리오·성공 기준 — 무엇을 구현할 것인지 방향 |
| 정책서 | 허용·차단·예외 규칙·역할별 처리 기준 — 조건절의 근거 |
| BPMN | 흐름·역할·판단 분기 지점 — 어떤 경로에서 어떤 처리가 일어나는지 |
| DMN | 조건 조합별 기대 결과 — 입력값·결과값 정의 |
| NFR | 성능·보안·가용성 기준 — 구현 조건에 반영할 수치 기준 |
| 프로토타입 | 화면 상태·UI 조건·사용자 경험 흐름 — 화면 메시지·상태 표현 기준 |

### 2-2. 사양서가 넘겨주는 출력

| 다음 산출물 | 받는 것 |
|---|---|
| **API 명세서** | 인터페이스·요청/응답 포맷·에러 코드·상태 정의 |
| **테스트케이스(TC)** | 상태값·메시지·API 응답 — 기대 결과의 검증 기준 |
| **개발팀** | 기능별 구현 조건·입력 검증 규칙·오류 처리 기준 |
| **RTM** | 기능 항목 ID — 요구사항 추적 연결 |
| **운영 정책서(OPD)** | 상태 전이 기준·에러 처리 패턴 — 운영 규칙 갱신 근거 |

> **핵심 원칙**: 사양서는 방향 문서가 아니라 해석 차이를 줄이는 문서다. PRD가 "무엇을"을 정했다면, 사양서는 "어떤 조건으로"를 정한다. 이 문서가 없으면 사람 개발자도, AI도 각자 해석으로 빈칸을 채운다.

---

## 3. 이 책에서 말하는 사양서의 정의

사양서는 다음처럼 정의할 수 있다.

> 기능이 어떤 입력, 어떤 조건, 어떤 상태, 어떤 메시지, 어떤 인터페이스를 기준으로 동작해야 하는지를 구현 가능한 수준으로 적는 문서

즉:

- PRD는 왜/무엇을 정의한다.
- 정책서는 규칙을 정의한다.
- BPMN은 흐름을 정의한다.
- DMN은 판단을 정의한다.
- 사양서는 그 내용을 `개발 가능한 조건`으로 바꾼다.

---

## 4. 뒤에서 개발이 나오려면 어느 수준까지 써야 하는가

사양서는 “읽으면 이해된다” 수준으로는 부족하다. `이 정도면 뒤에서 바이브 코딩이나 AI 코드 생성이 가능하다`는 기준으로 쓰려면 최소 아래가 보여야 한다.

- 상태 정의
- 입력값과 검증 규칙
- 이벤트와 전이 조건
- 화면/메시지 결과
- API 또는 시스템 상호작용
- 실패/예외 처리
- 로그/감사 기록 필요 여부

즉 구현자가 “무엇을 만들지”뿐 아니라 “무엇을 만들면 안 되는지”까지 보여야 한다.

## 4-A. AI 시대 Feature Spec의 5항목 구조

사양서가 어느 수준이어야 AI가 실제로 읽을 수 있는 입력이 되는가. 이론 수준의 체크리스트가 아니라 실무에서 검증된 최소 항목으로 정리하면 다섯 가지다.

| 항목 | 질문 | 없으면 생기는 문제 |
|---|---|---|
| ① 무엇을 만드는가 | 기능 범위, In/Out Scope, 주요 시나리오 | AI가 기능 경계를 추정해서 필요 없는 것을 구현하거나 필요한 것을 빠뜨린다 |
| ② 어떤 규칙으로 동작하는가 | 조건·정책·예외 | 조건이 없으면 AI가 "일반적인 방식"으로 채운다. 그 결과가 실제 정책과 다를 수 있다 |
| ③ 어떤 상태와 오류로 동작하는가 | 상태 전이, 오류 코드, API 명세 | 상태가 없으면 예외 처리가 빠진다. 오류 코드가 없으면 클라이언트와 서버가 각자 코드를 만든다 |
| ④ 어떻게 완료됐다고 판단하는가 | AC(인수 조건): Given/When/Then + 엣지 케이스 | 완료 기준이 없으면 개발자와 QA가 각자 기준을 만든다 |
| ⑤ 사용자에게 어떻게 보이는가 | 화면 목록, 상태별 UI, 오류 문구 | UI가 없으면 프론트엔드가 오류 메시지와 화면 상태를 재량으로 결정한다 |

### 5항목의 특징

이 5항목은 각기 다른 사람이 읽는다. ①은 팀 전체가, ②는 개발자·QA가, ③은 백엔드 개발자·QA가, ④는 QA·PM이, ⑤는 프론트엔드 개발자·디자이너가 주로 참조한다. 이 항목 중 하나가 빠지면 해당 역할의 사람이 빈칸을 자기 해석으로 채운다.

### 복잡도에 따른 추가 산출물 기준

기능 하나의 사양서를 작성할 때 복잡도에 따라 필요한 산출물이 달라진다.

| 복잡도 | 기준 | 필요한 산출물 |
|---|---|---|
| **단순** | 판단 조건 3개 이하, 역할 1개 | Feature Spec 1장으로 충분 |
| **중간** | 판단 조건 4~7개, 역할 2개 이상 | Feature Spec + 정책서 또는 DMN 추가 |
| **복잡** | 판단 조건 8개 이상, 역할 3개 이상 | Feature Spec + BPMN + DMN |

## 4-B. 서비스 사양서: Foundation + Feature Spec 2레벨 구조

기능이 하나일 때는 사양서 1장으로 충분하다. 그러나 서비스 전체에 기능이 여러 개 쌓이면 공통 정책, 오류 코드, 상태 정의가 각 Feature Spec에 중복되기 시작한다. 중복된 내용이 기능마다 조금씩 다르게 기술되면 개발자가 어떤 기준을 따를지 판단하지 못한다.

이 문제를 해결하는 구조가 **2레벨 설계**다.

### 레벨 1: Service Foundation (서비스 기반 문서)

서비스 전체에 적용되는 공통 내용을 한 곳에서 정의한다. 1회 작성하고 Feature Spec에서 참조만 한다.

Foundation이 담는 내용:

| 항목 | 예시 |
|---|---|
| 서비스 개요 및 기능 목록 | F-01 회원가입, F-02 로그인, F-03 소셜 로그인 |
| 핵심 엔티티 상태 정의 | 회원: PENDING_VERIFY / ACTIVE / AUTO_LOCKED / DORMANT / WITHDRAWN |
| 공통 정책 (ID 체계로 관리) | AUTH-P-01 세션 유효기간 24시간, PW-P-01 비밀번호 복잡도 기준 |
| 공통 오류 코드 (전체 목록) | AUTH_001~007, VAL_001~003, MEMBER_001~002 |
| 공통 API 규칙 | 인증 헤더 형식, 오류 응답 포맷 |
| 용어집 | 이메일 로그인 계정 vs 소셜 로그인 계정 구분 등 |
| 공통 UI 가이드 | 오류 노출 위치, 버튼 로딩 처리 방식 |
| 이번 릴리즈 Out Scope | Foundation을 읽는 AI가 포함하지 말아야 할 항목 명시 |

### 레벨 2: Feature Spec (기능별 문서)

기능마다 작성한다. 이 기능에만 있는 고유 정책, 상태, 오류, UI만 기술한다. Foundation과 중복되는 내용은 참조 표기로만 남긴다.

```
> 인증 정책 → Foundation AUTH-P-01~07 적용
> 비밀번호 정책 → Foundation PW-P-01, PW-P-02 적용
> 오류 코드 → Foundation 5-1 AUTH_* 참조
> UI 오류 노출 방식 → Foundation 8-1 참조
```

Foundation에서 정의한 AUTH_002 오류 메시지를 변경해야 할 경우 Foundation 1곳만 수정하면 해당 정책을 참조하는 모든 Feature Spec에 반영된다. 수정 지점이 분산되지 않는다.

### Foundation을 언제 만드는가

서비스 기능이 3개 이상 겹치는 공통 요소(상태, 정책, 오류 코드)가 나타나는 시점이 Foundation을 작성할 신호다. 기능이 하나일 때는 Feature Spec 1장에 모두 담고, 기능이 늘어나며 공통 요소가 반복되기 시작하면 Foundation을 분리한다.

> **→ 실전 예시**: [[foundation-auth-service-v1]] / [[feature-spec-F01-signup-v1]] / [[feature-spec-F02-login-v1]]
> 회원/인증 서비스의 실제 Foundation과 F-01, F-02 Feature Spec 예시가 appendix에 포함되어 있다.

## 6. 사양서를 실제로 쓰는 순서

### 5-1. 흐름 기준으로 기능을 쪼갠다

먼저 BPMN을 보고 어떤 단계가 실제 구현 단위가 되는지 나눈다.

### 5-2. 판단 기준을 별도로 매핑한다

DMN이 있는 기능은 규칙을 장문으로 다시 쓰지 말고, 어떤 조건이 어떤 처리로 연결되는지만 구현 관점으로 연결한다.

### 5-3. 상태와 메시지를 확정한다

실무에서 가장 자주 흔들리는 부분이다.

### 5-4. API와 데이터 처리를 명확히 적는다

바이브 코딩이나 AI 코드 생성으로 가려면 이 부분이 중요하다.

### 5-5. 프로토타입과 상호 검증한다

사양서의 상태, 메시지, 액션이 프로토타입에 드러나는지 맞춰봐야 한다.

## 7. 이 정도면 뒤에서 바이브 코딩이 가능하다는 기준

사양서가 아래 수준이면 뒤에서 AI 개발이나 바이브 코딩 입력으로 쓸 수 있다.

### 6-1. 입력값과 상태가 명확하다

예:

- `accountStatus: active | dormant | locked | withdrawn`
- `failedLoginCount: number`
- `identityVerified: boolean`

### 6-2. 조건별 결과가 분명하다

예:

- dormant + identityVerified=false → login blocked, identity verification required
- locked + adminApproval=true → unlock allowed

### 6-3. 화면/메시지/로그 결과가 같이 적혀 있다

예:

- 화면: 잠금 안내 모달 노출
- 메시지: “로그인 실패가 누적되어 계정이 잠겼습니다.”
- 로그: security.login.locked event 기록

### 6-4. API/이벤트 흐름이 보인다

예:

- POST /login
- 실패 시 audit event 저장
- 잠금 해제 시 notification event 발행

즉, 읽고 나면 “이 로직을 함수/컴포넌트/API로 어떻게 짤지”가 바로 떠오르는 수준이어야 한다.

## 8. 시스템 구성도와 사양서는 어디서 만나는가

사양서는 기능 레벨 구현 조건 문서이고, 시스템 구성도는 그 기능이 어떤 컴포넌트와 서비스 위에서 돌아가는지를 보여준다. 둘은 같이 가야 한다.

예를 들어 로그인 잠금 기능이라면:

- 사양서는 어떤 입력과 상태와 메시지가 필요한지 쓴다.
- 시스템 구성도는 Auth API, User Service, Security Module, Notification Service가 어떻게 연결되는지 보여준다.

즉 사양서만 있으면 로직은 보이지만 시스템 경계가 약해지고, 구성도만 있으면 구조는 보이지만 구현 조건이 약해진다.

## 10. AI와 함께 만드는 법

사양서는 이 책의 산출물 체인에서 AI 역할이 가장 큰 단계 중 하나다. BPMN(흐름) + DMN(판단) + ERD(데이터) + Feature/Scenario(기준)가 모두 입력으로 들어와서, 구현 가능한 조건으로 변환되는 지점이기 때문이다. 반대로, 이 변환에서 PM이 직접 판단해야 하는 항목이 빠지면 개발 단계에서 해석 차이가 그대로 코드로 들어간다.

### 12-1. Step 1: BPMN + DMN → Spec 초안 생성

**입력**

- BPMN 흐름 (텍스트 설명 또는 마크다운 형식)
- DMN 의사결정 테이블 (규칙 행 전체)
- Feature / Scenario 목록 (14장 결과)
- 정책서 규칙 항목

**AI 역할**: BPMN 각 단계와 DMN 규칙을 사양서 항목(상태·입력·메시지·API·예외) 형식으로 초안 변환한다. Feature별로 섹션을 나눠서 출력한다.

**프롬프트 예시**

```
아래 BPMN 흐름과 DMN 테이블을 기반으로 사양서 초안을 만들어줘.
Feature별로 섹션을 나눠서 각 섹션에 아래 항목을 포함해:
- 상태와 전이 (시작/종료/중간 상태, 전이 조건)
- 입력과 검증 규칙 (필수/선택, 포맷, 중복 체크)
- 메시지와 피드백 (성공/오류/차단/복구 안내)
- 인터페이스와 연동 (호출 API, 요청/응답 값, 실패 처리)
- 예외 처리 (타임아웃, 권한 없음, 상태 불일치, 중복 요청)
- 테스트 포인트 (정상/예외/경계값)

[BPMN 흐름]
(붙여넣기)

[DMN 의사결정 테이블]
(붙여넣기)

[Feature/Scenario 목록]
(붙여넣기)
```

**PM 검토 기준**

- BPMN의 모든 게이트웨이 분기가 Spec의 조건별 결과에 반영됐는가
- DMN 의사결정 테이블의 모든 규칙 행이 Spec 조건으로 내려왔는가
- 정책서 예외 규칙이 빠지지 않았는가
- 상태 정의가 ERD/TDD의 데이터 구조와 일치하는가

---

### 12-2. Step 2: 누락 엣지케이스·예외 처리 보완

**입력**

- Spec 초안 (Step 1 결과 또는 사람이 보정한 것)

**AI 역할**: Spec 초안을 읽고 경계값 / 예외 케이스 / 운영 개입 필요 시나리오 중 누락 가능성이 높은 것을 목록으로 출력한다.

**프롬프트 예시**

```
아래 사양서 초안을 보고 누락됐을 가능성이 있는 예외 케이스와
엣지케이스를 분류해서 목록으로 만들어줘.
각 항목은 다음 형식으로:
- 케이스명
- 어떤 상황에서 발생하는가
- 현재 Spec에서 처리 방식이 명시됐는가 (예/아니오)
- 처리 제안 (있으면)

[사양서 초안]
(붙여넣기)
```

**PM 검토 기준**

- 누락 목록 중 이번 범위에 반드시 포함해야 하는 것은 무엇인가
- 어떤 예외는 운영 정책서(OPD)로 처리하고, 어떤 것은 Spec에서 명시할 것인가
- 타임아웃, 권한 없음, 중복 요청 처리 기준이 정책서와 일치하는가

---

### 12-3. Step 3: API 명세 구조화

**입력**

- Spec의 인터페이스·연동 항목
- ERD / TDD 데이터 모델 (있으면)
- 기존 API 목록 (레거시 연계가 있으면)

**AI 역할**: 인터페이스 항목을 기반으로 API 엔드포인트 / 요청 파라미터 / 응답 형식 / HTTP 에러 코드 초안을 생성한다.

**프롬프트 예시**

```
아래 사양서의 인터페이스·연동 항목을 기반으로 API 명세 초안을 만들어줘.
각 API는 다음 형식으로:
- 엔드포인트: METHOD /path
- 요청 파라미터 (필수/선택, 타입, 설명)
- 성공 응답 (상태 코드, 응답 구조)
- 실패 응답 (에러 코드별 메시지)
- 비고 (비동기 처리, 로그 기록 여부 등)

ERD의 데이터 타입과 불일치하는 항목이 있으면 표시해줘.

[인터페이스·연동 항목]
(붙여넣기)

[ERD/TDD 데이터 모델]
(붙여넣기)
```

**PM 검토 기준**

- API 파라미터가 ERD/TDD의 데이터 타입과 일치하는가
- 에러 코드가 정책서 조건과 연결되는가
- 보안·권한 처리 설계가 포함됐는가 (인증 토큰, 권한 레벨 등)
- 비동기 이벤트가 필요한 부분이 명시됐는가

---

### 12-3-A. Foundation 기반 Feature Spec 생성 (다기능 서비스일 때)

서비스 기능이 여러 개인 경우, BPMN + DMN → Spec 생성 전에 Foundation을 먼저 입력으로 확보한다.

**입력**
- Service Foundation 전체 (공통 정책, 상태, 오류 코드, UI 가이드 포함)
- 이 기능만의 고유 규칙 (Foundation에 없는 것만)

**AI 역할**: Foundation을 컨텍스트로 읽고 Feature Spec 5항목(①~⑤)의 초안을 생성한다. Foundation과 중복되는 내용은 참조 표기로 처리한다.

**프롬프트 예시**

```
[Foundation 전체 내용 붙여넣기]

---
위 Foundation을 참조해서 아래 기능의 Feature Spec을 작성해줘.
Foundation에 이미 정의된 내용은 복사하지 말고 참조 표기로만 남길 것.

기능명: [기능 이름]
기능 개요: [2~3문장]
이 기능만의 고유 규칙: [Foundation에 없는 이 기능 전용 정책]

출력 형식:
① 무엇을 만드는가 (기능 범위, 시나리오, In/Out Scope)
② 어떤 규칙으로 동작하는가 (고유 정책만 — 공통은 Foundation 참조 표기)
③ 어떤 상태와 오류로 동작하는가 (상태 전이, 오류 코드, API 명세)
④ 어떻게 완료됐다고 판단하는가 (AC: Given/When/Then, 엣지 케이스)
⑤ 사용자에게 어떻게 보이는가 (화면 목록, 상태별 UI, 오류 문구)
```

**PM 검토 기준**

Foundation 없이 생성한 초안과 Foundation 기반으로 생성한 초안의 차이는 오류 코드, 상태값, 공통 정책의 일관성이다. Foundation 기반 초안에서 PM이 집중해서 확인해야 하는 것은 AI가 Foundation을 참조하지 않고 새로 정의한 항목이다. 이 항목들이 실제로 고유 정책인지, 아니면 Foundation과 중복 정의됐는지 확인한다.

| 확인 항목 | 이유 |
|---|---|
| AI가 신규 정의한 오류 코드 | Foundation에 이미 있는 코드와 충돌하는지 확인 |
| AI가 새로 기술한 정책 | 이 기능 전용인지, Foundation의 공통 정책으로 올려야 하는지 판단 |
| Foundation 참조 표기 정확성 | 잘못된 섹션 번호나 코드 ID 참조가 있는지 확인 |
| Out Scope 준수 | Foundation에 명시된 이번 릴리즈 미구현 항목이 AI 초안에 포함됐는지 확인 |

---

### 12-4. AI가 할 수 없는 것

이 단계에서 AI에게 맡기면 안 되는 판단이 있다. 이 네 가지는 PM 또는 아키텍트가 직접 결정해야 한다.

- **아키텍처 결정**: 어떤 서비스에서 이 로직을 처리할지는 시스템 전체 구조 판단이다. AI는 단일 기능 관점에서만 본다.
- **성능·보안 요건 기준**: 동시 요청 처리, 암호화 레벨, 토큰 만료 기준은 도메인 정책이다.
- **레거시 연계 제약**: 기존 API나 DB 구조의 제약은 AI가 모른다. PM이 직접 확인하고 Spec에 반영해야 한다.
- **운영 개입 기준**: 어떤 예외가 자동 처리되고 어떤 것이 수동 운영 대응인지는 운영 정책 결정이다.

> **→ 프롬프트 예시**: [[wiser-prompt-examples-v1]] — 5. 사양서 초안 예시

- [ ] 예외 처리 항목(타임아웃, 권한 없음, 상태 불일치)이 있는가

**연결 기준**
- [ ] BPMN 흐름의 모든 분기가 Spec 조건으로 내려왔는가
- [ ] DMN 의사결정 테이블의 모든 규칙 행이 반영됐는가
- [ ] ERD/TDD의 데이터 타입과 Spec 파라미터가 일치하는가

**품질 기준**

- [ ] 이 수준이면 뒤에서 바이브 코딩·AI 코드 생성 입력으로 쓸 수 있는가

## 12. 다음 단계 연결 규칙

사양서가 완성되면 아래 산출물로 연결된다.

| 이 산출물 | 넘어가는 다음 산출물 | 연결 내용 |
|---|---|---|
| **Spec (상태·조건)** | 테스트케이스(TC) | Spec의 테스트 포인트 항목 → TC 입력 조건. Spec 조건 1행 = TC 후보 1개 |
| **Spec (메시지·화면)** | 프로토타입 | 메시지·화면 항목 ↔ 프로토타입 화면 상호 검증 |
| **Spec (API·인터페이스)** | 개발 구현 | 입력값·상태·API가 개발 구현 기준 |
| **Spec (참조 산출물)** | RTM | PRD 참조 항목 → RTM의 요구사항 추적 기준 |
| **Spec (예외 처리)** | 운영 정책서(OPD) | 운영 개입 필요 예외 → OPD 규칙 항목 |
| **Spec (SAD 참조)** | 시스템 구성도 | Spec의 서비스 경계 → 시스템 아키텍처와 정합성 확인 |

**연결 확인 규칙**

- Spec에 "잠금 해제 시 notification event 발행"이 있다면, BPMN에 해당 이벤트 노드가 있어야 한다. 없으면 BPMN이 미완성이다.
- Spec의 에러 코드가 정책서 조건에서 나오지 않았다면, 그 에러 기준은 누가 정한 것인지 확인해야 한다. 정책서 보완이 먼저다.
- Spec이 TC보다 먼저 작성됐다면, 테스트 포인트 항목이 TC 설계의 직접 입력이 된다. 이 연결이 끊기면 TC가 Spec과 다른 기준을 검증한다.

## 이 장의 핵심 메시지

사양서는 기획자와 개발자가 "같은 것을 만들고 있다"는 것을 확인하는 문서다. 말로 맞춘 것은 사람마다 다르게 기억한다. 문서에 쓰인 것은 다시 돌아볼 수 있다.

AI와 일할 때 사양서의 역할은 더 직접적이다. AI는 사양서의 입력·출력·상태·예외를 그대로 코드 구조로 변환한다. 이 네 항목이 명확하게 기술이면 AI가 임의로 채우는 빈칸이 줄어든다.

---

## 정리

사양서는 옛날 문서의 잔재가 아니다. 좋은 사양서는 정책, 흐름, 판단을 구현 가능한 조건으로 바꾸고, 뒤에서 개발과 QA, AI 코드 생성이 같은 기준을 보게 만드는 핵심 문서다.

### 사양서에서 API 명세서로 이동하는 항목

| 사양서 항목 | 이동 형태 | API 명세서에서의 역할 |

| 기능별 출력 정의 | 응답 필드·타입 목록 | API Response 스키마 |

| 외부 호출 인터페이스 | 호출 대상·프로토콜 | API 외부 연동 엔드포인트 |
| 상태 전환 조건 | 상태 변경 트리거 | API의 사이드이펙트 설명 |
