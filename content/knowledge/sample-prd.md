# PRD-예시

상위 원칙: `아웃풋 기준 역설계`

## 문서 목적
이 문서는 어떤 기능이든 기본 골격으로 바로 가져다 쓸 수 있는 PRD 예시다.  
구체 도메인 샘플은 `PRD-회원가입-로그인`을 보고, 범용 구조는 이 문서를 보면 된다.

## 1. 문제 정의
- 현재 어떤 문제가 존재하는가
- 누가 불편을 겪고 있는가
- 왜 지금 이 문제를 다뤄야 하는가

예시 문장:
- 사용자가 원하는 작업을 완료하는 데 단계가 많아 이탈이 발생한다.

## 2. 목표
- 이 기능으로 무엇을 개선할 것인가
- 성공을 어떻게 판단할 것인가

예시 문장:
- 핵심 작업 완료율을 높인다.
- 반복 문의를 줄인다.

## 3. 대상 사용자
- 1차 사용자
- 2차 사용자
- 내부 운영 사용자(있다면)

## 4. 범위

### 포함
- 이번 릴리즈에서 반드시 다루는 항목

### 제외
- 이번에는 하지 않는 항목
- 다음 단계로 넘길 항목

## 5. 핵심 사용자 시나리오
- 시나리오 A
- 시나리오 B
- 예외 시나리오

예시 문장:
1. 사용자가 기능 화면에 진입한다.
2. 필요한 정보를 입력하거나 선택한다.
3. 시스템이 결과를 보여 준다.
4. 실패 시 대체 흐름을 안내한다.

## 6. 핵심 요구사항
- 반드시 있어야 하는 기능 요구사항
- 상태/메시지/권한/예외 요구사항
- 다른 시스템 연동 조건

## 7. 정책/예외 후보
- 기본 규칙
- 예외 규칙
- 사람 판단이 필요한 조건
- 차후 정책서로 분리해야 할 항목

## 8. 성공 기준
- 정량 기준
- 정성 기준
- 운영 기준

## 9. 리스크/의존성
- 외부 시스템 의존성
- 정책 확정 필요 항목
- 디자인/개발/운영 상 리스크

## 10. 다음 산출물 연결`r`n- 정책 후보`r`n- 흐름 분기`r`n- 판단 규칙`r`n- 상태/메시지`r`n- 화면 구조`r`n`r`n## 사용 팁
- 먼저 `문제 정의`, `범위`, `성공 기준`부터 채운다.
- 구현 세부사항은 PRD에 과도하게 넣지 않는다.
- 정책/예외가 많아지면 정책서로 분리한다.
